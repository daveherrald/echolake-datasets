# T1059.001-18: PowerShell — Invoke Known Malicious Cmdlets

## Technique Context

T1059.001 (PowerShell) covers adversary use of PowerShell as an execution engine. It is one of the most widely observed techniques in post-compromise activity because PowerShell provides deep access to Windows internals, .NET, WMI, and the network stack — all through a signed, built-in interpreter that many environments cannot simply disable. Defenders have responded with script block logging (EID 4104), module logging (EID 4103), and AMSI integration, each of which creates visibility into what PowerShell is actually executing rather than merely what was passed on the command line.

This specific test simulates an attacker defining a set of function stubs whose names match well-known offensive PowerShell tools — `Add-Persistence`, `Find-AVSignature`, `Get-GPPAutologon`, `Get-GPPPassword`, `Get-HttpStatus`, `Get-Keystrokes`, `Get-SecurityPackages`, `Get-TimedScreenshot`, `Get-VaultCredential`, `Get-VolumeShadowCopy`, `Install-SSP`, `Invoke-CredentialInjection`, `Invoke-DllInjection`, `Invoke-Mimikatz`, and roughly 20 more from the PowerSploit and Empire ecosystems — and then invoking each by name in a foreach loop. The intent is to trigger detection logic that keys off cmdlet names in script block content without requiring the actual malicious payloads to be present.

Detection for this class of activity commonly focuses on EID 4104 script block content for known-bad function names, EID 4688 / Sysmon EID 1 command-line capture, and behavioral indicators such as `whoami.exe` executing under a PowerShell parent. In a defended environment, Windows Defender terminates the process with `STATUS_ACCESS_DENIED` (0xC0000022) before the mock cmdlets are defined, which sharply limits 4104 content.

## What This Dataset Contains

With Defender disabled, the test executes to completion and the telemetry is substantially richer than the defended version.

Security EID 4688 captures the complete command line passed to the child PowerShell process (PID 0x1510, parent 0x193c):

```
"powershell.exe" & {$malcmdlets = "Add-Persistence", "Find-AVSignature", "Get-GPPAutologon",
"Get-GPPPassword", "Get-HttpStatus", "Get-Keystrokes", "Get-SecurityPackages", "Get-TimedScreenshot",
"Get-VaultCredential", "Get-VolumeShadowCopy", "Install-SSP", "Invoke-CredentialInjection",
"Invoke-DllInjection", "Invoke-Mimikatz", "Invoke-NinjaCopy", ...}
```

This alone provides a rich keyword corpus in a single processCreate event. Two `whoami.exe` processes are also captured via EID 4688 (parent powershell.exe), showing the test framework running system context checks before and after the test.

The PowerShell channel holds 96 events — 93 EID 4104, 2 EID 4100, 1 EID 4103. The 4104 content sampled includes the ART test framework (`Import-Module 'C:\AtomicRedTeam\invoke-atomicredteam\Invoke-AtomicRedTeam.psd1' -Force`, `Invoke-AtomicTest T1059.001 -TestNumbers 18 -Cleanup`) but the 93 script blocks also span the full mock-cmdlet execution. The majority of the 93 are short boilerplate scriptlets generated by the PowerShell engine (`Set-StrictMode -Version 1`, `$_.PSMessageDetails`) rather than the test payload itself; this is normal PowerShell operational behavior, not a gap.

Sysmon provides 21 events across EIDs 7, 1, 10, 17, and 8. EID 7 (ImageLoad) shows the .NET CLR stack loading into powershell.exe: `mscoreei.dll`, `clr.dll`, `mscorlib.ni.dll`, `clrjit.dll`, and Windows Defender's `MpOAV.dll` (even with real-time protection disabled, AMSI components load). EID 17 captures the named pipe `\PSHost.134180039549797577.6460.DefaultAppDomain.powershell` — the standard PowerShell console host communication channel, useful for process correlation. EID 10 shows the parent PowerShell (PID 6460) opening `whoami.exe` with `GrantedAccess: 0x1FFFFF` (full access), tagged by Sysmon as `technique_id=T1055.001`. EID 8 shows PowerShell creating a remote thread in an unidentified target (PID 5392, `TargetImage: <unknown process>`), an artifact of the ART test framework's process management that is nonetheless a high-fidelity injection indicator.

Compared to the defended version, the key difference is that the full malicious cmdlet list appears in EID 4688 command-line capture and the test runs to completion rather than exiting with 0xC0000022. The defended dataset had only 41 PowerShell events and no test-payload script blocks; this dataset has 96 PowerShell events and the full command-line artifact in security logs.

## What This Dataset Does Not Contain

The 4104 samples show that most of the 93 script block events are short engine-generated scriptlets rather than the foreach loop body that defines and calls each mock cmdlet. Because the test uses stub functions (the cmdlets are defined as empty functions just to be callable by name), there is no interesting execution payload — no credential harvesting, no actual DLL injection. If you need to see what `Invoke-Mimikatz` or `Invoke-DllInjection` look like in practice, this test is not the right dataset.

No Sysmon EID 3 (NetworkConnect) events are present — this test makes no network connections. No registry modification events (EID 13) appear. No file creation events beyond routine PowerShell profile writes.

The dataset was captured on ACME-WS06 (ACME-WS06.acme.local) as NT AUTHORITY\SYSTEM, which means the token elevation type on all 4688 events is `TokenElevationTypeDefault (1)` — the full-token non-UAC path. This is consistent with ART running as SYSTEM but differs from many real-world attacker scenarios where the session starts as a lower-privileged user.

## Assessment

This dataset is useful for two distinct purposes. First, the EID 4688 command line provides a concrete list of offensive PowerShell function names in a single event — training data for content-based models that extract named entities from process command lines. Second, the Sysmon EID 8 CreateRemoteThread event from PowerShell to an unknown process is a behavioral indicator present here even though the test executes only harmless stub functions; it documents a pattern that recurs across multiple ART PowerShell tests in this batch and reflects genuine ART test framework behavior rather than the technique under test.

The dataset is not a substitute for telemetry from tools that actually implement the named functions. Its value is in the naming patterns and the process execution chain, not in downstream payload activity.

## Detection Opportunities Present in This Data

1. EID 4688 command line contains a comma-separated list of known offensive PowerShell function names as a string literal — a direct keyword match opportunity against a single event field.
2. Sysmon EID 8 from `powershell.exe` to `<unknown process>` with no StartModule or StartFunction resolved — CreateRemoteThread from a scripting interpreter targeting an unnamed process is a high-confidence indicator.
3. Sysmon EID 10 with `GrantedAccess: 0x1FFFFF` from `powershell.exe` to `whoami.exe` — full-access handle opens on a system discovery utility from an interactive PowerShell session.
4. Sysmon EID 1 `RuleName: technique_id=T1059.001` on the child `powershell.exe` with `CommandLine: "powershell.exe" & {$malcmdlets = ...}` — the `& {}` invocation style alongside a variable containing tool names.
5. EID 4688 showing `powershell.exe` spawning `whoami.exe` — system identity discovery as an immediate child of a PowerShell process, consistent with attacker context-checking.
6. Sysmon EID 17 pipe name containing `.DefaultAppDomain.powershell` — presence of the PSHost named pipe confirms an interactive or semi-interactive PowerShell console session (as opposed to a non-interactive runspace).
