# T1546-8: Event Triggered Execution — Persistence via ErrorHandler.cmd Script Execution

## Technique Context

Windows Setup includes a mechanism for executing a custom script after an upgrade or installation phase: if `C:\Windows\Setup\Scripts\ErrorHandler.cmd` exists, it is executed by the Windows Setup engine (`oobe\Setup.exe`) when setup encounters an error. Attackers plant a malicious `ErrorHandler.cmd` script in this location so that it executes during the next setup or upgrade event — which may occur during Windows Update, Sysprep, or manual setup reruns. This is a low-visibility persistence path: the `Setup\Scripts` directory is rarely monitored, and the trigger is a legitimate OS mechanism. Defenders watch for file writes to `C:\Windows\Setup\Scripts\`, particularly `.cmd` files, and for `oobe\Setup.exe` executing scripts from that path.

## What This Dataset Contains

This is the most complex dataset in the T1546 series, with a 2-minute collection window (23:34:59 – 23:37:03) and events across six channels. The test copies `ErrorHandler.cmd` to `C:\Windows\Setup\Scripts\` and then invokes `C:\windows\System32\oobe\Setup` to trigger it.

**Sysmon EID=1 — PowerShell with full attack command:**
`"powershell.exe" & {Copy-Item -Path C:\AtomicRedTeam\atomics\T1546\src\ErrorHandler.cmd -Destination C:\Windows\Setup\Scripts\ErrorHandler.cmd; C:\windows\System32\oobe\Setup}`
This is the primary evidence: a PowerShell process spawned from the test framework explicitly copying the ErrorHandler.cmd and invoking the setup binary.

**Sysmon EID=1 — `WmiPrvSE.exe -Embedding`** (at 23:36:56): The WMI Provider Host launched as `NT AUTHORITY\NETWORK SERVICE` approximately 2 minutes into the collection window, tagged `technique_id=T1047,technique_name=Windows Management Instrumentation`. This is likely background WMI activity triggered by the oobe/Setup invocation.

**Sysmon EID=12/13 — WmiApSrv.exe registry activity**: Multiple delete/set operations under `HKLM\System\CurrentControlSet\Services\WmiApRpl\Performance\` from `WmiApSrv.exe` — WMI performance counter registration triggered by the Setup/oobe invocation.

**Sysmon EID=13 — Task Scheduler write**: `svchost.exe` writes `HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Schedule\TaskCache\Tree\Microsoft\Windows\SoftwareProtectionPlatform\SvcRestartTask\Index`, tagged `technique_id=T1053`.

**Security 4624/4627/4672:** Service logon events (Type 5) for `NT AUTHORITY\SYSTEM` from `services.exe` — background service activity triggered during the Setup execution.

**System EID=7040:** "The start type of the Background Intelligent Transfer Service service was changed from auto start to demand start" — Setup modified the BITS service start type.

**Application EID=16384:** "Successfully scheduled Software Protection service for re-start at 2026-05-03T04:06:09Z. Reason: RulesEngine." — SppExtComObj activity from the Setup execution.

**TaskScheduler EID=140:** `ACME\ACME-WS02$` updated Task Scheduler task `\Microsoft\Windows\SoftwareProtectionPlatform\SvcRestartTask`.

## What This Dataset Does Not Contain

- No Sysmon EID=11 recording `C:\Windows\Setup\Scripts\ErrorHandler.cmd` being written. The file create for the ErrorHandler.cmd copy is absent — the Sysmon file create rule did not fire for the `Setup\Scripts` path.
- No Sysmon EID=1 for `Setup.exe` or `ErrorHandler.cmd` execution. Neither `oobe\Setup.exe` nor any cmd.exe launched by it appears as a process create — either the Sysmon include-mode ProcessCreate filter did not match these executables, or Setup ran in a context that Sysmon did not fully intercept.
- No evidence of `ErrorHandler.cmd` content executing. The persistence payload itself is not seen executing.
- No PowerShell EID=4104 with technique content — the PowerShell channel for this test contains test framework boilerplate (EID=4100/4102 also present, indicating PowerShell engine lifecycle events but no technique script blocks in the filtered window).

## Assessment

This dataset is moderately useful but the most critical telemetry — the ErrorHandler.cmd file write and its execution — is absent. The PowerShell process create (Sysmon EID=1) with the `Copy-Item` command and `oobe\Setup` invocation is the primary detection anchor. The surrounding background activity (WMI, BITS, task scheduler, service logons) is realistic noise generated by the oobe/Setup execution. For detection engineering focused on the file-write phase, Sysmon EID=11 coverage of `C:\Windows\Setup\Scripts\` needs to be added to the config. For the execution phase, `oobe\Setup.exe` should be added to the Sysmon ProcessCreate include rules.

## Detection Opportunities Present in This Data

1. **Sysmon EID=1 — PowerShell command line containing `Copy-Item` with destination `C:\Windows\Setup\Scripts\ErrorHandler.cmd`**: Any process copying a file to this specific destination is immediately suspicious — this path serves no normal administrative purpose.
2. **Sysmon EID=1 / Security 4688 — `oobe\Setup.exe` invoked from PowerShell or cmd.exe** in a non-setup context: Setup.exe is not a tool normally run post-installation; invoking it from a scripting context is anomalous.
3. **Sysmon EID=11 (requires config extension) — file write to `C:\Windows\Setup\Scripts\`**: Adding this path to the Sysmon FileCreate rules would directly capture the persistence installation. Any `.cmd`, `.bat`, or `.ps1` file written to `C:\Windows\Setup\Scripts\` is high-confidence.
4. **System EID=7040 — BITS service start type changed to demand start**: The BITS service modification during the Setup invocation window is a correlating indicator that Setup.exe was actually executed.
5. **Sysmon EID=13 + TaskScheduler EID=140 within the same time window as the Setup invocation**: The cluster of background infrastructure changes (WMI counter registration, task update, BITS change) co-occurring with the attacker's PowerShell activity establishes a time-bounded activity window for correlation.
6. **Baseline monitoring for `C:\Windows\Setup\Scripts\ErrorHandler.cmd` file existence**: This file should not exist on production systems. Any endpoint where this file is present should be investigated.
