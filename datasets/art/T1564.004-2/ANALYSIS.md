# T1564.004-2: NTFS File Attributes — Store File in Alternate Data Stream (ADS)

## Technique Context

T1564.004 (NTFS File Attributes) includes techniques that use NTFS Alternate Data Streams (ADS) to hide files or data alongside legitimate files. This test demonstrates a specific social-engineering-adjacent variant: writing a binary into a stream attached to a `desktop.ini` file inside `C:\Users\Public\Libraries\yanki\`. The `desktop.ini` filename is chosen to blend with normal Windows shell folder metadata, making the ADS payload less conspicuous. The technique uses a PowerShell `Start-Process` wrapper to invoke `cmd.exe /c type ... > file.ini:stream`, transferring `cmd.exe` itself (as a stand-in for a payload) into the stream.

## What This Dataset Contains

**Security 4688** records the full PowerShell command line:
```powershell
& {if (!(Test-Path C:\Users\Public\Libraries\yanki -PathType Container)) {
    New-Item -ItemType Directory -Force -Path C:\Users\Public\Libraries\yanki
}
Start-Process -FilePath "$env:comspec" -ArgumentList "/c,type,c:\windows\system32\cmd.exe,>,`"C:\Users\Public\Libraries\yanki\desktop.ini:desktop.ini`""}
```
And the child `cmd.exe` with command:
```
"C:\Windows\system32\cmd.exe" /c,type,c:\windows\system32\cmd.exe,>,"C:\Users\Public\Libraries\yanki\desktop.ini:desktop.ini"
```
`cmd.exe` exits with `0x0` — the ADS write succeeded.

**Sysmon EID 15 (FileStream created)** fires twice for the same file path. The first record captures creation of the base file:
```
TargetFilename: C:\Users\Public\Libraries\yanki\desktop.ini
```
The second captures the named stream:
```
TargetFilename: C:\Users\Public\Libraries\yanki\desktop.ini:desktop.ini
Hash: SHA1=94BDAEB5...,MD5=7620F0BC...,SHA256=A6E3B3B2...
Image: C:\Windows\system32\cmd.exe
```
The SHA256 hash of the stream content is recorded, enabling hash-based lookup and IOC correlation.

**Sysmon EID 1** captures the PowerShell launcher and the `cmd.exe` ADS write process with full command lines and parent-child linkage.

**PowerShell 4104** records both the outer wrapper invocation and the exact `Start-Process ... desktop.ini:desktop.ini` script block, providing a third source for the technique command.

The Security log also includes `SearchProtocolHost.exe` and `SearchFilterHost.exe` process creations triggered by Windows Search indexing the newly created file path — authentic background activity generated by the OS in response to filesystem changes during the test.

## What This Dataset Does Not Contain (and Why)

No file deletion or cleanup events appear. Per-test cleanup is configured in the ART test framework, but Sysmon file-delete events (EID 23 and EID 26) are not among the collected channels — file creation is captured but deletion is not.

No EID 1 for `cmd.exe` appears in Sysmon. The sysmon-modular include rules do not match `cmd.exe` in this execution context. Security 4688 provides complete coverage of the cmd.exe process creation.

## Assessment

The ADS write succeeded. The dataset provides high-quality, multi-source documentation of the technique: the command line (Security 4688, Sysmon EID 1, PowerShell 4104) and the file system artifact (Sysmon EID 15) with a cryptographic hash of the stream content. The `desktop.ini`-named stream in a public user profile location is a realistic adversary choice for blending with OS-generated metadata files.

## Detection Opportunities Present in This Data

- **Sysmon EID 15 (FileStreamCreate) with `:` in `TargetFilename`**: the ADS stream name is visible as `filename.ext:streamname` in the EID 15 field, providing a direct and reliable indicator. Any EID 15 where the TargetFilename contains a colon after the extension should be investigated.
- **`cmd.exe` command line containing `>` with a `:` path separator** (ADS write syntax): the `type ... > file.ini:stream` pattern in a 4688 command line is a strong indicator of ADS creation.
- **PowerShell 4104 containing `desktop.ini:` or similar ADS-style paths**: script block logging captures these colon-path references verbatim.
- **`SearchProtocolHost.exe` triggered by an unusual file path**: Windows Search indexing a newly created file in `C:\Users\Public\Libraries\<unexpected_folder>` can surface as a secondary behavioral signal when correlated with prior suspicious process activity.
- **New directory creation under `C:\Users\Public\Libraries\`**: legitimate software rarely creates new subfolders here. A `New-Item -ItemType Directory` under this path warrants scrutiny.
