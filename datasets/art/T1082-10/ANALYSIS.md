# T1082-10: System Information Discovery — Griffon Recon

## Technique Context

T1082 System Information Discovery is a fundamental reconnaissance technique where adversaries gather information about the compromised system's hardware, operating system, configuration, and environment. This intelligence helps attackers understand their foothold, identify potential privilege escalation paths, and determine lateral movement opportunities. The "Griffon Recon" variant simulates a VBScript-based reconnaissance script that systematically collects system information through Windows Management Instrumentation (WMI) and native Windows utilities.

Detection engineers focus on identifying automated system enumeration patterns, particularly when multiple system discovery commands are executed in sequence or through scripting engines. The technique often manifests as chains of reconnaissance commands, WMI queries, or execution of native discovery utilities like `whoami`, `systeminfo`, or `wmic`. While individual discovery commands may be benign, rapid sequences or unusual calling patterns can indicate automated reconnaissance.

## What This Dataset Contains

This dataset captures the execution of a VBScript reconnaissance script through PowerShell. The primary execution chain shows PowerShell (PID 44284) spawning a child PowerShell process (PID 43952) with the command `"powershell.exe" & {cscript \"C:\AtomicRedTeam\atomics\T1082\src\griffon_recon.vbs\"}`, which then launches cscript.exe (PID 43168) with `"C:\Windows\system32\cscript.exe" C:\AtomicRedTeam\atomics\T1082\src\griffon_recon.vbs`.

Security event 4688 captures the complete process creation chain, including the `whoami.exe` execution with command line `"C:\Windows\system32\whoami.exe"` spawned by the initial PowerShell process. The cscript.exe process exits with status 0x1 (indicating an error condition), while other processes exit cleanly.

Sysmon provides rich telemetry including process GUIDs for correlation, image loads showing VBScript.dll and AMSI.dll loading into cscript.exe, and process access events (EID 10) showing PowerShell processes accessing child processes with full access rights (0x1FFFFF). The PowerShell channels capture only test framework boilerplate - Set-ExecutionPolicy Bypass commands and generic PowerShell error handling scriptblocks.

WMI activity is evidenced by the WmiPrvSE.exe process creation (PID 29700) with command line `C:\Windows\system32\wbem\wmiprvse.exe -Embedding`, indicating WMI provider host activation likely triggered by the reconnaissance script's WMI queries.

## What This Dataset Does Not Contain

The dataset lacks the actual reconnaissance output or WMI query specifics since the VBScript appears to have encountered execution errors (cscript.exe exit code 0x1). We don't see the typical system discovery command outputs that would normally be generated by successful execution of systeminfo, wmic queries, or registry enumeration commands.

The PowerShell script block logging only captures test framework setup commands rather than the actual reconnaissance script content, so the specific discovery techniques employed by the Griffon Recon script aren't visible in the PowerShell telemetry. Missing are potential network discovery commands, file system enumeration, or registry queries that might be part of comprehensive system reconnaissance.

The Sysmon configuration's include-mode filtering for process creation means we may be missing some native discovery utilities if they weren't explicitly included in the suspicious process patterns, though major tools like whoami.exe are captured.

## Assessment

This dataset provides good coverage of the process execution chain and demonstrates PowerShell-to-VBScript execution patterns common in reconnaissance activities. The Security channel's complete process creation logging with command lines offers excellent detection opportunities, while Sysmon's process access events show the parent-child relationships clearly.

However, the apparent execution failure limits the dataset's utility for understanding the full reconnaissance behavior. The lack of successful WMI queries, system command outputs, or comprehensive discovery patterns means this primarily demonstrates the attack setup rather than the complete technique execution. For building robust T1082 detections, additional datasets showing successful reconnaissance execution would be valuable.

The telemetry quality is strong for detecting the execution method (PowerShell → VBScript → system discovery) but limited for understanding the breadth of information gathering that successful system reconnaissance typically involves.

## Detection Opportunities Present in This Data

1. **PowerShell spawning VBScript execution** - Security 4688 events showing powershell.exe creating cscript.exe processes with .vbs file arguments, particularly with suspicious script paths like atomics directories

2. **Rapid process creation sequences** - Multiple process creations within seconds involving PowerShell, cscript, and discovery utilities like whoami.exe indicating automated reconnaissance chains

3. **WMI provider activation patterns** - WmiPrvSE.exe process creation with "-Embedding" parameter concurrent with scripting engine activity, suggesting WMI-based system enumeration

4. **Cross-process access for monitoring** - Sysmon EID 10 showing PowerShell processes accessing child processes with full access rights (0x1FFFFF), indicating parent process monitoring of reconnaissance activities

5. **VBScript engine loading with AMSI** - Sysmon EID 7 showing vbscript.dll and amsi.dll loading into cscript.exe processes, indicating potential script-based reconnaissance with anti-malware scanning interface engagement

6. **PowerShell execution policy bypass** - PowerShell 4103 events showing Set-ExecutionPolicy with Bypass parameter in process scope, often preceding malicious script execution

7. **System discovery utility execution** - Security 4688 events capturing whoami.exe or similar discovery tools spawned by scripting engines rather than direct user interaction
