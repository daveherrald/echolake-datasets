# T1112-7: Modify Registry — Change PowerShell Execution Policy to Bypass

## Technique Context

T1112 (Modify Registry) applied to PowerShell execution policy is one of the most frequently observed defense evasion steps in Windows intrusions. The PowerShell execution policy is a safety feature—not a security boundary—that governs whether unsigned scripts can run. It is stored in `HKLM\SOFTWARE\Microsoft\PowerShell\1\ShellIds\Microsoft.PowerShell\ExecutionPolicy`. Setting this value to `Bypass` removes all script execution restrictions, allowing any unsigned script to run without prompts or errors.

Attackers set execution policy to Bypass early in attack chains because many commodity malware frameworks, post-exploitation tools, and living-off-the-land scripts are unsigned. While a sophisticated attacker can bypass the execution policy without touching the registry (using `-ExecutionPolicy Bypass` as a command-line flag, or using the `bypass` scope flag), modifying the `LocalMachine` scope registry value provides a persistent change that survives PowerShell restarts and affects all users on the system. This is the more aggressive form of the technique: it does not just affect the current session but reconfigures the system for all future PowerShell invocations until explicitly reversed.

Detection of execution policy changes is particularly important because they frequently precede script-based credential harvesting, lateral movement scripts, or C2 implant deployment via PowerShell.

## What This Dataset Contains

This dataset captures the execution policy modification on a Windows 11 Enterprise domain workstation with Defender disabled, recorded at approximately 2026-03-14T23:53:50Z to 23:53:54Z.

Unlike the other T1112 tests in this batch—which use cmd.exe and reg.exe—this test modifies the registry through a native PowerShell cmdlet. Sysmon EID 1 captures a `powershell.exe` process (PID 232, ProcessGuid `{9dc7570a-f511-69b5-0113-000000000600}`, RuleName `technique_id=T1059.001`) with the command line: `"powershell.exe" & {Set-ExecutionPolicy -ExecutionPolicy Bypass -Scope LocalMachine}`

This PowerShell process runs from `C:\Windows\TEMP\` as `NT AUTHORITY\SYSTEM`. The command line exposes the complete action in plaintext.

The PowerShell channel provides unusually rich content for this test. EID 4104 (script block logging) captures the script block `& {Set-ExecutionPolicy -ExecutionPolicy Bypass -Scope LocalMachine}` as well as the inner block `{Set-ExecutionPolicy -ExecutionPolicy Bypass -Scope LocalMachine}`. More significantly, EID 4103 (module logging) records the cmdlet invocation with full parameter binding:

```
CommandInvocation(Set-ExecutionPolicy): "Set-ExecutionPolicy"
ParameterBinding(Set-ExecutionPolicy): name="ExecutionPolicy"; value="Bypass"
ParameterBinding(Set-ExecutionPolicy): name="Scope"; value="LocalMachine"
```

This EID 4103 event includes host context: `Host Application = powershell.exe & {Set-ExecutionPolicy -ExecutionPolicy Bypass -Scope LocalMachine}`, `Engine Version = 5.1.22621.2428`, and `Runspace ID = 382aba97-fabe-44f4-bfd5-00e60cf33b41`.

The Sysmon EID breakdown (7: 17, 3: 4, 10: 3, 1: 3, 17: 2, 13: 1, 11: 1) shows several distinctive differences from the cmd/reg-based T1112 tests. The presence of 4 Sysmon EID 3 (network connection) events reflects DNS multicast (mDNS) activity from `svchost.exe` at UDP port 5353, which is background OS noise unrelated to the technique. The EID 13 event records the actual registry write to the ExecutionPolicy key. The EID 11 (file create) event may reflect a temporary file written during the PowerShell session.

Security EID 4688 (3 events) captures process creations including `whoami.exe` (system discovery) and the `powershell.exe` process carrying the Set-ExecutionPolicy command line.

The total event count (Sysmon: 31, Security: 3, PowerShell: 40) is the highest among the single-value registry modification tests in this batch, reflecting the additional telemetry generated by the PowerShell-native execution path.

## What This Dataset Does Not Contain

This dataset does not capture the state of the execution policy before the modification. There are no registry read events (Sysmon EID 12 would cover key creation, but query events are not logged by Sysmon by default). The pre-modification value—whether it was `Restricted`, `RemoteSigned`, or `Undefined`—is not visible.

No downstream script execution that takes advantage of the Bypass policy appears in this dataset. The test is the single modification step.

The EID 13 event recording the write to `HKLM\SOFTWARE\Microsoft\PowerShell\1\ShellIds\Microsoft.PowerShell\ExecutionPolicy` is in the full dataset but not in the sample subset.

The mDNS network events (Sysmon EID 3) are background OS activity—`svchost.exe` making mDNS announcements on `224.0.0.251:5353`—coincident with the test window. They are not related to the technique.

## Assessment

The undefended dataset (Sysmon: 31, Security: 3, PowerShell: 40) compared to the defended variant (Sysmon: 37, Security: 10, PowerShell: 39) shows a reduction in both Sysmon and Security events. The Security channel drops from 10 to 3 events—Defender generates additional process inspection events when active. Sysmon drops from 37 to 31, likely because Defender's scan process (MsMpEng.exe) activity is reduced when real-time protection is disabled.

The most important contrast between this test and its neighbors is the execution mechanism. T1112-63 through T1112-67 all use cmd.exe → reg.exe; T1112-7 uses a native PowerShell cmdlet. This produces qualitatively different telemetry: instead of process creation with a command-line argument containing the registry path, you get PowerShell script block logging and module logging events that capture the parameter binding at the PowerShell layer. Both paths are fully detectable, but through different primary evidence types.

## Detection Opportunities Present in This Data

**PowerShell command-line argument (Sysmon EID 1 / Security EID 4688):** The command line `"powershell.exe" & {Set-ExecutionPolicy -ExecutionPolicy Bypass -Scope LocalMachine}` is captured verbatim. `Set-ExecutionPolicy` with `-Scope LocalMachine` and `-ExecutionPolicy Bypass` in a command-line argument is a high-confidence signal, especially when running as SYSTEM.

**Script block logging (PowerShell EID 4104):** The script block `Set-ExecutionPolicy -ExecutionPolicy Bypass -Scope LocalMachine` is captured with its ScriptBlock ID. This provides detection coverage even when the command line is obfuscated, as PowerShell decodes the script before logging it.

**Module logging / cmdlet invocation (PowerShell EID 4103):** The parameter binding record explicitly names the cmdlet (`Set-ExecutionPolicy`), the execution policy value (`Bypass`), and the scope (`LocalMachine`). This is the most structured and parseable form of the evidence.

**Registry value set (Sysmon EID 13):** The write to `HKLM\SOFTWARE\Microsoft\PowerShell\1\ShellIds\Microsoft.PowerShell\ExecutionPolicy` with value `Bypass` provides a direct, process-independent detection path covering both this technique variant and any other mechanism (Group Policy, direct API calls) that writes to the same key.
