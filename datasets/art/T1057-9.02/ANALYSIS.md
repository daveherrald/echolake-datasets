# T1057-9: Process Discovery — Launch Taskmgr from cmd to View Running Processes

## Technique Context

T1057 Process Discovery encompasses both command-line enumeration tools and GUI-based methods. This test takes a less common but operationally realistic approach: launching Task Manager (`taskmgr.exe`) from the command line with the `/7` parameter, which forces the "Details" tab view showing full process information including PIDs, status, CPU, memory, and description fields. Attackers with interactive desktop access — through RDP, VNC, or an implant with GUI capability — may use Task Manager instead of command-line tools because it's visually accessible, doesn't require knowledge of specific commands, and is less likely to be blocked by application whitelisting policies that focus on terminal utilities.

The `/7` parameter is notable. Task Manager's command-line parameters are not well-documented, and their use indicates foreknowledge or deliberate testing. Parameter `/7` corresponds to the "Details" tab, which provides the richest process view available in Task Manager. An adversary using this specific flag understands the tool at a level beyond basic "open Task Manager and look around."

Detection of GUI-launched process discovery is inherently harder than command-line tool detection. Task Manager is a trusted system binary invoked constantly by legitimate users. Distinguishing malicious use requires context: who launched it, from what parent process, under what account, and whether the launch was automated (scripted parent) vs. interactive. This dataset shows a scripted launch from cmd.exe under SYSTEM — a pattern that would be rare in legitimate use.

An additional artifact in this dataset is a Windows Error Reporting (WER) crash, caused by Task Manager failing when launched with `/7` in a non-interactive session. The crash generates WerFault.exe events and security logon events, providing ancillary data not typically associated with process discovery.

## What This Dataset Contains

The dataset spans six seconds (2026-03-14T23:17:43Z to 23:17:49Z) and records 147 events across four channels: Sysmon (17), PowerShell (104), Security (24), and Application (2).

**Security EID 4688** records ten process creation events, the most of any T1057 test in this batch. Key events include:

- `"cmd.exe" /c taskmgr.exe /7` — the cmd.exe invocation launching Task Manager with the Details tab parameter
- `taskmgr.exe  /7` — Task Manager spawned by cmd.exe (note the double space, an artifact of command parsing)
- `C:\Windows\System32\svchost.exe -k WerSvcGroup` — the Windows Error Reporting service host, started in response to the crash
- `C:\Windows\system32\WerFault.exe -pss -s 384 -p 3432 -ip 3432` — WerFault doing a process snapshot before the report
- `C:\Windows\system32\WerFault.exe -u -p 3432 -s 1660` — WerFault's user-mode error reporting for the crash

The WerFault events confirm that `taskmgr.exe` crashed (PID 3432 in the samples) when run under SYSTEM in a non-interactive session — it requires a logged-on user session to function and generates an unhandled exception without one.

**Sysmon EID 1** captures the same process chain with rule tagging. The `cmd.exe` event fires `technique_id=T1059.003`. The WerFault events fire `technique_id=T1546.008,technique_name=Windows Error Reporting`, which documents the crash precisely. The Task Manager launch from cmd.exe is the primary technique artifact.

**Sysmon EID 10 (ProcessAccess)** shows six cross-process access events — higher than other T1057 tests. The additional events include access to `cmd.exe` in addition to `whoami.exe`. All use `GrantedAccess: 0x1FFFFF` and CLR call traces from the test framework.

**Sysmon EID 7 (ImageLoad)** shows `urlmon.dll` loading into both `powershell.exe` and `Taskmgr.exe`. The Taskmgr DLL load is interesting because it occurred during the brief window before the crash — Taskmgr started initialization, loaded system libraries, and then failed.

**Security EID 4624 (Logon)** and **EID 4672 (Special Privileges Assigned)** appear twice each. These are logon events generated by the WER service and crash reporting infrastructure establishing a security context for error reporting. `LogonType: 3` (network logon) and `IpAddress: fe80::...` (IPv6 loopback) appear — the crash reporting subsystem uses loopback network authentication.

**Security EID 5379 (Credential Manager Read)** appears ten times. These are generated by the WerFault process reading from Credential Manager as part of crash dump collection, which gathers environment context including stored credentials metadata.

**Application EID 1000** and **EID 1001** (Application Error and Windows Error Reporting) document the Task Manager crash, providing the crash module, exception type, and report GUID.

Compared to the defended version (35 sysmon, 29 security, 34 PowerShell, 2 application), this undefended dataset is nearly identical — Defender does not interfere with Task Manager execution or WER crash handling.

## What This Dataset Does Not Contain

No Sysmon EID 11 (FileCreate) events for the WER report files (typically written to `%LocalAppData%\Microsoft\Windows\WER\`). The crash dump and report files are written to disk but not captured in these samples.

No events show what Task Manager actually displayed before crashing — the enumeration result itself is not logged.

The PowerShell EID 4104 samples are dominated by framework boilerplate and do not include the taskmgr invocation script block in the provided samples.

## Assessment

This dataset is unusual in the T1057 group because it documents both the discovery technique attempt and its failure (Task Manager crash in non-interactive session). The WER events are genuine Windows artifacts from the crash and provide additional forensic context. For detection engineering, the dataset demonstrates that GUI tool launches from scripted parents under SYSTEM are detectable through the same process creation logging used for command-line tools. The crash artifacts (WerFault.exe, EID 1000/1001, Security EID 5379) are secondary indicators that can help identify when discovery attempts fail or behave unexpectedly.

## Detection Opportunities Present in This Data

1. **taskmgr.exe spawned by cmd.exe with /7 parameter**: Sysmon EID 1 and Security EID 4688 both capture `taskmgr.exe  /7` with parent `"cmd.exe" /c taskmgr.exe /7`. Task Manager being launched programmatically by cmd.exe (rather than by explorer.exe or user interaction) is anomalous. The `/7` parameter adds specificity.

2. **cmd.exe /c taskmgr.exe in any non-interactive context**: The command `"cmd.exe" /c taskmgr.exe /7` under `NT AUTHORITY\SYSTEM` or as a child of a script interpreter is a reliable indicator. Legitimate Task Manager launches come from Explorer or direct user execution, not from cmd.exe child processes.

3. **WerFault.exe with parent Taskmgr.exe shortly after scripted taskmgr launch**: The chain `cmd.exe → taskmgr.exe → WerFault.exe` within seconds indicates Task Manager crashed. When this follows a scripted launch (as documented by parent process context), it's a behavioral signature for failed GUI tool abuse.

4. **Security EID 5379 (Credential Manager Read) correlated with WerFault**: Ten Credential Manager read events generated during crash reporting is unusual. Detecting EID 5379 in close temporal proximity to an application crash from a discovery-context process (taskmgr.exe or similar) can indicate that crash reporting is probing credential storage.

5. **Security EID 4624 LogonType=3 from IPv6 loopback during crash handling**: The loopback network logons (`fe80::...`, LogonType=3) generated by WER are distinctive. Seeing multiple Type 3 logons from loopback during an anomalous process chain indicates WER infrastructure activity that can be correlated with the root cause event.

6. **Application EID 1000/1001 combined with preceding scripted process creation**: Application crash events for `taskmgr.exe` or other trusted binaries that were launched programmatically (evidenced by Security EID 4688 parent-child chain) provide a link between technique execution and its forensic aftermath. This correlation helps build timeline-aware detection.
