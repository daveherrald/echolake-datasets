# T1021.003-1: Distributed Component Object Model — PowerShell Lateral Movement using MMC20

## Technique Context

T1021.003 covers the use of Distributed Component Object Model (DCOM) for lateral movement. DCOM extends COM to work across network boundaries, allowing remote process activation and method invocation. Adversaries exploit specific DCOM application objects that expose execution interfaces to run arbitrary commands on remote systems. The technique is attractive because it uses the Windows RPC infrastructure and appears in logs as legitimate COM activation traffic.

The MMC20.Application object is one of the most documented DCOM lateral movement vectors. Its `Document.ActiveView.ExecuteShellCommand()` method was designed to allow management console scripts to launch processes, but attackers use it to execute arbitrary commands. The method takes four parameters: the executable path, window directory, parameters, and window style. When activated against a remote host via `[type]::GetTypeFromProgID("MMC20.application", "<hostname>")`, it creates an MMC process on the target and runs the specified command through it.

When executed locally (as in this test, targeting `localhost`), the activation flow is the same but traverses the local RPC endpoint. The characteristic signature is `mmc.exe -Embedding` — the `-Embedding` argument indicates COM object server activation (as opposed to user-launched MMC), and it appears whenever DCOM activates `mmc.exe` regardless of whether the call is local or remote. The subsequent child process spawned by that `mmc.exe` instance is the payload.

## What This Dataset Contains

The dataset covers approximately 12 seconds (23:03:28–23:03:40 UTC on 2026-03-14) and totals 169 events across three channels. This is the key dataset for capturing the complete DCOM MMC20 execution chain, including both the MMC activation and the payload process creation.

The full process chain is documented in Security EID 4688:

- PowerShell (PID 1484) spawns with the DCOM command: `"powershell.exe" & {[activator]::CreateInstance([type]::GetTypeFromProgID("MMC20.application","localhost")).Document.ActiveView.ExecuteShellCommand("c:\windows\system32\calc.exe", $null, $null, "7")}`
- `mmc.exe` (PID 0x125c) is created by `svchost.exe` (PID 0x394) with command line `C:\Windows\system32\mmc.exe -Embedding` — the COM activation signature
- `calc.exe` (PID 0xfdc) is created by `mmc.exe` (PID 0x125c) with command line `"C:\windows\system32\calc.exe"` — the payload process

This three-tier chain (PowerShell → DCOM → svchost → mmc.exe -Embedding → calc.exe) is the defining artifact of this technique. The `svchost.exe` intermediate is the COM surrogate activating the MMC server, which then spawns the target process.

Sysmon EID 1 captures the PowerShell invocation and a third PowerShell process (the cleanup phase). The Sysmon channel (64 events) is dominated by 29 EID 7 image load events for .NET and Defender DLLs loading into three successive PowerShell processes, 18 EID 22 DNS query events generated by background network activity, and 3 EID 17 named pipe creation events for PowerShell host pipes. The image load events for `mscoreei.dll`, `clr.dll`, `mscorlib.ni.dll`, and `clrjit.dll` in the technique's PowerShell process confirm the .NET DCOM call was loaded and executed.

Compared to the defended version (41 Sysmon, 14 Security, 38 PowerShell), this undefended dataset has significantly more Sysmon events (64) primarily due to the DNS and image load events from the successful full execution, and 8 Security events including the critical `mmc.exe -Embedding` and `calc.exe` process creation events that were absent in the defended run.

## What This Dataset Does Not Contain

There are no Sysmon EID 3 network connection events even though this test uses the `localhost` target. DCOM local activation uses named pipes rather than TCP, so no network connection is logged. If targeting a remote host, Sysmon would capture a TCP connection to the target's RPC endpoint (port 135 initially, then a dynamically assigned port). No file system activity from the calc.exe payload is captured. The PowerShell channel contains only test framework boilerplate — no DCOM invocation details appear in EID 4104 because the DCOM call is made via .NET object invocation rather than script text that would trigger script block logging for the technique itself.

## Assessment

This dataset provides the complete and clean execution chain for DCOM lateral movement via MMC20. The three-tier process chain (DCOM activation → `mmc.exe -Embedding` → calc.exe) is definitively captured in Security EID 4688 and is exactly what detection engineers need for validation. The `mmc.exe -Embedding` event is the single most valuable detection artifact: it only appears when `mmc.exe` is activated via COM, and when combined with an unusual parent (svchost.exe in response to a DCOM call rather than user session) it is a near-zero-false-positive indicator. This dataset is well-suited for testing detections that focus on the MMC DCOM activation pattern.

## Detection Opportunities Present in This Data

1. **EID 4688 / Sysmon EID 1 — mmc.exe with -Embedding argument**: `mmc.exe -Embedding` created by `svchost.exe` is the defining DCOM activation signature. In normal desktop use, MMC is launched without this argument and by the user's explorer.exe or shell. This event alone is a high-confidence detection for DCOM-based MMC abuse.

2. **EID 4688 — calc.exe (or any unexpected process) spawned by mmc.exe**: The payload process is created by `mmc.exe` rather than by the expected shell or installer. Any process creation where the parent is `mmc.exe` and the child is not a legitimate MMC snap-in should be alerted.

3. **Sysmon EID 1 / EID 4688 — PowerShell command line containing GetTypeFromProgID + MMC20**: The full PowerShell command line contains `GetTypeFromProgID("MMC20.application"` — this string pattern in a PowerShell command line (whether in EID 4688, EID 1, or EID 4104) is a reliable indicator of attempted DCOM execution via the MMC application object.

4. **EID 4688 — process chain analysis**: The sequence svchost.exe → mmc.exe (when mmc.exe has `-Embedding`) → child process is the complete lateral movement chain. Detection rules that traverse the parent-child relationships across multiple hops would catch this pattern even if individual events are missed.

5. **Sysmon EID 7 — .NET CLR loading into a new PowerShell process associated with DCOM calls**: The combination of a child PowerShell process loading the full .NET stack (mscoreei.dll + clr.dll + mscorlib.ni.dll) within seconds of `mmc.exe -Embedding` appearing in another process is a timing-based correlation opportunity.
