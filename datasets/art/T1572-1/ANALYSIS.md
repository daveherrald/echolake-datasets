# T1572-1: Protocol Tunneling — DNS over HTTPS Large Query Volume

## Technique Context

T1572 (Protocol Tunneling) covers adversary use of application-layer protocols to
encapsulate C2 traffic within other protocols to evade detection. DNS over HTTPS (DoH)
tunnels DNS queries inside HTTPS connections to port 443 of public resolvers such as
Google (8.8.8.8) or Cloudflare (1.1.1.1), making the traffic appear as ordinary HTTPS.
This test generates a high-volume burst of DoH queries: a loop sending 1001 requests
to Google's DoH endpoint (`https://8.8.8.8/resolve`) using randomized subdomains
under the `xip.io` wildcard domain. High query volume is one behavioral indicator used
to distinguish DNS tunneling from legitimate DoH use.

## What This Dataset Contains

**Sysmon EID 3** — outbound TCP network connection from `powershell.exe`:

> `DestinationIp: 8.8.8.8`
> `DestinationPort: 443`
> `SourcePort: 50612`
> `Protocol: tcp`
> `RuleName: technique_id=T1059.001,technique_name=PowerShell`

This is a single connection record. Sysmon records the connection establishment, not
each individual HTTP request over the connection — the 1001 DoH queries share the same
TCP connection (HTTP/2 multiplexing). This is important context for detection design:
the volume pattern is not visible in Sysmon EID 3 alone.

**Security EID 4688** — process creation for `powershell.exe` and `whoami.exe` under
SYSTEM. The PowerShell process that makes the connection is created ~4 seconds before
the Sysmon EID 3 captures it.

**Security EID 4703** — token right adjustment.

**PowerShell EID 4104** — the test payload is captured verbatim:

> `{for($i=0; $i -le 1000; $i++) { (Invoke-WebRequest "https://8.8.8.8/resolve?name=atomicredteam.$(Get-Random -Minimum 1 -Maximum 999999).127.0.0.1.xip.io&type=TXT" -UseBasicParsing).Content }}`

This script block appears twice: once as the outer ART test framework invocation wrapper
(`& { ... }`) and once as the inner block. The query pattern uses `atomicredteam` as
a fixed subdomain prefix with a random numeric component — a clear indicator in the
EID 4104 data.

## What This Dataset Does Not Contain (and Why)

**No individual DNS query events (Sysmon EID 22).** The 1001 DoH queries travel inside
HTTPS to an IP address; they bypass the OS DNS resolver entirely. Sysmon EID 22 captures
only OS-level DNS resolution calls. DoH traffic goes directly to `8.8.8.8:443` without
touching the Windows DNS client stack, so no EID 22 events are generated by the tunnel
traffic itself. Only traditional DNS resolution (e.g., for DNS name servers or other
hosts) would appear.

**No per-request network events.** As noted above, Sysmon records the TCP connection
once; the 1001 HTTP/2 streams within it are not individually logged.

**No HTTP response content.** The script captures `.Content` from each response but
this is not logged in Windows telemetry. Network-layer logging (e.g., proxy, PCAP)
would be required to see the response data.

**Volume is a single estimate.** The 48-second window (02:00:35Z to 02:01:23Z) for
the complete loop provides a rough rate: ~20 requests/second averaged across
the test window.

## Assessment

This dataset has a small Sysmon and Security footprint but a clear detection signal in
the PowerShell EID 4104 script block. The randomized subdomain pattern with a fixed
`atomicredteam` prefix is an obvious IOC in this ART test; real-world DNS tunneling
would use encoded payload data as subdomains. The detection architecture lesson here
is that host-level telemetry (script block logging) provides the strongest signal, while
network telemetry would be required to detect the volume anomaly without the command
text.

## Detection Opportunities Present in This Data

- **PowerShell EID 4104** — `Invoke-WebRequest` to `https://8.8.8.8/resolve` with
  `name=` and `type=TXT` query parameters is the canonical DoH-via-PowerShell pattern;
  direct-IP HTTPS to `8.8.8.8` or `1.1.1.1` is highly anomalous.
- **Sysmon EID 3** — `powershell.exe` establishing a TCP connection to `8.8.8.8:443`
  is suspicious; legitimate browsers and OS components do not connect to DNS resolver
  IPs directly over HTTPS in enterprise environments.
- **Security EID 4688** — the long-running PowerShell process (48 seconds) correlates
  with the DoH loop; process duration combined with destination IP detection strengthens
  the signal.
- **Network-layer detection** (not in this dataset) — repeated HTTPS sessions to
  `8.8.8.8:443` with high request rates and TXT query parameters in the URI would
  be visible to proxy or HTTPS inspection tools.
- **Absence of Sysmon EID 22** for the query targets — if a process is making DNS-like
  queries but EID 22 is absent, DoH may be in use; correlating `Invoke-WebRequest`
  to resolver IPs with no corresponding OS DNS lookup is a heuristic worth implementing.
