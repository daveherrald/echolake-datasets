# T1112-83: Modify Registry — Modify UsePartialEncryptionKey Registry Entry

## Technique Context

T1112 (Modify Registry) is used here to set `UsePartialEncryptionKey` to `2` in `HKLM\SOFTWARE\Policies\Microsoft\FVE`. The `UsePartialEncryptionKey` value controls whether BitLocker permits the use of partial (fractional) encryption keys—a feature related to how BitLocker keys are stored and distributed in enterprise environments. Setting it to `2` changes the policy to "Allow" mode, relaxing any stricter requirement that had been in place.

This is the third FVE policy modification in this batch alongside T1112-79 (`UseTPMPIN`) and T1112-81 (`UseTPMKeyPIN`). The three tests collectively target the BitLocker authentication policy namespace from multiple angles. An adversary applying all three modifications lowers the bar for BitLocker protection across TPM+PIN, TPM+Startup Key+PIN, and partial key scenarios—a systematic weakening of disk encryption posture.

In the context of ransomware pre-staging, weakening BitLocker policy is an offensive capability that reduces the likelihood of BitLocker interfering with the ransomware payload's ability to read and encrypt files on protected volumes. BitLocker policy changes require SYSTEM-level access and direct registry writes when Group Policy is not in use, making the tool chain (powershell → cmd.exe → reg.exe) an observable indicator.

## What This Dataset Contains

This dataset captures the `UsePartialEncryptionKey` registry modification on a Windows 11 Enterprise domain workstation with Defender disabled. Events occur at approximately 2026-03-17T16:35:54Z, making it the last test in the March 17 session cluster.

The attack chain is PowerShell (SYSTEM) → cmd.exe → reg.exe. Sysmon EID 1 captures both child processes:

- `cmd.exe` (PID 15760, ProcessGuid `{9dc7570a-82ea-69b9-153a-000000000900}`, RuleName `technique_id=T1059.003`) with command line: `"cmd.exe" /c reg add "HKLM\SOFTWARE\Policies\Microsoft\FVE" /v UsePartialEncryptionKey /t REG_DWORD /d 2 /f`
- `reg.exe` (PID 16456, ProcessGuid `{9dc7570a-82ea-69b9-173a-000000000900}`, RuleName `technique_id=T1012`) with command line: `reg  add "HKLM\SOFTWARE\Policies\Microsoft\FVE" /v UsePartialEncryptionKey /t REG_DWORD /d 2 /f`

Both run from `C:\Windows\TEMP\` as `NT AUTHORITY\SYSTEM`. Security EID 4688 independently records the same process chain.

The Sysmon EID breakdown (7: 9, 1: 4, 10: 3, 17: 1) is structurally consistent with T1112-79 and T1112-64 through T1112-67—the standard single-step registry modification template. The PowerShell channel contains 35 EID 4104 events including the cleanup wrapper `Invoke-AtomicTest T1112 -TestNumbers 83 -Cleanup`.

## What This Dataset Does Not Contain

No BitLocker operational events, TPM activity, or volume encryption status changes appear. The test is the registry write in isolation.

Security EID 4657/4663 events are absent—no SACL on the FVE key. The EID 13 registry write event is in the full dataset but not represented in the sample subset.

This test does not include the WMI activity observed in T1112-81—the execution here follows the simpler PowerShell → cmd.exe → reg.exe path without a WMI intermediary.

## Assessment

The undefended dataset (Sysmon: 17, Security: 4, PowerShell: 35) compared to the defended variant (Sysmon: 37, Security: 12, PowerShell: 34) shows the largest Sysmon differential in the batch: 37 defended events versus 17 undefended. This substantial reduction—20 events—suggests Defender's active monitoring of FVE key modifications generates significant Sysmon overhead (likely through Defender's own process inspection activity, which itself generates Sysmon events). With Defender disabled, only the core process creation and DLL load events remain.

The Security reduction is consistent (12 → 4). The PowerShell channel remains nearly identical (34 vs. 35), confirming that the technique execution itself generates a stable baseline of script block events regardless of Defender state.

This is the most "clean" dataset in the FVE cluster: the smallest total event count, no secondary WMI or enumeration activity, and a direct reflection of the minimal telemetry generated by a single reg.exe write when Defender is not present.

## Detection Opportunities Present in This Data

**Process creation command line (Sysmon EID 1 / Security EID 4688):** The full command line `reg add "HKLM\SOFTWARE\Policies\Microsoft\FVE" /v UsePartialEncryptionKey /t REG_DWORD /d 2 /f` is captured in both channels. This is the primary detection evidence.

**FVE key namespace (Sysmon EID 13):** The direct registry write to `HKLM\SOFTWARE\Policies\Microsoft\FVE` is present in the full dataset. A detection covering any write to this key from non-Group Policy process contexts catches all three FVE modification tests (T1112-79, T1112-81, T1112-83) simultaneously.

**Cluster correlation across the session:** This test is the final step in a sequence that includes T1112-74 (disable RDP), T1112-75 (enforce smart card), T1112-79 and T1112-81 and T1112-83 (weaken BitLocker), and T1112-8 (BlackByte UAC changes)—all within approximately 30 seconds of each other on March 17. In a real intrusion, this density of security-policy registry modifications within a single PowerShell session is a high-priority alert signal. The individual detections matter, but the temporal correlation across the cluster matters more.

**Process ancestry from TEMP (Sysmon EID 1):** The `reg.exe` from `C:\Windows\TEMP\` under PowerShell → cmd.exe at SYSTEM integrity is the shared ancestry indicator across all tests in this batch and provides a reliable structural signal independent of the specific registry keys targeted.
