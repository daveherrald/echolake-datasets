# T1018-1: Remote System Discovery — Remote System Discovery - net

## Technique Context

T1018 Remote System Discovery is a fundamental reconnaissance technique where adversaries enumerate systems within the target network. Identifying available hosts and shared resources is a prerequisite for lateral movement and is almost universal in post-exploitation playbooks. The `net` command suite is one of the oldest and most widely abused tools for this purpose: `net view` lists computers visible on the local network via the Browser service or SMB, while `net view /domain` extends that query to all systems registered in the Windows domain.

Detection engineers focus on `net view` and `net view /domain` executions because the combination is a reliable indicator of active network reconnaissance. Legitimate administrator use of these commands is rare and tends to be single, targeted invocations; adversaries typically chain them together in a single shell session as part of systematic discovery. The commands generate no network authentication traffic, making them invisible to firewall logs, which means endpoint telemetry is the primary detection surface.

In this test, the commands are issued over an automated test framework (ART via QEMU guest agent), which means they run as NT AUTHORITY\SYSTEM from `C:\Windows\TEMP\`. In a real intrusion the calling context would typically be an interactive session or a beacon process, but the core command-line pattern is identical.

## What This Dataset Contains

With Windows Defender disabled, the full execution chain is captured across Sysmon and Security channels with no interference. The 15-second window (22:57:59–22:58:14 UTC on 2026-03-14) contains 1,435 events across five channels.

The key activity is documented in Sysmon EID 1 process creation events. PowerShell (PID 6080) spawns `cmd.exe` (PID 2116) with the command line `"cmd.exe" /c net view /domain & net view`. That cmd.exe instance then spawns two `net.exe` processes in sequence: PID 6128 runs `net  view /domain` and PID 6512 runs `net  view`. Both net.exe invocations are tagged by the Sysmon configuration with `RuleName: technique_id=T1018,technique_name=Remote System Discovery`. All processes run as NT AUTHORITY\SYSTEM from `C:\Windows\Temp\`.

Security EID 4688 records the same process creations from the Windows audit log perspective, providing corroborating process command-line fields under the SYSTEM security context (logon ID 0x3E7).

A Sysmon EID 3 network connection event shows MsMpEng.exe (Windows Defender, even when its real-time protection is disabled, still runs) making a TCP connection from 192.168.4.16, unrelated to the discovery commands. The Security channel is dominated by 1,306 EID 4664 hard-link creation events generated by a Windows servicing operation running concurrently — these represent standard OS maintenance activity, not technique artifacts.

Compared to the defended version (which had only 28 Sysmon events and 14 Security events), this dataset is substantially larger: 17 Sysmon events and 1,315 Security events. The core technique evidence — the `net.exe` process creates and their command lines — is present in both versions; the main difference is the absence of any Defender termination or remediation events here, and the Security channel's servicing noise inflating the count.

## What This Dataset Does Not Contain

There are no Sysmon EID 3 network events for the `net.exe` processes themselves, which is expected: `net view` uses SMB/NetBIOS name resolution internally, and Sysmon network monitoring in this configuration does not capture those connections. If the domain has functional domain controllers and network shares, `net view /domain` would return a list of computer names — that output is not captured in any channel here. There are no WMI, LDAP, or DNS query events associated with the commands.

The PowerShell channel (93 events, all EID 4104) contains only ART test framework boilerplate — the `Set-StrictMode` fragments, the `Import-Module` for the ART module, and the cleanup invocation. There are no PowerShell script blocks reflecting the actual `net` command execution because the commands were spawned as a child `cmd.exe` process, not run within PowerShell directly.

## Assessment

This dataset provides clean, unobstructed evidence of the canonical `net view` network discovery pattern. The process chain (powershell.exe → cmd.exe → net.exe x2) with full command lines is well-represented in both Sysmon EID 1 and Security EID 4688. The Sysmon rule tagging makes the events immediately recognizable. This is a good training and validation dataset for detection rules targeting `net.exe` spawned from scripting engines, command-line content matching `net view`, or the parent-child process relationship powershell → cmd → net. The concurrent OS servicing activity in the Security channel provides realistic background context.

## Detection Opportunities Present in This Data

1. **EID 4688 / Sysmon EID 1 — command-line matching**: Process create events where `New Process Name` ends in `net.exe` and `Process Command Line` contains `view` are present twice. Narrowing further to include `view /domain` distinguishes domain-wide enumeration from share browsing.

2. **EID 4688 / Sysmon EID 1 — parent-child chain**: `cmd.exe` spawned by `powershell.exe` with a command line containing both `net view /domain` and `net view` chained with `&` is a high-fidelity pattern for automated network discovery.

3. **Sysmon EID 1 — working directory anomaly**: Both `net.exe` invocations have `CurrentDirectory: C:\Windows\Temp\`, which is unusual for net commands and consistent with execution via an automation framework. Filtering on net.exe launched from temp directories would reduce false positives from legitimate administrator use.

4. **Sysmon EID 1 — execution context**: All processes run as NT AUTHORITY\SYSTEM (logon ID 0x3E7, integrity level System). `net view` running as SYSTEM is unusual in normal operations; most legitimate use comes from user-context sessions.

5. **EID 4688 — rapid sequential execution**: The two `net.exe` processes start within approximately five seconds of each other (22:58:02 and 22:58:07). Detecting multiple `net view` invocations within a short time window from the same parent process is a behavioral clustering opportunity.
