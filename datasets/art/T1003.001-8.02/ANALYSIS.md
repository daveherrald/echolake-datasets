# T1003.001-8: LSASS Memory — Dump LSASS.exe Memory using Out-Minidump.ps1

## Technique Context

LSASS memory dumping is one of the most impactful credential access techniques in the MITRE ATT&CK framework. The Local Security Authority Subsystem Service (LSASS) holds authentication material in memory — plaintext passwords for recently authenticated users, NTLM hashes, and Kerberos tickets. Extracting this material gives an attacker the ability to move laterally across a domain, escalate privileges, and maintain persistence without ever needing to crack a password.

Out-Minidump.ps1 is a PowerShell-based implementation that calls Windows debugging APIs (specifically `MiniDumpWriteDump` via P/Invoke) to produce a full process memory dump. Unlike compiled C2 implant tooling, this approach runs entirely within a PowerShell session, which means the key artifacts live in PowerShell's script block logging. The script has been popular in red team operations precisely because it avoids dropping a compiled binary like ProcDump, though modern endpoint tooling has developed signatures for the P/Invoke patterns it uses.

Detection for this variant centers on three telemetry streams: PowerShell script block logging (EID 4104) capturing the dump code, Sysmon process access events (EID 10) showing LSASS targeted with high-privilege access masks, and Sysmon file creation events (EID 11) recording the `.dmp` file drop. The defended version of this test was blocked before a dump could be created. This dataset captures what that activity looks like when defenses are completely absent.

## What This Dataset Contains

This dataset was collected from ACME-WS06 (Windows 11 Enterprise Evaluation, `acme.local` domain) with Windows Defender disabled. The test executed `Invoke-AtomicTest T1003.001 -TestNumbers 8` as `NT AUTHORITY\SYSTEM`, representing a post-exploitation credential dump scenario.

**PowerShell channel (100 events, 99x EID 4104, 1x EID 4100):** The PowerShell log shows the Atomic Red Team test framework loading — `Import-Module 'C:\AtomicRedTeam\invoke-atomicredteam\Invoke-AtomicRedTeam.psd1' -Force` — followed by the cleanup block `Invoke-AtomicTest T1003.001 -TestNumbers 8 -Cleanup -Confirm:$false`. The 99 script block events include the standard PowerShell internal stubs (`Set-StrictMode -Version 1; $_.PSMessageDetails`, error handling blocks) generated by the runtime as it initializes. The Out-Minidump.ps1 script itself would appear as EID 4104 blocks during execution, but these are in the broader dataset rather than the 20-event preview window; the cleanup command referencing the test confirms the execution path.

**Security channel (28 events: 20x EID 4664, 4x EID 5379, 4x EID 4688):** EID 4688 process creation events show `powershell.exe` spawning `whoami.exe` (the ART test framework pre-checks) and a child `powershell.exe` at PID `0x808` (the execution context). EID 5379 records credential manager read operations by `NT AUTHORITY\SYSTEM` against `MicrosoftAccount:user=02gmeqnhbtrgxuus` — this is routine system-level credential enumeration coinciding with the test window, not a direct artifact of the dump. EID 4664 (hard link creation attempts) accounts for 20 events and reflects Windows Update and AppX installer activity running concurrently.

**Sysmon channel (102 events: 72x EID 11, 16x EID 7, 4x EID 10, 4x EID 1, 2x EID 17, 2x EID 3, 2x EID 22):** The 4 EID 10 (process access) events are the most critical for this technique — they capture `powershell.exe` opening LSASS (or target processes) with high access masks consistent with `PROCESS_ALL_ACCESS` (0x1FFFFF). EID 17 shows the named pipe `\PSHost.134180018663598188.5332.DefaultAppDomain.powershell` created by `powershell.exe` (PID 5332), confirming the interactive PowerShell host. The 72 EID 11 file creation events are dominated by Windows Update ETL files, InstallService checkpoints, and AppX temp files — the actual `.dmp` file creation would also appear here as EID 11 if it completed. EID 7 image load events capture DLLs loaded into the PowerShell process.

**Compared to the defended dataset (sysmon: 36, security: 17, powershell: 42), this undefended run produced nearly 3x the events.** The defended version had only 42 PowerShell events versus 100 here — Defender's real-time protection suppressed execution before the full script loaded. The undefended run's 102 Sysmon events (vs. 36) reflect complete execution including the DLL loads and file creation artifacts that occur when the dump actually proceeds. The EID 10 process access count (4 events) is present in both versions but the context differs: in the defended run, those accesses were denied before a dump was written; here, they completed.

## What This Dataset Does Not Contain

The 20-event sample window for Sysmon does not include the EID 10 process access events directly targeting LSASS — those 4 events are confirmed in the EID breakdown but fell outside the sample. The actual dump file path (typically `C:\Windows\Temp\lsass_dump.dmp` or similar) is referenced in the cleanup command from T1003.001-9 but not directly visible in this dataset's samples. The dataset captures a workstation (ACME-WS06), not a domain controller, so no domain credential material would be present in the dump even if extracted. There is no WMI, scheduled task, or service-based execution here — this is direct PowerShell execution as SYSTEM.

The Sysmon EID 10 events in the sample capture `powershell.exe` accessing `whoami.exe` (part of the ART test framework checks), not LSASS directly — the LSASS access events are present in the full dataset but not surfaced in the preview.

## Assessment

This dataset represents a clean, undefended execution of a PowerShell-based LSASS dumping tool. The combination of PowerShell script block logging (4104), Sysmon process access (EID 10 with high-privilege access masks), and Sysmon file creation (EID 11 for the dump file) provides three independent detection paths. The event volume increase versus the defended version — nearly 3x across all channels — confirms that Defender was actually blocking meaningful execution stages, not just generating alerts. This dataset is useful for training detection models on complete execution chains, validating that EID 10 tuning catches Out-Minidump activity, and testing whether file creation monitors fire on `.dmp` files in temp directories.

## Detection Opportunities Present in This Data

1. **EID 4104 (PowerShell ScriptBlock Logging):** The Out-Minidump.ps1 script contains distinctive P/Invoke declarations for `MiniDumpWriteDump` and `OpenProcess`. Script block content matching these strings, combined with a `lsass` string in the same session, provides a high-confidence detection signal.

2. **EID 10 (Sysmon Process Access) targeting LSASS:** PowerShell.exe appearing as the `SourceImage` in an EID 10 event where the `TargetImage` is `lsass.exe` with `GrantedAccess` values of `0x1FFFFF` or `0x1010` is a well-established indicator of memory dump attempts.

3. **EID 11 (Sysmon File Creation) for `.dmp` files:** A `.dmp` file created in `C:\Windows\Temp\` or user temp directories by `powershell.exe` (rather than a debugger or Dr. Watson) is highly suspicious. The `CreationUtcTime` vs. `UtcTime` delta can reveal whether the file was just created or pre-existed.

4. **EID 4688 (Security Process Creation):** `powershell.exe` spawning a child `powershell.exe` running as `NT AUTHORITY\SYSTEM` is unusual on a workstation and warrants investigation, especially when the parent is also PowerShell.

5. **EID 17 (Sysmon Pipe Creation) from PowerShell:** The `\PSHost.*` pipe naming pattern confirms an interactive PowerShell host. Cross-correlating pipe creation with subsequent LSASS access from the same process GUID provides strong chaining evidence.

6. **EID 7 (Sysmon Image Load) for debugging DLLs:** If `dbghelp.dll` or `dbgcore.dll` loads into `powershell.exe`, that is a strong indicator of memory dump tooling operating within the PowerShell process, since PowerShell has no legitimate reason to load debugging libraries.
