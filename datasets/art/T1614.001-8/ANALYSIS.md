# T1614.001-8: System Language Discovery — Discover System Language by Windows API Query

## Technique Context

T1614.001 (System Language Discovery) covers adversary enumeration of system language and locale settings. Rather than querying the registry or invoking command-line tools, this test calls Windows API functions directly through a `cmd.exe` invocation that runs a compiled or script-based binary. The Windows API approach — using functions such as `GetSystemDefaultLCID()`, `GetUserDefaultLCID()`, `GetSystemDefaultLocaleName()`, or `GetUserDefaultUILanguage()` — is used by malware that wants to avoid the process-creation artifacts of `reg.exe`, `chcp.com`, or `dism.exe`, and instead perform language checks in-memory with no child process spawned for the query itself.

## What This Dataset Contains

The dataset spans roughly 5 seconds across three log sources (26 Sysmon events, 10 Security events, 34 PowerShell events).

**Sysmon Event 1** (ProcessCreate) records:
- `whoami.exe` — ART test framework identity check (tagged `technique_id=T1033`)
- `cmd.exe` — command shell that runs the API-based language query executable (tagged `technique_id=T1059.003`)

**Security Event 4688** records `cmd.exe` and its child process creation, with full command lines (command-line auditing is enabled). The child process invoked by `cmd.exe` for the API call would appear in the 4688 event for that process.

**Security Event 4689** (process exit) records `cmd.exe` completion, confirming the execution ran to completion.

**Sysmon Event 10** (ProcessAccess) fires on the test framework PowerShell accessing the cmd child process.

**Sysmon Event 7** (ImageLoad) shows .NET runtime and Defender DLL loads into each PowerShell instance.

**Sysmon Event 17** (PipeCreate) captures the PowerShell named pipe for each PowerShell instance.

**PowerShell Events 4103/4104** contain only ART test framework (`Set-ExecutionPolicy Bypass`). The API-based query runs through `cmd.exe` outside PowerShell.

## What This Dataset Does Not Contain

The Windows API calls themselves (`GetSystemDefaultLCID` etc.) are not directly visible in any log source. API call tracing would require an EDR with API hooking, a kernel ETW provider, or application-layer instrumentation — none of which are present here. The result of the language query is not recorded.

Sysmon's include-mode ProcessCreate filter does not have a rule for the specific binary used by this ART test, so if a small compiled tool is invoked, it may not appear in Sysmon Event 1 — only in Security Event 4688. The Security log provides the critical coverage gap fill.

No registry reads, file writes, or network connections are generated by this technique.

## Assessment

This variant is notable for what it does not generate: no registry write, no language-specific process name, no distinctive command-line keyword. The only reliable trace is the process creation event showing `cmd.exe` spawning a tool that calls locale APIs. Without knowing the binary name or hash, detection depends on behavioral context — SYSTEM-context `cmd.exe` spawned from PowerShell with a short-lived child process, in an environment where such patterns are rare. For defenders, this is a useful reminder that API-based techniques often produce minimal log footprint compared to registry or command-line equivalents. Defender was active and did not block this test.

## Detection Opportunities Present in This Data

- **Security Event 4688**: `cmd.exe` spawned from PowerShell as SYSTEM, followed immediately by a short-lived child process — the child process name and command line are key indicators and require command-line auditing to be present.
- **Sysmon Event 1**: `cmd.exe` spawned from `powershell.exe` in a SYSTEM context is unusual on a managed workstation and warrants follow-on investigation.
- **Process lifetime**: The API-query binary exits within milliseconds; a process that starts and exits in under 100ms with no network or file activity is a behavioral indicator of a reconnaissance probe.
- **Correlation with other discovery events**: API-based language checks are often part of a sequence. If `chcp.com`, `reg.exe`, or `dism.exe` locale queries appear in close temporal proximity, the API variant should be treated as part of the same reconnaissance activity.
- **EDR telemetry**: Detection of `GetSystemDefaultLCID`, `GetUserDefaultLCID`, or `GetSystemDefaultLocaleName` API calls from unexpected processes requires an EDR with API monitoring beyond what standard Windows audit logs provide.
