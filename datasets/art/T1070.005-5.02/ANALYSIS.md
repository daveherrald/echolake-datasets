# T1070.005-5: Network Share Connection Removal — Remove Administrative Shares

## Technique Context

Windows automatically creates three administrative shares on every system: `C$` (root of the C: drive), `ADMIN$` (the Windows directory), and `IPC$` (inter-process communication endpoint for remote management). These shares are used extensively by Windows administration tools: PsExec, WMI remote execution, Service Control Manager, and many EDR/SIEM collection agents rely on them to operate remotely.

Removing these shares at runtime (as opposed to suppressing their auto-creation via registry as in T1070.005-4) immediately disrupts any tool or attacker that relies on administrative share access to reach the target system. This is both a defensive hardening action and an attacker technique: adversaries who have established an alternate access path (such as a C2 channel) may remove administrative shares to prevent defenders from using standard remote administrative tools to investigate or remediate the system.

The test uses `cmd.exe` to execute `net share <share> /delete` for all three administrative shares in sequence, then uses the cleanup phase to recreate them. This produces a complete write-then-restore sequence for all three shares, visible across multiple process creation events.

## What This Dataset Contains

The technique execution produces an extensive process creation chain. Security EID 4688 records every step:

Attack phase (share deletion):
- `"cmd.exe" /c for %i in (C$ IPC$ ADMIN$) do net share %i /delete`
- `net share C$ /delete`
- `C:\Windows\system32\net1 share C$ /delete`
- `net share IPC$ /delete`
- `C:\Windows\system32\net1 share IPC$ /delete`
- `net share ADMIN$ /delete`
- `C:\Windows\system32\net1 share ADMIN$ /delete`

Cleanup phase (share recreation):
- `"cmd.exe" /c net share ADMIN$ /UNLIMITED >nul 2>&1 & net share C$=C:\ >nul 2>&1 & net share IPC$ >nul 2>&1`
- `net share ADMIN$ /UNLIMITED`
- `C:\Windows\system32\net1 share ADMIN$ /UNLIMITED`
- `net share C$=C:\`
- `C:\Windows\system32\net1 share C$=C:\`
- `net share IPC$`

Sysmon EID 1 records the same process chains, with `cmd.exe` invocations tagged `technique_id=T1059.003,technique_name=Windows Command Shell` and `net.exe`/`net1.exe` tagged `technique_id=T1018,technique_name=Remote System Discovery` (a Sysmon rule artifact — the actual technique is T1070.005).

The WMI channel contributes 5 EID 5858 (WMI provider error) events recording failed WMI method calls: `Start IWbemServices::ExecMethod - ROOT\Microsoft\Windows\Smb : MSFT_SmbShare::FireShareChangeEvent`. These failures occur because removing `IPC$` disrupts the WMI SMB provider's ability to signal share change events through its normal communication channel. The errors reference `ResultCode = 0x80041007` (WBEM_E_NOT_FOUND / provider not capable). This is a side effect of removing IPC$ — it severs the WMI plumbing that the SmbShare module uses for event notification.

Sysmon EID 11 records a file creation for the PowerShell startup profile cache (`StartupProfileData-Interactive`). Sysmon EID 10 records process access events for the PowerShell instance.

The dataset contains 161 total events: 105 PowerShell (including 2 EID 4103 pipeline events), 16 Security, 32 Sysmon, and 5 WMI.

## What This Dataset Does Not Contain

No Windows Security log events specifically for share deletion or creation (EID 5142 for share deleted, EID 5143 for share created) are present. These events require auditing Object Access → Detailed File Share to be enabled, which was not configured on this system. The Security log captures only the process creation (EID 4688) chains.

No direct record of which users or processes had active connections to the administrative shares at the time of deletion is present. If remote tools were actively using the shares when they were deleted, their sessions would have been disrupted, but that disruption is not logged here.

The WMI EID 5858 errors record that the `MSFT_SmbShare::FireShareChangeEvent` method call failed, but they do not record successful share state changes. They are error events, not confirmation events.

No Defender or behavioral alert events are present. No registry events are generated by runtime share deletion (in contrast to T1070.005-4, which wrote registry values).

## Assessment

This is one of the richer datasets in the T1070.005 series because it captures both the deletion phase (all three administrative shares removed) and the cleanup phase (all three recreated), and includes the unexpected secondary signal of WMI provider errors triggered by the IPC$ removal.

The WMI EID 5858 entries are particularly notable. They are not caused by the test explicitly invoking WMI — they are a side effect of IPC$ removal disrupting the Windows SMB WMI provider. This kind of secondary, cascading telemetry is exactly what you see in real environments when aggressive administrative share removal occurs. It demonstrates that removing IPC$ has observable consequences beyond the share deletion itself.

Compared to the defended variant (24 Sysmon, 22 Security, 26 PowerShell, 3 WMI), the undefended run has more events overall, particularly in PowerShell (105 vs. 26) — consistent with the ART test framework behavior documented across this series. The WMI event counts are similar (5 vs. 3), reflecting the same IPC$-disruption side effect occurring in both variants.

## Detection Opportunities Present in This Data

**`net share C$ /delete`, `net share ADMIN$ /delete`, `net share IPC$ /delete` in quick succession:** Security EID 4688 and Sysmon EID 1 capture each `net.exe`/`net1.exe` invocation individually. The deletion of all three administrative shares within a short time window — especially in a `for %i in (C$ IPC$ ADMIN$)` loop pattern — is a strong indicator of deliberate administrative share removal rather than accidental deletion. The `cmd.exe` command line with the `for` loop is directly captured.

**WMI EID 5858 with `MSFT_SmbShare::FireShareChangeEvent` operation:** These errors are generated automatically when IPC$ is removed, disrupting the WMI SMB event notification path. While the errors themselves do not state that a share was deleted, their sudden appearance on a system — especially concurrent with `net.exe` processes handling share deletion — is a useful corroborating signal. The `ResultCode: 0x80041007` and operation string `FireShareChangeEvent` are specific enough to query for.

**`net1.exe` child processes for share management:** Every `net share /delete` spawns a corresponding `net1.exe` child. Both processes carry the share name in their command lines. Monitoring for `net1.exe` with `share` and `/delete` in the command line (Sysmon EID 1 or Security EID 4688) catches the actual execution regardless of whether monitoring for `net.exe` is in place.

**Share recreation after deletion (cleanup detection):** The cleanup phase is also visible: `net share ADMIN$ /UNLIMITED`, `net share C$=C:\`, `net share IPC$` in Security EID 4688. If you observe administrative shares being deleted and then recreated within the same session (same parent PID chain), this lifecycle pattern indicates testing or scripted activity rather than a permanent security configuration change.
