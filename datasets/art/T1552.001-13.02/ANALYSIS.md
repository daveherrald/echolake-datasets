# T1552.001-13: Credentials In Files — List Credential Files via PowerShell

## Technique Context

T1552.001 (Credentials in Files) includes enumerating the Windows Credential Manager. The Windows Credential Manager stores credentials in DPAPI-encrypted blobs in two locations: `%APPDATA%\Microsoft\Credentials\` (roaming credentials, synchronized to domain controllers) and `%LOCALAPPDATA%\Microsoft\Credentials\` (local credentials). These blobs contain saved passwords for applications, network shares, and websites. The files themselves are encrypted, but listing them reveals which credentials exist, their metadata (file sizes, creation times), and allows an attacker to copy the blobs for offline DPAPI decryption using tools like Mimikatz's `dpapi::cred` module combined with the user's master key.

This test is fundamentally different from the WinPwn series (tests 7–12). It uses native PowerShell cmdlets — specifically `Get-ChildItem -Hidden` — rather than a downloaded external tool. No external download occurs, no AMSI evaluation of malicious content is triggered, and the operation completes (or fails) entirely through built-in Windows functionality. Because Defender's AMSI signatures target known malicious scripts, this technique is generally not blocked by AMSI — only by the absence of credential files at the target path.

In both the defended and undefended variants, the script ran to completion. The difference is that without Defender active, there is no real-time protection response to the file system enumeration attempt.

## What This Dataset Contains

The dataset spans approximately four seconds of telemetry (2026-03-17T17:19:13Z–17:19:17Z) across three log sources, with 158 total events.

**Security EID 4688 — four process creates:**
1. `whoami.exe` (PID 0x3c2c) — ART pre-check
2. Attack `powershell.exe` child (PID 0x4694) with the credential file enumeration command:
   ```
   "powershell.exe" & {$usernameinfo = (Get-ChildItem Env:USERNAME).Value
   Get-ChildItem -Hidden C:\Users\$usernameinfo\AppData\Roaming\Microsoft\Credentials\
   Get-ChildItem -Hidden C:\Users\$usernameinfo\AppData\Local\Microsoft\Credentials\}
   ```
3. `whoami.exe` (PID 0x45ac) — intermediate check
4. Post-cleanup `powershell.exe` (PID 0x4060) with command `"powershell.exe" & {}` — the empty cleanup block, confirming no teardown is needed for this test

The attack PowerShell process (PID 0x4694) ran entirely without restriction. No process kill, no AMSI error, no Defender intervention.

## What This Dataset Does Not Contain

Credential file content is not captured in any log source — Windows Credential Manager blobs are DPAPI-encrypted and their contents would only be meaningful after offline decryption. File read/access auditing is not enabled on this host (object access is set to `none`), so even if credential files had existed and been opened, no Security EID 4663 events would appear.

The `Get-ChildItem` calls returned no results (the machine account running under SYSTEM does not accumulate Credential Manager entries at these user-profile paths), so there are no filenames or metadata visible in the telemetry. A real-world execution as a compromised interactive user would likely find files at these paths and return their sizes and creation timestamps, but no credential content — reading encrypted blobs requires a follow-on tool.

## Assessment

This dataset demonstrates a technique that runs successfully in both the defended and undefended variants — the fundamental difference from the WinPwn series is that `Get-ChildItem` is a built-in cmdlet with no malicious signature. The Sysmon `T1083` tag on the attack process is the only automated classification signal. The real forensic value lies in the PowerShell EID 4103 module log, which records the exact `Get-ChildItem` invocations with their resolved paths, and the Security EID 4688 command line, which captures the full script including the credential path enumeration. The EID 4688 command line is present identically in both defended and undefended runs, making the defended dataset equally useful for this technique from a detection standpoint. The undefended run adds confidence that no protection mechanism would terminate the enumeration.

## Detection Opportunities Present in This Data

1. PowerShell EID 4104 containing `Microsoft\Credentials\` in a `Get-ChildItem` or `dir` call — this specific path is the primary target of Credential Manager file enumeration and has limited legitimate use from a script context.

2. PowerShell EID 4103 recording `Get-ChildItem` with parameter `-Hidden` and a path under `%APPDATA%\Microsoft\Credentials\` — the module log captures parameter bindings verbatim, providing exact path evidence.

3. Security EID 4688 command line containing `AppData\Roaming\Microsoft\Credentials` or `AppData\Local\Microsoft\Credentials` — the path strings are specific enough to be low-false-positive indicators in process creation logs.

4. Sysmon EID 1 tagged `T1083` (File and Directory Discovery) for a `powershell.exe` process whose parent is a script host or ART test framework process — the classification tag surfaces the discovery intent, and correlation with the credential path in the command line elevates it.

5. Correlation of Security EID 4688 showing the `Get-ChildItem Env:USERNAME` environment variable lookup immediately followed by `Get-ChildItem -Hidden` calls using the resolved path — the pattern of resolving the current user context then accessing their credential storage is a specific two-step behavior sequence.

6. Absence of EID 4100 (AMSI block) following a suspicious PowerShell invocation — for techniques that use only native cmdlets like this one, the absence of AMSI telemetry does not indicate successful blocking; it indicates the technique is not covered by AMSI signatures and requires behavioral detection.
