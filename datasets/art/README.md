# Atomic Red Team Telemetry Datasets

1,654 datasets of Windows security telemetry generated by executing [Atomic Red Team](https://github.com/redcanaryco/atomic-red-team) tests on instrumented Windows 11 Enterprise workstations joined to an Active Directory domain.

Each dataset captures the telemetry from a single ART test execution — process creation, registry modification, network connections, script execution, and other system activity as recorded by Sysmon, Windows Security audit logging, and PowerShell logging.

## Structure

```
T1003.001-2/           # Technique T1003.001, ART test 2
  dataset.yaml         # Metadata, environment config, MITRE mapping
  ANALYSIS.md          # What the dataset contains and detection opportunities
  README.md            # Quick summary
  data/
    sysmon.jsonl       # Sysmon operational events
    security.jsonl     # Windows Security events
    powershell.jsonl   # PowerShell operational events
    ...                # Other channels as applicable
```

## Naming Convention

- `T1003.001-2` — MITRE ATT&CK technique T1003.001 (LSASS Memory), ART test number 2
- `T1003-2.02` — Same technique and test, second run in a different environment

Some techniques include sub-technique IDs (e.g., T1003.001) and some do not (e.g., T1003). This follows the MITRE ATT&CK taxonomy directly.

## Environments

Datasets were collected from two Windows 11 Enterprise Evaluation workstations in the `acme.local` lab domain:

- **ACME-WS02** — Windows Defender fully enabled (real-time protection, behavior monitoring, AMSI). Many techniques are blocked before completion, producing attempt telemetry but not success telemetry.
- **ACME-WS06** — Windows Defender disabled via Group Policy. Techniques execute to completion, producing the full behavioral footprint.

The endpoint protection configuration for each dataset is described in `dataset.yaml` under `environment.windows.endpoint_protection`.

## Instrumentation

Both hosts share identical logging configuration.

### Sysmon v15.15

Config: [sysmon-modular](https://github.com/olafhartong/sysmon-modular) by Olaf Hartong
Hash: `SHA256=4516404FA30EE87CEA558567820CDC78863CC4AB07889519E49EAC3CCA92E0D2`
Hashing: SHA1, MD5, SHA256, IMPHASH

The sysmon-modular config is a balanced, community-maintained configuration designed for production use. It uses a combination of include and exclude rules per event type:

| EID | Event Type | Mode | Notes |
|-----|-----------|------|-------|
| 1 | ProcessCreate | include + exclude | **Filtered** — only processes matching known-suspicious patterns (LOLBins, accessibility bypass, fltmc, attrib, sc, etc). Not a blanket capture. Security 4688 provides complementary full coverage. |
| 2 | FileCreateTime | include + exclude | Timestomp patterns. |
| 3 | NetworkConnect | include + exclude | Suspicious outbound connections only. Standard OS traffic excluded. |
| 5 | ProcessTerminate | include | Specific processes only. |
| 6 | DriverLoad | exclude | All except known-benign (allowlist). |
| 7 | ImageLoad | include + exclude | **Filtered** — DLL loads matching known-suspicious patterns only (injection indicators, AMSI, .NET, Defender). |
| 8 | CreateRemoteThread | include + exclude | Remote thread creation. Broad coverage. |
| 9 | RawAccessRead | include | Raw disk access. |
| 10 | ProcessAccess | include + exclude | **Filtered** — cross-process access with suspicious GrantedAccess masks only. |
| 11 | FileCreate | include + exclude | Suspicious file creation patterns only. |
| 12/13/14 | RegistryEvent | include + exclude | **Filtered** — known-sensitive keys only (Run keys, services, etc). |
| 15 | FileCreateStreamHash | include | Alternate data streams. |
| 17/18 | PipeEvent | include + exclude | Known-suspicious named pipe patterns. |
| 19/20/21 | WmiEvent | include + exclude | WMI persistence (consumers, filters, bindings). |
| 22 | DnsQuery | exclude | All except known-benign (allowlist). |
| 23 | FileDelete | include | Specific patterns. |
| 24 | ClipboardChange | include | Clipboard content changes. |
| 25 | ProcessTampering | exclude | Process hollowing/herpaderping. |
| 27 | FileBlockExecutable | include | Blocked executable drops. |
| 28 | FileBlockShredding | include | File shredding attempts. |
| 29 | FileExecutableDetected | include + exclude | Executable file creation. |

**Key implication**: ProcessCreate (EID 1) is filtered. Many process creations — including some attack tools — will be absent from Sysmon. Security 4688 with command-line auditing captures all process creation and is the complementary data source.

### Windows Audit Policy

Configured via Advanced Audit Policy (`auditpol`):

**Audited:**
| Category | Subcategories | Setting |
|----------|--------------|---------|
| Logon/Logoff | Logon, Logoff, Account Lockout, Special Logon, Group Membership, all IPsec, NPS, User/Device Claims | Success and Failure |
| Detailed Tracking | Process Creation, Process Termination, DPAPI, RPC, PnP, Token Right Adjusted | Success |
| Account Logon | Kerberos Service Ticket, Kerberos Auth, Credential Validation, Other | Success and Failure |

**Not audited** (events in these categories will not appear):
- Object Access (File System, Registry, Kernel Object, SAM)
- Privilege Use
- Policy Change (Audit Policy, Auth/Authz Policy, Firewall)
- Account Management (User, Computer, Security Group)
- DS Access

**Key implications:**
- **4688 (Process Creation)** with command-line logging is the single most important data source. Every process creation is captured with full command line.
- **4689 (Process Termination)** captures exit codes — `0xC0000022` (ACCESS_DENIED) indicates Defender blocked the technique.
- **4703 (Token Right Adjusted)** captures privilege changes for escalation detection.
- **4624/4625/4634 (Logon)** are fully audited for lateral movement visibility.
- **Object Access is NOT audited** — no 4663/4656/4658 events. Sysmon EID 11 (file create) and 12/13/14 (registry) partially fill this gap.
- **Account Management is NOT audited** — techniques creating local accounts (T1136.001) won't produce Security events.

### PowerShell Logging

| Setting | Effect |
|---------|--------|
| ScriptBlock Logging (4104) | Captures script content. Does NOT capture arguments to external processes launched via `cmd.exe` or `Start-Process`. |
| ScriptBlock Invocation Logging | Script block start/stop events. |
| Module Logging (4103, all modules) | Cmdlet invocations with parameters. |
| Transcription | Full session transcripts to disk. |

When AMSI blocks a script before execution, ScriptBlock Logging captures only PowerShell engine initialization — not the intercepted content.
