# T1018-22: Remote System Discovery — Remote System Discovery - Enumerate Remote Hosts with Netscan

## Technique Context

T1018 Remote System Discovery involves adversaries attempting to get a listing of other systems by IP address, hostname, or other logical identifier on a network that may be used for lateral movement. This technique is commonly used during the discovery phase of an attack to map out the network topology and identify potential targets for further exploitation.

The Atomic Red Team test T1018-22 specifically uses the Netscan utility, a third-party network scanning tool, to perform host discovery on a specified IP range. Netscan is a legitimate network administration tool that can perform ping sweeps, port scans, and basic host enumeration. Attackers often use legitimate network tools like this to blend in with normal administrative activities while conducting reconnaissance.

Detection engineers typically focus on monitoring for network scanning behavior, execution of known reconnaissance tools, and suspicious process chains involving command-line network utilities. This technique generates telemetry across multiple data sources including process creation, command-line logging, and potentially network flow data.

## What This Dataset Contains

This dataset captures the execution of Netscan through PowerShell with the following key activities:

**Process Chain Evidence:**
- Security EID 4688 shows PowerShell process creation with command line: `"powershell.exe" & {cmd /c 'C:\AtomicRedTeam\atomics\..\ExternalPayloads\netscan\64-bit\netscan.exe' /hide /auto:\"\"$env:temp\T1018NetscanOutput.txt\"\" /range:'127.0.0.1-127.0.0.1'}`
- Security EID 4688 shows cmd.exe spawned with: `"C:\Windows\system32\cmd.exe" /c C:\AtomicRedTeam\atomics\..\ExternalPayloads\netscan\64-bit\netscan.exe /hide /auto:C:\Windows\TEMP\T1018NetscanOutput.txt /range:127.0.0.1-127.0.0.1`
- Sysmon EID 1 captures the same process creations with full command lines and parent-child relationships

**PowerShell Script Block Logging:**
- PowerShell EID 4104 captures the script block: `& {cmd /c 'C:\AtomicRedTeam\atomics\..\ExternalPayloads\netscan\64-bit\netscan.exe' /hide /auto:"$env:temp\T1018NetscanOutput.txt" /range:'127.0.0.1-127.0.0.1'}`
- Additional PowerShell EID 4103 shows Set-ExecutionPolicy bypass operations

**File Creation Activity:**
- Sysmon EID 11 shows PowerShell profile file creation activity but notably does not show the creation of the output file `T1018NetscanOutput.txt`

**Process Access Events:**
- Multiple Sysmon EID 10 events show PowerShell accessing child processes with full access rights (0x1FFFFF)

## What This Dataset Does Not Contain

**Missing Netscan Process Creation:** The actual netscan.exe process creation is not captured in Sysmon EID 1 events, likely because the sysmon-modular configuration uses include-mode filtering for ProcessCreate events and netscan.exe doesn't match the known-suspicious patterns.

**Missing Network Activity:** No DNS queries, network connections, or other network-related telemetry from the actual scanning activity is present, despite Sysmon being configured to capture network events.

**Missing Tool Output:** The dataset does not contain evidence of the output file creation (`T1018NetscanOutput.txt`) that should have been generated by the netscan tool, suggesting either the scan failed or the file was cleaned up before Sysmon could log it.

**Missing Scan Results:** Since the test scanned only the loopback address (127.0.0.1-127.0.0.1), this represents a minimal test case rather than realistic network reconnaissance behavior.

**Exit Code Information:** While Security events show cmd.exe processes exiting with status 0x1 (indicating failure), there's no detailed error information about why the netscan execution may have failed.

## Assessment

This dataset provides excellent process execution and command-line telemetry for detecting the invocation of network reconnaissance tools through PowerShell. The Security 4688 events with command-line logging and Sysmon ProcessCreate events capture the full attack chain including tool paths, parameters, and output file specifications.

However, the dataset's utility is limited by the apparent failure or incomplete execution of the actual scanning activity. The lack of network telemetry and the restricted localhost scan range make this more suitable for detecting the process execution patterns rather than the network reconnaissance behavior itself.

The PowerShell script block logging provides strong evidence for PowerShell-based tool execution, though most events are test framework boilerplate. The process access events demonstrate PowerShell's interaction with spawned processes, which could be useful for behavioral detection.

## Detection Opportunities Present in This Data

1. **Command-line detection for netscan.exe execution** - Monitor Security 4688 or Sysmon 1 events for processes containing "netscan.exe" with scanning parameters like "/range" and "/auto"

2. **PowerShell invoking network reconnaissance tools** - Detect PowerShell script blocks (EID 4104) containing network scanning tool execution patterns with cmd /c invocation

3. **Suspicious process chains** - Alert on PowerShell spawning cmd.exe with command lines containing network scanning tools and output redirection parameters

4. **Network tool output file creation** - Monitor for file creation events with naming patterns like "NetscanOutput.txt" or similar reconnaissance tool output files

5. **Process access patterns** - Detect PowerShell processes accessing multiple child processes with full access rights, potentially indicating tool orchestration

6. **Command-line parameter analysis** - Look for scanning-related parameters like "/hide", "/auto", "/range" combined with IP address ranges in process command lines

7. **PowerShell execution policy bypass** - Correlate Set-ExecutionPolicy bypass operations with subsequent network tool execution as potential attack staging
