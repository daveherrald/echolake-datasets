# T1614.001-7: System Language Discovery — Discover System Language with dism.exe

## Technique Context

T1614.001 (System Language Discovery) covers adversary enumeration of configured language and locale settings on a target system. Adversaries use locale data to tailor payloads — many malware families check whether the system language indicates a CIS country and exit rather than execute in order to avoid prosecution in those jurisdictions. The Deployment Image Servicing and Management tool (`dism.exe`) is a Windows inbox utility primarily used for image servicing, but its `/online /Get-Intl` flag queries currently installed language packs and locale configuration on a running system. Because `dism.exe` is a signed Microsoft binary that IT administrators use legitimately, its invocation for locale discovery can blend with administrative activity. The sysmon-modular configuration applies the tag `technique_id=T1548.002` to `dism.exe` invocations due to a known UAC bypass path involving DISM — note that this tag is a false-positive label in the locale-discovery context, though the process creation event is still correctly captured.

## What This Dataset Contains

The dataset spans approximately 5 seconds across two log sources: 107 PowerShell events and 22 Security events. Sysmon events are absent (see below).

**Security log (22 events)** provides the primary execution record:

- **EID 4688** (7 events) — Process creation with command-line auditing. Key process chain: `powershell.exe` (test framework) spawns `whoami.exe` (identity check), then spawns `cmd.exe` with the command line `"cmd.exe" /c dism.exe /online /Get-Intl`. `cmd.exe` then spawns `C:\Windows\System32\Dism.exe` with `dism.exe /online /Get-Intl`, and `Dism.exe` in turn spawns `C:\Windows\System32\Dism\DismHost.exe` (its internal worker process).
- **EID 4689** (11 events) — Process exit events for each process in the chain, including both `Dism.exe` and `DismHost.exe` exiting with status `0x0` (success), confirming the language query completed.
- **EID 4624** (1 event) — A service logon (Type 5) for NT AUTHORITY\SYSTEM. DISM's worker process generates a new logon session, which is atypical and distinguishes this technique from simpler enumeration commands.
- **EID 4627** (1 event) — Group membership enumeration associated with the logon generated by DISM internals.
- **EID 4672** (1 event) — Special privileges assigned to the new SYSTEM logon session, including `SeAssignPrimaryTokenPrivilege`, `SeTcbPrivilege`, `SeDebugPrivilege`, and other high-privilege tokens — confirming execution at SYSTEM integrity level.
- **EID 4703** (1 event) — Token right adjustment for the PowerShell process (`SeSecurityPrivilege` disabled).

The complete process chain — `powershell.exe` → `cmd.exe` → `Dism.exe` → `DismHost.exe` — is visible in the Security log, with parent-child relationships captured through the Creator Process ID fields.

**PowerShell log (107 events)** — 104 EID 4104 script block events and 3 EID 4103 module pipeline events. The EID 4104 blocks are PowerShell runtime infrastructure fragments (`Set-StrictMode`, `$_.PSMessageDetails`, etc.) and `Set-ExecutionPolicy Bypass -Scope Process -Force`. The DISM invocation itself runs through `cmd.exe`, outside the PowerShell process, so it does not appear in script block logging.

## What This Dataset Does Not Contain

**Sysmon events are absent** from this dataset. The sysmon-modular include-mode ProcessCreate configuration does have a rule that fires on `dism.exe` (via the UAC-bypass tag), so Sysmon EID 1 events for `cmd.exe` and `dism.exe` would be expected. Their absence from this dataset likely reflects a collection window gap or a per-test Sysmon channel isolation issue, not a coverage failure in the Sysmon configuration itself — the defended variant captured 27 Sysmon events for the identical technique.

The output of `dism /online /get-intl` — the actual locale and language pack data returned — is not present in any log channel. DISM writes its results to the console, not to an auditable file or registry key, so you will not find the discovered locale value in these events.

## Assessment

The undefended execution produces a broadly similar telemetry profile to the defended variant, with one important addition: the undefended run captures the full execution chain through to success (both `Dism.exe` and `DismHost.exe` exit with `0x0`), while in the defended run Defender's behavior monitoring may have introduced additional noise or briefly delayed execution. The Security log event counts are comparable (22 undefended vs. 18 defended), with the extra events here attributable to the logon/privilege sequence generated by DISM's worker process.

The distinctive characteristic of this technique is the logon session DISM creates (EID 4624 Type 5 + EID 4672) — this is unusual for a command-line enumeration tool and provides a secondary detection anchor beyond the process name alone. You will find the full process tree in the EID 4688 events, with parent process IDs linking each step.

This is a useful, well-populated dataset for the technique. The absence of Sysmon events is the primary gap, but the Security log coverage is complete.

## Detection Opportunities Present in This Data

- **EID 4688: `Dism.exe` with `/Get-Intl` or `/get-intl` in the command line** — The command-line argument `dism.exe /online /Get-Intl` (case-insensitive) is a high-specificity indicator. Legitimate administrative use of DISM for locale queries from non-administrative processes or non-IT accounts is rare. Alert on `NewProcessName` containing `Dism.exe` and `CommandLine` containing `/Get-Intl` or `/get-intl`.

- **EID 4688: `cmd.exe` spawned by `powershell.exe` with `dism.exe` in the command line** — The pattern of PowerShell spawning `cmd.exe` which then invokes `dism.exe` for locale discovery is detectable via the Creator Process Name / New Process Name combination in back-to-back 4688 events sharing process IDs.

- **EID 4624 Type 5 logon + EID 4672 generated by `Dism.exe`** — A service logon (LogonType=5) immediately following `Dism.exe` process creation (correlate by timestamp and logon session ID) is an unusual pattern. This secondary logon event provides detection coverage even if command-line auditing is disabled.

- **EID 4689: `DismHost.exe` process exit** — The spawning of `C:\Windows\System32\Dism\DismHost.exe` is specific to DISM servicing operations. Its appearance in process exit events (which always log the process name) can alert on DISM execution even when process creation auditing is incomplete.

- **Correlation: `whoami.exe` → `cmd.exe` → `dism.exe` within a short time window** — The ART-canonical sequence of an identity check (`whoami.exe`) immediately before the technique-specific command is a pattern worth modelling, though this correlation is most useful for sandbox or detonation environments rather than production detection.
