# T1135-6: Network Share Discovery — View available share drives

## Technique Context

T1135 Network Share Discovery is a foundational discovery technique where adversaries enumerate network shares to identify accessible resources, potential lateral movement targets, and sensitive data repositories. The `net share` command is one of the most common and straightforward methods for this discovery on Windows systems. Detection engineers focus on monitoring process creation of network enumeration tools, command-line patterns indicating share discovery, and the process chains that execute these commands. This technique is often part of initial reconnaissance after gaining access to a system, helping adversaries understand the network topology and available resources.

## What This Dataset Contains

This dataset captures a clean execution of `net share` discovery through PowerShell. The Security 4688 events show the complete process chain: PowerShell (PID 15920) → cmd.exe with command line `"cmd.exe" /c net share` (PID 44220) → net.exe with command line `net share` (PID 42948) → net1.exe with command line `C:\Windows\system32\net1 share` (PID 34636). The Sysmon events provide additional context with EID 1 process creation events tagged with relevant MITRE techniques (T1059.003 for cmd.exe and T1018 for both net.exe and net1.exe), and EID 10 process access events showing PowerShell accessing both the cmd.exe and net.exe child processes. The PowerShell channel contains only test framework boilerplate (Set-ExecutionPolicy Bypass and Set-StrictMode scriptblocks).

## What This Dataset Does Not Contain

The dataset lacks the actual output of the `net share` command - there's no indication of what shares were discovered or network traffic generated by the enumeration. No Sysmon EID 3 network connection events are present, suggesting the share enumeration only queried local shares rather than remote systems. The dataset also doesn't contain any file access events showing potential follow-up activities like accessing discovered shares. Additionally, there are no registry access events that might accompany more sophisticated share discovery techniques.

## Assessment

This dataset provides excellent coverage for detecting basic network share discovery through the standard `net share` command execution path. The Security 4688 events with full command-line logging capture the technique definitively, while Sysmon process creation events add valuable context with MITRE technique tagging. The process access events from Sysmon EID 10 provide additional behavioral indicators. The clean execution without interference from Windows Defender makes this ideal for understanding normal technique telemetry patterns. However, the dataset's limitation to local share enumeration means it doesn't demonstrate network-based discovery activities that would generate additional detection opportunities.

## Detection Opportunities Present in This Data

1. **Command-line pattern detection** - Security 4688 events contain the exact command line `"cmd.exe" /c net share` and `net share`, enabling regex-based detection of share enumeration commands.

2. **Process chain analysis** - The PowerShell → cmd.exe → net.exe → net1.exe execution chain in Security 4688 events provides a behavioral signature for scripted network discovery.

3. **MITRE technique correlation** - Sysmon EID 1 events tag the processes with T1018 (Remote System Discovery), enabling detection rules based on technique-specific process creation.

4. **Parent-child process relationships** - Sysmon ProcessGuid and ParentProcessGuid fields enable detection of suspicious parent processes launching network enumeration tools.

5. **Elevated privilege context detection** - Security 4688 events show TokenElevationTypeDefault with System integrity level, indicating privileged discovery activity.

6. **Process access behavioral indicators** - Sysmon EID 10 events show PowerShell accessing spawned child processes with full access rights (0x1FFFFF), indicating process injection-style behavior.

7. **Execution from temp directories** - Both net.exe and net1.exe executed from C:\Windows\Temp\, providing a file system location-based detection opportunity.
