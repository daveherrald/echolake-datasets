# T1112-81: Modify Registry — Modify UseTPMKeyPIN Registry Entry

## Technique Context

T1112 (Modify Registry) is used here to modify `UseTPMKeyPIN` to `2` in `HKLM\SOFTWARE\Policies\Microsoft\FVE`. The `UseTPMKeyPIN` value controls whether BitLocker requires a PIN alongside the TPM and startup key (a USB key containing the BitLocker key). Setting it to `2` moves the policy to "Allow TPM+Startup Key+PIN," weakening any stronger requirement (such as a mandatory PIN configuration) that may have been set by Group Policy.

This test is one of three FVE policy modifications in this batch (T1112-79, T1112-81, T1112-83), each targeting a different BitLocker authentication factor control. Together they represent a systematic approach to weakening BitLocker authentication requirements across multiple dimensions: TPM+PIN (`UseTPMPIN`), TPM+Startup Key+PIN (`UseTPMKeyPIN`), and partial encryption keys (`UsePartialEncryptionKey`). An attacker who applies all three in sequence leaves the system in a state where BitLocker's strongest authentication requirements have been individually relaxed.

This dataset is also notable for its unusual Security channel composition. The presence of EID 4799 and 4798 events (local group membership enumeration) alongside the standard EID 4688 process creation events indicates that the test execution triggered WMI-based activity that the test framework or system generated as part of the execution context.

## What This Dataset Contains

This dataset captures the `UseTPMKeyPIN` registry modification on a Windows 11 Enterprise domain workstation with Defender disabled. Events occur at approximately 2026-03-17T16:35:37Z to 16:35:40Z, in the same session as T1112-74, T1112-75, T1112-79, T1112-8, and T1112-83.

The attack chain is PowerShell (SYSTEM) → cmd.exe → reg.exe. Security EID 4688 provides the clearest evidence of the execution chain:

- `cmd.exe` (PID 0x4368) launched by `powershell.exe` (PID 0x43fc) with command line: `"cmd.exe" /c reg add "HKLM\SOFTWARE\Policies\Microsoft\FVE" /v UseTPMKeyPIN /t REG_DWORD /d 2 /f`
- `reg.exe` (PID 0x43b0) launched by `cmd.exe` (PID 0x4368) with command line: `reg  add "HKLM\SOFTWARE\Policies\Microsoft\FVE" /v UseTPMKeyPIN /t REG_DWORD /d 2 /f`

Both run as `NT AUTHORITY\SYSTEM` at `Mandatory Label: S-1-16-16384`.

A distinguishing feature of this dataset is the WMI activity. Sysmon EID 1 captures `WmiPrvSE.exe` (PID 16380, ProcessGuid `{9dc7570a-82d9-69b9-fc39-000000000900}`, RuleName `technique_id=T1047`) launching as `NT AUTHORITY\NETWORK SERVICE` with `C:\Windows\system32\wbem\wmiprvse.exe -Embedding`. The corresponding Security EID 4688 shows `WmiPrvSE.exe` (PID 0x3ffc) created by `svchost.exe` (PID 0x39c) under `LogonId: 0x3E4` (NETWORK SERVICE).

The Security channel contains 29 events—substantially more than the 4-event baseline seen in simpler tests—broken down as EID 4799: 19 events (user security-enabled local group membership enumeration), EID 4688: 5 events, EID 4798: 5 events (user local group membership enumeration). The 19 EID 4799 and 5 EID 4798 events are generated by `WmiPrvSE.exe` enumerating local group membership, which is a side effect of the WMI call made during this test execution. This enumeration activity is captured in the Security log because the system has auditing enabled for Account Management events.

The Sysmon EID breakdown (7: 10, 1: 5, 10: 3, 17: 1) includes 10 DLL load events from the PowerShell process and the WMI host process.

The PowerShell channel contains 96 EID 4104 events—consistent with the elevated counts seen in the March 17 session. The cleanup wrapper `Invoke-AtomicTest T1112 -TestNumbers 81 -Cleanup` is visible.

## What This Dataset Does Not Contain

No Sysmon EID 13 (registry value set) appears in the EID breakdown or sample subset, which is unusual—this is the only test in the batch that lacks a captured EID 13. The registry write may have been performed through the WMI path rather than through `reg.exe` directly, or the EID 13 may have been generated but missed by the collection window or filtered during dataset packaging.

No BitLocker-specific events (TPM status changes, BitLocker driver events, or `manage-bde.exe` activity) are present. The modification is to a policy value that controls future BitLocker behavior, not to the current encryption state.

## Assessment

The undefended dataset (Sysmon: 19, Security: 29, PowerShell: 96) presents a dramatically different profile from the defended variant (Sysmon: 17, Security: 14, PowerShell: 35). The Security channel is larger in the undefended run (29 vs. 14)—the only test in this batch where the undefended Security count exceeds the defended count. This inversion is explained by the WMI activity: the 19 EID 4799 events from group membership enumeration by `WmiPrvSE.exe` inflate the undefended Security count. In the defended run, Defender's monitoring may have intercepted or altered the WMI execution path such that fewer enumeration events were generated.

The Sysmon counts are nearly equal (17 defended vs. 19 undefended), consistent with the WMI process creation events dominating the Sysmon count in both variants.

This dataset is the most structurally complex in the batch and provides the richest view of secondary activity (WMI, group enumeration) that co-occurs with registry modification in a real test environment.

## Detection Opportunities Present in This Data

**Process creation command line (Security EID 4688):** The command lines for both `cmd.exe` and `reg.exe` writing `UseTPMKeyPIN=2` to the FVE key are captured. This is the primary technique evidence and the most reliable detection point.

**WMI host activity (Sysmon EID 1 / Security EID 4688):** The `WmiPrvSE.exe -Embedding` process creation from `svchost.exe` in the same time window as the registry modification is secondary but correlated evidence. WMI execution coincident with registry writes to security policy keys is worth investigating.

**Local group enumeration (Security EID 4799 / EID 4798):** The 19 EID 4799 and 5 EID 4798 events from `WmiPrvSE.exe` represent a discovery activity co-occurring with the registry modification. This combination—group membership enumeration immediately followed by security policy registry writes—is a recognizable pre-staging pattern.

**FVE key namespace (full dataset):** Monitoring `HKLM\SOFTWARE\Policies\Microsoft\FVE` for any write from non-Group Policy service processes provides a unified detection for T1112-79, T1112-81, and T1112-83 simultaneously.
