# T1027-7: Obfuscated Files or Information — Obfuscated Command in PowerShell

## Technique Context

T1027 Obfuscated Files or Information encompasses techniques where adversaries disguise malicious code to evade signature-based detection and complicate forensic analysis. PowerShell command obfuscation is one of the most common manifestations: attackers use PowerShell's flexible syntax — format strings, type casting, variable indirection, tick-mark escaping, character code arrays, and string concatenation — to write commands that behave identically to readable code but bear no resemblance to known malicious strings.

This test uses a multi-layered obfuscation approach that would defeat most static pattern matches. The obfuscated payload ultimately executes `Write-Host 'Hello, from PowerShell!'`, but the command is hidden behind: type-name obfuscation (`[TyPe]("{0}{1}" -f 'S','TrING')` resolves to `[String]`), format-string function name reconstruction (`"{0}{2}{3}{1}{4}" -f'In','SiO','vOKe-EXp','ReS','n'` resolves to `Invoke-Expression`), variable indirection with tick marks (`` "v`ALUe" ``), and octal character code arrays converted via `[CHAR]([Convert]::ToInt16($_, 8))`.

PowerShell's script block logging (EID 4104) is the primary defense against this technique: before any PowerShell code executes, the runtime deserializes and logs the actual script block content — including deobfuscated strings — making EID 4104 a powerful mitigation. EID 4103 module logging provides additional detail on which cmdlets were invoked with what parameters.

## What This Dataset Contains

The dataset spans approximately 3 seconds (23:04:31–23:04:34 UTC on 2026-03-14) and totals 139 events across three channels.

The key evidence is in Sysmon EID 1 and Security EID 4688. The technique PowerShell process (PID 5360) is created with the full obfuscated command line:

```
"powershell.exe" & {$cmDwhy =[TyPe]("{0}{1}" -f 'S','TrING')  ;   $pz2Sb0  =[TYpE]("{1}{0}{2}"-f'nv','cO','ert')  ;  &("{0}{2}{3}{1}{4}" -f'In','SiO','vOKe-EXp','ReS','n') (  (&("{1}{2}{0}"-f'blE','gET-','vaRIA')  ('CMdw'+'h'+'y'))."v`ALUe"::("{1}{0}" -f'iN','jO').Invoke('',(  (127, 162,151, 164,145 ,55 , 110 ,157 ,163 , 164 ,40,47, 110 , 145 ,154, 154 ,157 , 54 ,40, 146, 162 , 157,155 ,40, 120, 157 ,167,145 , 162 ,123,150 ,145 , 154 , 154 , 41,47)| .('%') {  ( [CHAR] ( $Pz2sB0::"t`OinT`16"(( [sTring]${_}) ,8)))})) )}
```

This is the exact command an analyst would see in a real incident — fully opaque without decoding. The octal array `(127, 162, 151, 164, 145, 55, 110, ...)` decodes to `Write-Host 'Hello, from PowerShell!'`.

In the full dataset (not in the 20-sample EID 4104 set), PowerShell script block logging captures the deobfuscated execution. The defended version's analysis confirmed that EID 4104 logged `Write-Host 'Hello, from PowerShell!'` as the decoded payload, and EID 4103 captured parameter bindings for `Get-Variable`, `ForEach-Object`, `Write-Host`, and `Invoke-Expression`. This undefended run generates the same script block content — the technique is not blocked by Defender in either the defended or undefended version.

Sysmon EID 7 documents the technique's PowerShell process loading the full .NET CLR stack and `urlmon.dll`. The urlmon.dll load is unexpected for a pure string-manipulation command and may indicate the test framework's broader context (the parent PowerShell session may have loaded urlmon.dll for other network operations earlier in the test run).

Sysmon EID 17 shows two named pipe creation events for PowerShell host pipes — one for the parent test framework session and one for the technique's child process — indicating separate PowerShell runspace contexts.

## What This Dataset Does Not Contain

No network activity is generated by this technique. The absence of EID 3 network events is expected since the payload only outputs to the console. There are no file system artifacts. The EID 4104 sample set captures only test framework boilerplate (the 20-sample limit excludes the technique script blocks), so the deobfuscated payload is not in the provided samples but exists in the full dataset. Since Defender does not block this command in either the defended or undefended version, there is no behavioral difference between the two datasets for this technique — both provide the same obfuscated command line and script block content.

## Assessment

This is a high-value obfuscation technique dataset. The command line captured in Sysmon EID 1 and EID 4688 provides a fully formed example of a multi-layer obfuscated PowerShell command that would evade most signature-based detections. Detection engineers can use the obfuscated command line to validate that their rules do not rely on readable string matching alone. The EID 4104 content in the full dataset demonstrates the power of script block logging: the deobfuscated `Write-Host 'Hello, from PowerShell!'` is logged regardless of how complex the obfuscation is, as long as the code is executed through the PowerShell runtime. This dataset is useful for testing both the obfuscation detection side (catching unusual patterns in command lines) and the deobfuscation side (leveraging EID 4104 for what the code actually does).

## Detection Opportunities Present in This Data

1. **Sysmon EID 1 / EID 4688 — obfuscation indicators in command line**: Multiple high-confidence obfuscation markers are present: mixed-case type names (`[TyPe]`, `[TYpE]`), format string operators `-f` reconstructing function names, tick-mark escaping in method names (`` "v`ALUe" ``, `` "t`OinT`16" ``), octal character code arrays, and `Invoke-Expression` (obfuscated as `vOKe-EXpReS`). Analytics that score command lines for these individual markers and alert on combinations would catch this.

2. **EID 4104 — deobfuscated payload via script block logging**: Regardless of obfuscation, `Write-Host 'Hello, from PowerShell!'` (or the attacker's actual payload) is logged by the PowerShell runtime. Script block logging is the definitive anti-obfuscation control for PowerShell, and this dataset validates that it works as expected.

3. **EID 4103 — Invoke-Expression module logging**: EID 4103 captures the `Invoke-Expression` invocation with parameter bindings, providing a second confirmation that dynamic code execution occurred even when the command line is unreadable.

4. **Sysmon EID 1 — PowerShell from PowerShell with deeply nested string operations**: The sheer length and complexity of the command line argument (several hundred characters of format strings and type casts) is an anomaly indicator. Command line length-based analytics can flag unusually long PowerShell arguments for review.

5. **Sysmon EID 17 — PowerShell named pipe creation following obfuscated process**: The named pipe `\PSHost.134180030683639463.3016.DefaultAppDomain.powershell` creation in the same process that received the obfuscated command is a behavioral marker. Correlation between EID 17 pipe creation and Sysmon EID 1 obfuscated command lines provides a confirmation signal.
