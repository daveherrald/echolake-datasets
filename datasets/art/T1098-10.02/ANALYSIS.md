# T1098-10: Account Manipulation — Domain Password Policy Check: Short Password

## Technique Context

T1098 (Account Manipulation) covers adversary modification of account credentials, permissions, or authentication settings to maintain access or escalate privileges. This specific test attempts to set a domain user's password to a deliberately short, weak password (`Uplow-1`) — one that would typically be rejected by standard domain password policy — to test whether the password policy is enforced and to demonstrate what unauthorized password changes look like in the event log.

In real attack scenarios, password manipulation serves several goals: maintaining persistence by changing a compromised account's password before the victim resets it, creating fallback accounts with known credentials, testing whether a domain's minimum password length and complexity requirements are actually enforced (some misconfigured domains have policy set but not enforced), and demonstrating to the victim organization that their account security posture is weaker than believed.

The technique uses `Set-ADAccountPassword` — a PowerShell Active Directory module cmdlet — with the caller's own credentials stored in a local file (`$env:LOCALAPPDATA\AtomicRedTeam\$env:USERNAME.txt`). This credential-file-to-AD-password-change pattern is realistic: attackers often stage credentials in accessible locations and use them via PowerShell to make programmatic account changes without interactive prompts.

## What This Dataset Contains

The dataset spans approximately ten seconds (2026-03-14T23:39:09Z–~23:39:20Z) on ACME-WS06.acme.local and contains 148 events across three channels.

**The core execution command** appears in both Security EID 4688 and Sysmon EID 1. The full command line (truncated in the Security event, fully captured in Sysmon) is:

```powershell
"powershell.exe" & {
  $credFile = "$env:LOCALAPPDATA\AtomicRedTeam\$env:USERNAME.txt"
  if (Test-Path $credFile) {
    $cred = New-Object -TypeName System.Management.Automation.PSCredential -ArgumentList $env:USERNAME, (Get-Content $credFile | ConvertTo-SecureString)
    if($cred.GetNetworkCredential().Password -eq "Uplow-1") {
      Write-Host -ForegroundColor Yellow "The new password is the same as the password stored in the credential file. Please specify a different new password."
      exit -1
    }
    try {
      $newPassword = ConvertTo-SecureString Uplow-1 -AsPlainText -Force
      Set-ADAccountPassword -Identity $env:USERNAME -OldPassword $cred.password -NewPassword $newPassword
    }
    catch {
      $_.Exception
      $errCode = $_.Exception.ErrorCode
      ...
    }
  }
}
```

This script: reads stored credentials from a local file, validates the new password isn't identical to the stored one, attempts `Set-ADAccountPassword` to change the password to `Uplow-1`, and catches exceptions (which would include a domain policy rejection of the short password).

The credential file path `$env:LOCALAPPDATA\AtomicRedTeam\$env:USERNAME.txt` resolves to `C:\Users\<username>\AppData\Local\AtomicRedTeam\<username>.txt`. This is the ART pre-requisite staging location for test credentials.

All execution runs under `NT AUTHORITY\SYSTEM` (LogonId 0x3E7).

## What This Dataset Does Not Contain

No Security 4723 (attempt to change account password) or 4724 (attempt to reset account password) events appear. These events would be generated by the domain controller if the `Set-ADAccountPassword` call successfully reached the DC and was processed. Their absence suggests either the command failed before reaching the DC (e.g., credential validation failure, network issue in the lab), or that the test ran in a context where DC-side logging was not captured. You would need DC security event logs to see the password change attempt at the directory level.

No 4625 (logon failure) or 4648 (explicit credential logon) events appear in the Security channel, which would be present if the credential file authentication step failed with bad credentials.

No network connection events (Sysmon EID 3) capture the LDAP/Kerberos traffic to the domain controller that `Set-ADAccountPassword` would generate.

The actual credential stored in `$env:LOCALAPPDATA\AtomicRedTeam\$env:USERNAME.txt` is not captured in any event — the file is read, but its contents are not logged by Windows event subsystems.

## Assessment

With Defender disabled, the script executed fully: the `Write-Host "DONE"` EID 4103 event confirms the script body ran to completion. The complete command line — including the credential file path, the `Set-ADAccountPassword` call, and the target password `Uplow-1` — is preserved in both Security EID 4688 and Sysmon EID 1.

Compared to the defended variant (46 Sysmon, 12 Security, 38 PowerShell), the undefended dataset is smaller in Sysmon (39 vs. 46) and Security (4 vs. 12) but substantially larger in PowerShell (104 vs. 38). The Security and Sysmon reduction reflects the absence of Defender inspection processes. The PowerShell count increase reflects fuller script block logging without AMSI interference.

The temporary file at `C:\Windows\Temp\01dcb40bc6223c9f` (Sysmon EID 11) is a forensic artifact unique to this test's execution — likely the AD module's internal working file — not present in the defended variant, providing an additional indicator of whether the AD password change operation was attempted.

## Detection Opportunities Present in This Data

**PowerShell command line containing `Set-ADAccountPassword`**: Security EID 4688 and Sysmon EID 1 both capture the full command line. `Set-ADAccountPassword` in a PowerShell process command line — especially invoked under SYSTEM context from a parent PowerShell test framework — is a high-confidence indicator of programmatic password manipulation.

**Credential file access pattern**: The script reads from `$env:LOCALAPPDATA\AtomicRedTeam\$env:USERNAME.txt`. More generally, PowerShell scripts that read credential files from `AppData\Local` directories and immediately use them with AD cmdlets represent a stealthy credential-chaining technique. File access monitoring on credential staging paths would surface this.

**PowerShell EID 4104 script block with `ConvertTo-SecureString -AsPlainText -Force`**: The pattern of converting a plaintext password string to a SecureString using `-AsPlainText -Force` in a script block (as opposed to interactive user input) captures hardcoded or file-sourced credentials being converted for programmatic use. This specific pattern frequently appears in credential theft and account manipulation scripts.

**Temporary file creation under C:\Windows\Temp by PowerShell during AD operations**: The Sysmon EID 11 file creation of `C:\Windows\Temp\01dcb40bc6223c9f` — a hex-named temp file under System context — coincides with the AD module operation. Correlating temp file creation with AD cmdlet execution in PowerShell provides a behavioral anchor for the operation's attempt.
