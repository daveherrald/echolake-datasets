# T1003.004-1: LSA Secrets — Dumping LSA Secrets

## Technique Context

T1003.004 (LSA Secrets) is a credential access technique where attackers extract the Local Security Authority secrets database from the Windows registry to harvest sensitive authentication material. The LSA secrets hive lives at `HKLM\SECURITY\Policy\Secrets` and stores service account passwords, cached domain logon credentials, machine account passwords, and auto-logon credentials. Because accessing this key requires SYSTEM privileges or SeBackupPrivilege, attackers typically elevate first, then use tools like `reg save` or PsExec's `-s` flag (which runs commands as SYSTEM) to extract the hive to a local file for offline parsing.

This test uses PsExec to run `reg save HKLM\security\policy\secrets` as SYSTEM, which is a direct and noisy method — the resulting dump file contains encrypted secrets that can be decrypted with the corresponding SYSTEM hive. Detection historically focuses on registry reads of the Security hive, process access to LSASS, creation of registry dump files in writable directories, and privilege token manipulation associated with the backup/restore capabilities needed for the operation.

The undefended variant of this test is significant because with Defender active, the PsExec binary is typically flagged immediately. Without Defender, the process chain can run to completion and the dump file will be written to disk.

## What This Dataset Contains

This dataset captures the full execution of the LSA secrets dump attempt via PsExec. The Security channel's EID 4688 events show the complete process creation chain. PowerShell (PID acting as ART test framework) spawns cmd.exe with the command:

```
"cmd.exe" /c "C:\AtomicRedTeam\atomics\..\ExternalPayloads\T1003.004\bin\PsExec.exe" -accepteula -s reg save HKLM\security\policy\secrets %temp%\secrets /y
```

A subsequent cmd.exe is spawned with `"cmd.exe" /c del %temp%\secrets >nul 2> nul`, confirming the cleanup step that follows successful file creation. The presence of this cleanup command is a meaningful indicator — it implies the dump file was written, otherwise there would be nothing to delete. In the defended version, the cmd.exe process exited with status 0x1 (failure) and no cleanup command appeared.

The Security channel's EID 5379 events record seven Credential Manager read operations against the machine account (`ACME-WS06$`) at logon ID `0x3e7` (SYSTEM). These enumerate Windows Live and Microsoft Account credential targets, reflecting the LSA secrets enumeration activity.

Sysmon EID 29 (File Executable Detected) captures the Defender signature updater (`MpSigStub.exe`) writing delta VDM files (`mpasdlta.vdm`, `mpavdlta.vdm`) — these are routine Defender updates occurring in parallel with the test, not attack artifacts.

Compared to the defended version (26 sysmon events, 10 security events, 34 PowerShell events), this undefended dataset is larger: 54 sysmon events, 158 security events, 105 PowerShell events. The security channel jump from 10 to 158 events is primarily EID 4664 (hard link creation attempts) generated by concurrent Windows Update activity, and the additional 4688 events include the cleanup command that was absent in the defended run.

## What This Dataset Does Not Contain

The actual PsExec binary execution and its spawning of SYSTEM-context processes are not directly captured in the sysmon EID 1 samples — the samples are dominated by OS-level file creation events. PsExec's SYSTEM-context `reg.exe` process creation, which would be the clearest indicator of success, would appear in the full 4688 event stream but is not in the five sampled 4688 events. There are no Sysmon EID 12/13 events showing direct registry access or modification of the `HKLM\SECURITY` hive itself.

There are no network events (Sysmon EID 3) even though PsExec communicates over named pipes. No LSASS process access events (EID 10 targeting lsass.exe) appear, which is expected since this technique dumps via `reg save` rather than direct LSASS memory access. The PowerShell channel shows only framework boilerplate (Set-StrictMode, ErrorCategory_Message) with no technique-specific script content, because the test executes through cmd.exe rather than native PowerShell cmdlets.

## Assessment

This dataset provides strong process execution telemetry for the LSA secrets dump approach using PsExec and `reg save`. The most valuable evidence is in the Security 4688 events: the explicit `PsExec.exe -accepteula -s reg save HKLM\security\policy\secrets` command line, followed by the cleanup delete command that implies successful file creation. This combination is high-fidelity for detection engineering. The dataset is suitable for building and testing detections around PsExec-mediated SYSTEM escalation for credential dumping, specifically the `reg save` approach to LSA secrets extraction.

## Detection Opportunities Present in This Data

1. Security EID 4688 with a command line containing both `PsExec.exe` and `reg save` and `HKLM\security` in the same invocation is a very specific indicator with minimal expected false positive rate.

2. Security EID 4688 showing `cmd.exe` spawning a `del` command targeting `%temp%\secrets` (or any temporary path with the word "secrets") immediately after a PsExec execution in the same PowerShell session is a behavioral sequence indicator.

3. The combination of Security EID 5379 (Credential Manager enumeration) occurring in the same narrow time window as a `reg save` targeting the Security hive in EID 4688 strengthens confidence in a credential access scenario.

4. Sysmon EID 1 showing `PsExec.exe` with the `-s` flag (run as SYSTEM) and a `reg save` subcommand targeting any path under `HKLM\SECURITY` is directly actionable.

5. File creation events (Sysmon EID 11) in `%TEMP%` with names like `secrets`, `lsa`, or other credential-suggestive names in close temporal proximity to PsExec execution would confirm the dump file was written.

6. The presence of both `SeBackupPrivilege` and `SeRestorePrivilege` being enabled on a PowerShell process (Security EID 4703) in the same session as a `reg save HKLM\SECURITY` command is a corroborating indicator worth correlating.
