# T1218.009-2: Regsvcs/Regasm — Regsvcs Uninstall Method Call Test

## Technique Context

T1218.009-2 uses `RegSvcs.exe` (the .NET Component Services Registrar) rather than `RegAsm.exe` as the execution proxy. While functionally similar to T1218.009-1, `regsvcs.exe` requires the target assembly to have a strong name (a cryptographic key signature), making this variant slightly more complex to implement but potentially more evasive. The test generates a strong name key, compiles the malicious assembly with it, and then invokes `regsvcs.exe` to register and execute the assembly's COM+ component code.

A distinctive characteristic of this test is the PowerShell command line: it includes a long Base64-encoded string representing the RSA strong name key material. This is an operational detail of how the ATH framework generates the key for signing the compiled assembly, but from a detection standpoint, a PowerShell command line containing hundreds of characters of Base64-encoded data is itself a high-confidence behavioral indicator.

The Security EID 5888 and 5890 events (COM+ object creation and access audit events) are unique to this test — they appear when `regsvcs.exe` interacts with the COM+ catalog infrastructure, providing a detection opportunity not present in the Regasm variants.

The dataset was collected on ACME-WS06 (Windows 11 Enterprise, domain-joined to acme.local) with Windows Defender disabled.

## What This Dataset Contains

The dataset contains 172 total events: 99 PowerShell, 17 Security, 55 Sysmon, and 1 Application. This is the largest event set in the T1218.009 group.

**Security EID 4688 captures the full attack chain:**

1. `"powershell.exe" & {$key = 'BwIAAAAkAABSU0EyAAQAAAEAAQBhXtvkSeH85E31z64cAX+X2PWGc6DHP...` (truncated, 900+ characters) — PowerShell with embedded Base64 RSA key material for strong-name signing
2. `"C:\Windows\Microsoft.NET\Framework\v4.0.30319\csc.exe" /r:System.EnterpriseServices.dll /out:C:\Windows\TEMP\T1218.009.dll /target:library /keyfile:C:\Windows\TEMP\key.snk C:\AtomicRedTeam\atomics\T1218.009\src\T1218.009.cs` — compilation with strong-name key
3. `C:\Windows\Microsoft.NET\Framework\v4.0.30319\cvtres.exe /NOLOGO /READONLY /MACHINE:IX86 ...` — resource converter
4. `"C:\Windows\Microsoft.NET\Framework\v4.0.30319\regsvcs.exe" C:\Windows\TEMP\T1218.009.dll` — regsvcs executing the assembly
5. `"C:\Windows\system32\whoami.exe"` — ATH success verification (two executions)

**Security EID 5888 (COM+ object creation)** records 9 events documenting COM+ catalog interactions as `regsvcs.exe` registers and instantiates the COM+ component. These events capture the object GUID, the application name, and the component class name.

**Security EID 5890 (COM+ object access)** records 1 event for COM+ catalog access.

**Sysmon EID 1** captures process creation with parent-child relationships:
- `powershell.exe` → child `powershell.exe` (the Base64-key command, `RuleName: technique_id=T1083`)
- `child powershell.exe` → `csc.exe` (`/r:System.EnterpriseServices.dll /out:...T1218.009.dll /target:library /keyfile:...key.snk`, `RuleName: technique_id=T1127`)
- Regsvcs process visible in Security 4688 but not in Sysmon EID 1 (include-mode filter gap)
- Two `whoami.exe` executions

**Sysmon EID 11 (File Created)** records 12 events — the highest in the T1218.009 group. These document:
- `C:\Windows\Temp\key.snk` — the generated strong-name key file
- `C:\Windows\Temp\T1218.009.dll` — the compiled malicious DLL
- `C:\Windows\Temp\CSCC282599254A540388D70FAD8C6C8C237.TMP` — csc temporary file
- A type library file generated by regsvcs during assembly registration
- PowerShell profile files

**Sysmon EID 7 (Image Load)** produces 28 events across the two PowerShell processes.

**Sysmon EID 10 (Process Access)** records 6 events, including full-access from PowerShell to the child PowerShell process and to `whoami.exe`.

**Sysmon EID 22 (DNS Query)** records 1 event — ambient system activity.

**Sysmon EID 17 (Pipe Created)** records 2 events.

**Application EID 4879** records 1 event — a .NET component services registration event from regsvcs.

**PowerShell EID 4104** captures 98 events including the cleanup scriptblock `Invoke-AtomicTest T1218.009 -TestNumbers 2 -Cleanup -Confirm:$false`.

## What This Dataset Does Not Contain

`regsvcs.exe` process creation does not appear in Sysmon EID 1 — same include-mode gap as `regasm.exe` in T1218.009-1. The regsvcs execution is documented only in Security EID 4688.

The `ComUnregisterFunction` (or equivalent COM+ method) execution inside `regsvcs.exe` is not directly captured; only the `whoami.exe` executions confirm it ran.

No network events appear, consistent with local compilation and execution.

## Assessment

This is the richest T1218.009 dataset. The combination of Security EID 4688 (full command lines with the Base64 key material), Security EID 5888/5890 (COM+ catalog interactions), and Sysmon EID 11 (12 file creation events documenting key, DLL, and type library) creates a comprehensive evidentiary record. The `whoami.exe` executions confirm successful unmanaged code execution through `regsvcs.exe`.

Compared to the defended variant (53 Sysmon, 17 Security, 37 PowerShell, 1 Application), this undefended run produced very similar event volumes (55 Sysmon, 17 Security, 99 PowerShell, 1 Application). The Security channel count is identical — the COM+ audit events (5888/5890) appear in both environments regardless of Defender state, as they are driven by the COM+ infrastructure audit policy.

## Detection Opportunities Present in This Data

**Security EID 4688 (PowerShell with Base64 key):** A PowerShell command line containing hundreds of characters of Base64-encoded RSA key material (`BwIAAAAkAABSU0EyAAQAAAEAAQ...`) is a high-fidelity indicator of this specific technique workflow. Any process creation with massive Base64 blobs in PowerShell command lines warrants investigation.

**Security EID 4688 (csc.exe with /keyfile):** `csc.exe` compiling with `/r:System.EnterpriseServices.dll /target:library /keyfile:...key.snk` is the characteristic compile step. The combination of EnterpriseServices reference and keyfile signing from a temp directory is unusual for legitimate development.

**Security EID 5888/5890 (COM+ Audit):** COM+ object creation events generated by `regsvcs.exe` registering a freshly-compiled assembly from `C:\Windows\TEMP\` are anomalous. Legitimate COM+ registrations come from known installation directories, not temp paths.

**Sysmon EID 1 (csc.exe, RuleName=T1127):** The Trusted Developer Utilities Proxy Execution rule fires on csc.exe being run from a PowerShell-spawned child process chain.

**Sysmon EID 11 (File Created):** `C:\Windows\Temp\key.snk` creation is a distinctive artifact — strong name key generation to a temp directory is not a legitimate software development pattern in production environments.
