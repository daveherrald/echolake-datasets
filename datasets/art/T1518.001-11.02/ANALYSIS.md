# T1518.001-11: Security Software Discovery — Windows Defender Exclusion Settings via WMIC

## Technique Context

T1518.001 (Security Software Discovery) extends beyond simply knowing that a security product is running. Discovering *how* it is configured — specifically, what paths, extensions, and processes are excluded from scanning — is directly actionable intelligence. Files dropped into an excluded path will not be scanned by Defender, making exclusions a reliable staging area for payloads. An adversary who queries and finds, for example, `ExclusionPath: C:\Temp` can drop their next-stage implant there with confidence that real-time protection will not inspect it.

This test queries Defender's WMI provider for exclusion configuration using the legacy WMIC command-line client and filters for specific exclusion-related keys:

```
cmd.exe /c wmic /Node:localhost /Namespace:\\root\Microsoft\Windows\Defender Path MSFT_MpPreference Get /format:list | findstr /i /C:"DisableRealtimeMonitoring" /C:"ExclusionPath" /C:"ExclusionExtension" /C:"ExclusionProcess"
```

The use of `MSFT_MpPreference` — the Defender WMI class that holds all policy and configuration preferences — is more specific than the `SecurityCenter2` namespace used in T1518.001-7 and provides deeper operational detail.

## What This Dataset Contains

The dataset spans approximately 8 seconds (2026-03-17 17:05:37–17:05:45 UTC) on ACME-WS06 running as NT AUTHORITY\SYSTEM. It contains 166 events across three channels: 107 PowerShell, 31 Security, and 28 Sysmon.

**Security (31 events, EIDs 4688, 4798, 4799):** This is the most informative channel. Seven EID 4688 process creation events document the execution chain:

1. `wmiprvse.exe -Embedding` — WMI Provider Host, spawned by `svchost.exe -k DcomLaunch -p`, launched to handle the WMIC request
2. `"C:\Windows\system32\whoami.exe"` — test framework pre-flight
3. `"cmd.exe" /c wmic /Node:localhost /Namespace:\\root\Microsoft\Windows\Defender Path MSFT_MpPreference Get /format:list | findstr /i /C:"DisableRealtimeMonitoring" /C:"ExclusionPath" /C:"ExclusionExtension" /C:"ExclusionProcess"` — the core technique command
4. `wmic /Node:localhost /Namespace:\\root\Microsoft\Windows\Defender Path MSFT_MpPreference Get /format:list` — WMIC as a separate process creation
5. `findstr /i /C:"DisableRealtimeMonitoring" /C:"ExclusionPath" /C:"ExclusionExtension" /C:"ExclusionProcess"` — filter applied to WMIC output
6. `"C:\Windows\system32\whoami.exe"` — post-execution check
7. `"cmd.exe" /c` — cleanup

The complete `MSFT_MpPreference` query with all four exclusion-related filter strings is preserved verbatim in events 3, 4, and 5.

Beyond process creations, the Security channel contains 5 EID 4798 events (user's local group membership enumerated) and 19 EID 4799 events (security-enabled local group membership enumerated). Examining these events reveals they were generated by two distinct processes. The EID 4798 events (enumerating `Administrator`, `Guest`, `DefaultAccount`, `WDAGUtilityAccount`, `mm11711`) were generated by `WmiPrvSE.exe` (PID 0x1064) — the WMI provider processing the Defender query. The EID 4799 events (enumerating 13+ local groups including `Administrators`, `Backup Operators`, `Event Log Readers`, etc.) were generated by `cribl.exe` (PID 0x15f4) — the Cribl Edge collection agent performing its own local group enumeration as part of its telemetry collection operations. The Cribl Edge 4799 events are collector behavior, not technique behavior.

**Sysmon (28 events, EIDs 1, 7, 10, 11, 17):** Sysmon EID 1 captures four process creations: `wmiprvse.exe -Embedding` (tagged `T1047,Windows Management Instrumentation`), `whoami.exe` (tagged `T1033`), `cmd.exe` with the full WMIC command (tagged `T1059.003`), and a second `whoami.exe`. Notably, `wmic.exe` itself and `findstr.exe` do not appear as Sysmon EID 1 entries in the 28 surfaced events — the sysmon-modular include-mode config did not match the WMIC command here (consistent with the defended variant finding). Security EID 4688 covers this gap via full process creation auditing.

Sysmon EID 7 records 16 DLL load events. EID 10 fires four times (ProcessAccess, `T1055.001`). EID 17 records one named pipe create. EID 11 records one file creation.

**PowerShell (107 events, EIDs 4103, 4104):** All ART test framework boilerplate. EID 4103 records 3 module logging events; EID 4104 records 104 internal formatter stubs.

## What This Dataset Does Not Contain

- **No exclusion data.** Whether ACME-WS06 had any Defender exclusions configured — and what they were — is not captured in event logs. The query ran and produced output, but that output exists only in the process's stdout buffer.
- **No EID 4104 script block for the WMIC command.** The command was invoked via `cmd.exe` not an inline PowerShell block, so no 4104 entry carries the `MSFT_MpPreference` query text. Security EID 4688 and Sysmon EID 1 are the authoritative sources.
- **Cribl Edge 4799 events as noise context:** The 19 EID 4799 events from `cribl.exe` (the collection agent) enumerating local groups represent real system activity from the instrumentation layer. They are present in this dataset and represent the kind of background telemetry that real infrastructure always generates.

## Assessment

This dataset is distinguished by three features. First, it captures a particularly sensitive query: `MSFT_MpPreference` exclusion settings are precisely the data an adversary needs to stage subsequent payloads safely. Second, the `wmiprvse.exe` spawn is captured both in Sysmon EID 1 (tagged `T1047`) and Security EID 4688, documenting the WMI provider instantiation that the WMIC query triggers — this is a useful process chain anchor. Third, the EID 4798 and 4799 volume from two different sources (`wmiprvse.exe` and `cribl.exe`) illustrates the kind of multi-source activity that appears in real datasets: technique-generated events (EID 4798 from WMI) interleaved with infrastructure events (EID 4799 from the collection agent).

In the defended variant (33 Sysmon, 16 Security, 34 PowerShell), the Security channel was smaller, suggesting fewer background events fell within the collection window. The undefended dataset's 31 Security events include the Cribl Edge group enumeration background activity that was apparently not present in the defended run's window.

## Detection Opportunities Present in This Data

- **Security EID 4688 command line:** The full WMIC command including `MSFT_MpPreference`, `ExclusionPath`, `ExclusionExtension`, and `ExclusionProcess` filter strings is present verbatim in three separate 4688 events (cmd.exe, wmic.exe, and findstr.exe). Any of these strings in a process command line argument is a high-specificity indicator for Defender exclusion enumeration.
- **Sysmon EID 1 for `wmiprvse.exe` (tagged T1047):** The WMI provider host spawning in direct temporal proximity to the WMIC exclusion query is captured and tagged. Correlating a `wmiprvse.exe -Embedding` creation with a preceding `wmic.exe` command that references `MSFT_MpPreference` is a strong detection chain.
- **Security EID 4798 (local group membership enumerated):** Five EID 4798 events generated by `WmiPrvSE.exe` reflect the WMI provider querying local account information as part of handling the Defender query. This is an indirect indicator — `wmiprvse.exe` generating account enumeration events is a signal that the WMI provider is doing more than expected.
- **Security EID 4799 from `cribl.exe`:** The 19 EID 4799 events from the Cribl Edge agent are baseline infrastructure activity. In a production environment, identifying the collection agent's process as the source of local group enumeration events is important for filtering true positives from known-good processes.
