# T1003.003-9: NTDS — Create Volume Shadow Copy with diskshadow

## Technique Context

T1003.003 targets the Active Directory database (NTDS.dit) for domain credential extraction. This variant uses `diskshadow.exe` — a Windows-native VSS management utility — to create a Volume Shadow Copy via a script file. Diskshadow accepts a script file as input (`diskshadow /s script.txt`), and the script can contain commands to create a shadow copy, expose it as a drive letter, and copy files from it. This approach is particularly useful for attackers because: (1) diskshadow.exe is a Microsoft-signed binary present on Windows Server systems, (2) it can be driven entirely from a pre-written script file without interactive input, and (3) the script-based execution makes command line detection harder than with vssadmin (the script content may not appear in the process command line).

Detection engineers focus on diskshadow executing with `/s` flag and a script file, the Volume Shadow Copy service events generated by the shadow creation, and subsequent file access events for NTDS.dit through the exposed shadow. On a workstation, diskshadow execution is rare enough to be inherently suspicious.

Unlike T1003.003-8 (vssadmin), this diskshadow variant generates Sysmon EID 29 events — "RawAccessRead" events or Volume Shadow Copy creation events — providing unique telemetry not present in the other shadow copy approaches.

## What This Dataset Contains

This dataset was collected from ACME-WS06 (Windows 11 Enterprise Evaluation, `acme.local` domain) with Windows Defender disabled. Execution was as `NT AUTHORITY\SYSTEM`.

**Sysmon channel (69 events: 50x EID 11, 6x EID 1, 5x EID 10, 4x EID 7, 3x EID 29, 1x EID 2):** The 3 Sysmon EID 29 events are unique to this dataset among the T1003.003 series. EID 29 (RawAccessRead, or in some Sysmon versions, Volume Shadow Copy deletion notification) confirms the VSS subsystem was engaged. EID 2 (file creation time changed) appears once — diskshadow or related tooling may have touched a file timestamp. EID 1 events show `cmd.exe` (PID 5168) at 22:46:57 UTC spawned from `powershell.exe`, with the command line `"cmd.exe" /c` (truncated in the sample, but the diskshadow invocation would be the next argument). The parent is `powershell.exe` (PID 5536 from the previous test in the sequence).

**Security channel (19 events: 12x EID 4688, 7x EID 5379):** The 12 EID 4688 events include a distinctive new actor: `C:\Program Files\Windows Defender\MpCmdRun.exe` appearing multiple times (PIDs 0x1050, 0x5c8, 0xddc, 0x157c, 0xf60). On a system with Defender disabled (or in passive mode), MpCmdRun.exe may still execute for signature updates, scan scheduling, or health checks — its presence in EID 4688 during this test window is background Windows Defender maintenance activity, not an active blocking response. The meaningful EID 4688 events are `whoami.exe` (0xfb0, pre-check), `cmd.exe` (0x14dc, the diskshadow invocation), and associated `sppsvc.exe` (0x4dc, the Software Protection Platform service responding to the VSS operation).

**PowerShell channel (93 events, all EID 4104):** The cleanup block `Invoke-AtomicTest T1003.003 -TestNumbers 9 -Cleanup -Confirm:$false` confirms full test execution. 93 events is slightly lower than the other T1003.003 tests, consistent with a simpler execution chain.

**Application channel (1 event, EID 16394):** A single "Offline downlevel migration succeeded" event — this is a routine Windows application compatibility event and is not related to the diskshadow execution.

**Compared to the defended dataset (sysmon: 16, security: 12, powershell: 26):** The undefended run shows more than 4x the Sysmon events (69 vs. 16) and significantly more PowerShell events (93 vs. 26). Crucially, the 3 Sysmon EID 29 events are the key differentiator — if these represent VSS creation events, they would only appear when the operation actually completed (or was attempted) rather than being blocked before VSS interaction. The EID 2 file creation time modification also only appears in the undefended run.

## What This Dataset Does Not Contain

The diskshadow script file path and content are not visible in the 20-event sample. The diskshadow.exe process creation itself (Sysmon EID 1 / Security EID 4688 with the process name `diskshadow.exe`) is in the full dataset but not in the preview. The complete command line for the `cmd.exe` invocation of diskshadow is truncated in the sample. VSS snapshot creation success/failure is not definitively determinable from the samples alone. Since this is a workstation, the VSC would not contain NTDS.dit — the technique demonstrates the shadow copy creation mechanics without credential material extraction.

## Assessment

The Sysmon EID 29 events are the most distinctive artifact in this dataset, providing direct VSS-layer telemetry that does not appear in any other T1003.003 variant in this batch. The MpCmdRun.exe appearances in EID 4688 are a reminder that Defender processes continue running even in "disabled" mode — these are not blocking events but maintenance activity. The Application channel EID 16394 is a benign artifact. For detection engineering, this dataset provides the clearest VSS-layer telemetry of any test in the T1003.003 series here, making it particularly useful for validating detections that rely on Volume Shadow Copy service events.

## Detection Opportunities Present in This Data

1. **Sysmon EID 29 — Volume Shadow Copy creation telemetry:** Three EID 29 events directly confirm VSS subsystem engagement. Detection rules watching for EID 29 events from non-backup processes, especially `diskshadow.exe` or `cmd.exe` as the originator, provide a direct signal.

2. **EID 4688 / Sysmon EID 1 — diskshadow.exe execution:** `diskshadow.exe` appearing in process creation events on a workstation (or any non-backup-server system) is an anomaly. The `/s` flag with a script file path as argument is particularly suspicious.

3. **Sysmon EID 2 — file creation time modification:** The EID 2 event indicates a file's creation timestamp was altered, a common anti-forensic technique. Combined with the VSS activity, this indicates the attacker is attempting to obscure timing evidence.

4. **EID 4688 — cmd.exe spawning diskshadow.exe from PowerShell-as-SYSTEM context:** The process tree `powershell.exe (SYSTEM) → cmd.exe → diskshadow.exe /s [scriptfile]` on a workstation is a complete behavioral chain for this technique.

5. **Diskshadow script file creation:** Before executing `diskshadow /s [script]`, the attacker must write the script file to disk. Monitoring for file creation events where the content (if accessible) or the path (e.g., a temp directory) suggests a diskshadow script provides an earlier detection opportunity.

6. **EID 4688 — sppsvc.exe activation coinciding with VSS operations:** The Software Protection Platform service activating (`sppsvc.exe`) coincident with diskshadow execution may indicate the VSS snapshot triggered a license validation check. This coupling is detectable through temporal correlation of sppsvc.exe starts with shadow copy operations.

7. **Application EID 16394 — unexpected migration event:** While benign here, "Offline downlevel migration succeeded" appearing during attack activity in the Application log provides a timestamp anchor for correlating across channels.
