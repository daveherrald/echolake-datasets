# T1562.010-3: Downgrade Attack — PowerShell Version 2 Downgrade

## Technique Context

MITRE ATT&CK T1562.010 (Downgrade Attack) covers techniques that force a system back to
an older, less-secure version of a component to shed modern security controls. PowerShell
version 2 is the canonical example: it predates Script Block Logging (EID 4104), AMSI
integration, Constrained Language Mode, and other protective features introduced in
PowerShell 3.0 and later. Invoking `powershell.exe -version 2` forces the engine to use
the .NET 2.0/3.5 runtime if it is present on the system, creating a session that has none
of the telemetry infrastructure defenders rely on. Threat actors including APT29 have used
this to run Mimikatz and Empire payloads while producing no script block log records.

In the defended variant of this dataset, Windows Defender intercepted `Invoke-Mimikatz`
during process initialization, producing a `0xC0000022` (Access Denied) exit code for the
PowerShell v2 child process and a Sysmon EID 8 (CreateRemoteThread) consistent with
Defender's behavior monitoring. The technique was attempted but not completed.

## What This Dataset Contains

With Defender disabled, the PowerShell v2 downgrade executes and the test framework completes
its full lifecycle. The dataset contains 112 PowerShell channel events and 10 Security
channel events spanning approximately 2 seconds.

**PowerShell channel (112 events) — EIDs 4104, 4103, 4100:**

The 106 EID 4104 script block logging events are exclusively ART test framework boilerplate —
internal PowerShell error-handling fragments (`{ Set-StrictMode -Version 1; $_.PSMessageDetails }`,
`{ Set-StrictMode -Version 1; $this.Exception.InnerException.PSMessageDetails }`,
`{ Set-StrictMode -Version 1; $_.ErrorCategory_Message }`, `{ Set-StrictMode -Version 1;
$_.OriginInfo }`) and a `Set-ExecutionPolicy Bypass -Scope Process -Force` block. These
are generated by the outer PowerShell 5.1 test framework that wraps the ART test.

The critical point is what is **absent**: there are no EID 4104 script block records for
any `Invoke-Mimikatz` invocation. This is the expected behavior of the downgrade attack —
the PowerShell v2 engine does not generate script block log events, so the actual payload
executed within the v2 session is invisible to this logging channel.

The 4 EID 4103 (module logging) events include `Set-ExecutionPolicy Bypass` and a terminal
`Write-Host "DONE"` showing the test completed. The 2 EID 4100 events record PowerShell
engine start/stop.

**Security channel (10 events) — EIDs 4688, 4689:**

Security EID 4688 records show the process creation chain. A `powershell.exe` child process
is launched by the outer test framework PowerShell:

```
Process Command Line: "powershell.exe" & {Pow...
Creator Process Name: C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe
```

This child process exits with status `0xC0000022` in the defended dataset; in the undefended
dataset it exits with `0x0` (success), confirming that the v2 engine launched and ran.
`whoami.exe` appears twice (pre- and post-execution ART identity checks), both spawned by
the outer `powershell.exe` test framework, both exiting `0x0`.

A second PowerShell child process with `Process Command Line: "powershell.exe" & {}` exits
cleanly. The `cmd.exe` process exits with status `0x1` (error), indicating the outer command
interpreter wrapper encountered an issue after the v2 session completed.

## What This Dataset Does Not Contain

**No Invoke-Mimikatz EID 4104 record.** This is the defining characteristic of the downgrade
attack's success. The PowerShell v2 engine executed the `Invoke-Mimikatz` command inside a
session that has no script block logging. No scriptblock ID corresponding to the credential
harvesting payload appears anywhere in the PowerShell channel.

**No Sysmon events.** The Sysmon channel is not present in this dataset. The undefended
collection does not include Sysmon process creation or DLL load events for the v2 PowerShell
process or any Mimikatz-related activity (LSASS access, credential dumping, etc.).

**No Sysmon EID 8 (CreateRemoteThread).** The defended dataset contained a CreateRemoteThread
event consistent with Defender intercepting process initialization. Without Defender, no such
event fires, which means the injection-detection signal that alerted defenders in the defended
variant is absent here.

**No evidence of Invoke-Mimikatz output or credential data.** The technique, if `Invoke-Mimikatz`
ran within the v2 session, would produce LSASS credential material. None of that appears in
any collected channel because the v2 session generates no log output and Sysmon is not
providing process injection telemetry.

## Assessment

This dataset illustrates one of the most important logging gaps in Windows security
architecture: the PowerShell v2 downgrade creates a complete blind spot in script block
logging. You can confirm the test framework ran to completion (the `Write-Host "DONE"` EID 4103
and `0x0` exit codes are present), but the payload itself — `Invoke-Mimikatz` and everything
that ran under it — produces no log records in the collected channels.

The gap between the defended and undefended datasets is stark. In the defended version,
Defender blocked execution and produced visible artifact (the `0xC0000022` exit code and
EID 8). In this undefended version, execution completes silently with only test framework
boilerplate visible.

The practical implication: in environments where the `MicrosoftWindowsPowerShellV2Root`
Windows feature has not been removed, PowerShell v2 is available and this attack succeeds
in the same way it does here. The `0x0` exit of the child `powershell.exe` process launched
from the test framework is the best available indicator in this dataset that the v2 engine ran to
completion — but it provides no information about what payload it executed.

## Detection Opportunities Present in This Data

**Security EID 4688 — parent-child PowerShell chain:** The outer test framework launches a child
`powershell.exe` process. In environments auditing command-line arguments (as this one is),
the command line fragment `"powershell.exe" & {Pow...` combined with a parent that is also
`powershell.exe` is a detectable pattern. The truncation in the 4688 command line field is
a dataset artifact; in a live environment the full `-version 2 -command 'Invoke-Mimikatz'`
would be visible.

**Absence of EID 4104 following a `powershell.exe -version 2` invocation:** If your pipeline
detects the `-version 2` flag in a 4688 command line but finds no corresponding EID 4104
session events for that process, that absence is itself a signal. The attacked host produced
112 EID 4104 events — all from the outer v5.1 test framework — and none from any v2 session.

**EID 4100 (PowerShell engine start):** The two EID 4100 events record PowerShell engine
lifecycle for the outer session. A v2 engine session would also generate EID 4100, but with
`EngineVersion: 2.0` rather than `5.1.22621.6345`. Look for `EngineVersion: 2.0` in EID
4100 events as a strong indicator of downgrade activity.

**Process exit code correlation:** The `0x0` exit of the child `powershell.exe` process,
combined with no EID 4104 from that PID, and no Defender block (no EID 1116/1117), is a
detectable pattern that distinguishes a successful downgrade from a failed one.
