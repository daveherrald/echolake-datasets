# T1059.001-14: PowerShell — ATHPowerShellCommandLineParameter -Command parameter variations with encoded arguments

## Technique Context

T1059.001 (PowerShell) represents one of the most versatile and widely-abused execution techniques in the MITRE ATT&CK framework. PowerShell's native Windows integration, extensive API access, and ability to execute both interactive commands and complex scripts make it a preferred tool for attackers across all phases of an operation. This specific test focuses on command-line parameter variations with encoded arguments — a common obfuscation technique where PowerShell commands are Base64-encoded to evade detection systems that rely on plaintext command analysis.

The detection community prioritizes monitoring PowerShell execution patterns, particularly focusing on command-line arguments, script block content, and process relationships. Encoded commands (typically using the `-EncodedCommand` or `-EC` parameters) are especially significant because they represent deliberate obfuscation attempts. Security teams commonly build detections around Base64-encoded PowerShell patterns, unusual command-line switches, and parent-child process relationships involving PowerShell.

## What This Dataset Contains

The dataset captures a test execution of the ATH (Atomic Threat Hunter) PowerShell framework testing various command-line parameter combinations with encoded arguments. The primary evidence appears in Security event 4688: `"powershell.exe" & {Out-ATHPowerShellCommandLineParameter -CommandLineSwitchType Hyphen -CommandParamVariation C -UseEncodedArguments -EncodedArgumentsParamVariation EA -Execute -ErrorAction Stop}`.

Sysmon captures two PowerShell process creations (EID 1) - the parent PowerShell process (PID 32404) and a child PowerShell process (PID 31928) with the command line showing the ATH framework execution. Process access events (EID 10) show PowerShell accessing both a `whoami.exe` process and another PowerShell process, indicating command execution and potential process interaction.

The PowerShell operational channel contains extensive script block logging (EID 4104) but most entries are test framework boilerplate showing `Set-StrictMode` calls and error handling templates. Two significant script blocks appear: one showing the ATH command execution and another referencing the PowerShell profile path at `C:\Windows\system32\config\systemprofile\Documents\WindowsPowerShell\Microsoft.PowerShell_profile.ps1`.

Multiple image load events (EID 7) capture PowerShell loading .NET runtime components including `mscoree.dll`, `mscoreei.dll`, `clr.dll`, and the PowerShell automation assembly `System.Management.Automation.ni.dll`. Windows Defender components (`MpOAV.dll`, `MpClient.dll`) are also loaded, indicating real-time protection interaction.

## What This Dataset Does Not Contain

The dataset lacks the actual Base64-encoded command content that would be generated by the ATH framework. While the command line shows the framework being executed with encoded argument parameters, the decoded command content is not visible in the captured events. This represents a gap in understanding what specific PowerShell commands were actually encoded and executed.

The PowerShell script block logging predominantly captures framework boilerplate rather than the substantive PowerShell commands being tested. The actual encoded commands that would demonstrate various parameter variations are not present in the logged script blocks.

No network activity events appear in the Sysmon data, suggesting the test focused purely on local command-line parameter variations rather than network-based PowerShell operations. The test also doesn't show process injection or other advanced PowerShell attack techniques beyond basic command execution.

## Assessment

This dataset provides moderate value for detection engineering focused on PowerShell command-line analysis and process relationships. The Security 4688 events with full command-line logging effectively capture the parent-child PowerShell execution pattern that would be typical in many real-world scenarios involving encoded PowerShell commands.

However, the dataset's utility is limited by the absence of actual Base64-encoded command content, which is the primary artifact that detection rules would target. The ATH framework execution shown here provides good telemetry for detecting testing tools, but doesn't demonstrate the full encoded command obfuscation techniques that the test purports to exercise.

The Sysmon process creation and access events provide valuable context for understanding PowerShell execution patterns, particularly the parent-child process relationships and system resource access that could indicate suspicious activity.

## Detection Opportunities Present in This Data

1. **PowerShell Parent-Child Process Chains** - Security 4688 events showing PowerShell spawning additional PowerShell processes with suspicious command-line parameters
2. **ATH Framework Detection** - Command lines containing "Out-ATHPowerShellCommandLineParameter" or similar testing framework identifiers
3. **Encoded Command Parameter Usage** - Command lines referencing encoded argument variations or Base64 encoding parameters
4. **PowerShell Process Access Patterns** - Sysmon EID 10 events showing PowerShell processes accessing other processes with full access rights (0x1FFFFF)
5. **System Profile PowerShell Activity** - PowerShell processes running as NT AUTHORITY\SYSTEM accessing system profile paths
6. **Suspicious PowerShell Module Loading** - Multiple .NET runtime and PowerShell automation DLL loads in rapid succession
7. **PowerShell Named Pipe Creation** - Sysmon EID 17 events showing PSHost named pipe creation patterns
8. **Command-Line Obfuscation Attempts** - Security events showing PowerShell execution with parameters designed to test evasion techniques
