# T1562.001-11: Disable or Modify Tools — Unload Sysmon Filter Driver

## Technique Context

T1562.001 (Disable or Modify Tools) includes adversary actions that disable or tamper with security monitoring software to blind defenders. This test unloads the SysmonDrv kernel-mode filter driver using `fltmc.exe`, the Windows Filter Manager Control Program. Successfully unloading SysmonDrv terminates Sysmon's ability to monitor process creation, network connections, file operations, registry changes, and driver loads — immediately eliminating a primary source of endpoint telemetry.

`fltmc.exe unload` is a Windows-native operation. The Filter Manager is a kernel component that manages minifilter drivers, and `fltmc.exe` provides a documented command-line interface to load, unload, list, and query filter drivers. Any administrator can run it; unloading a security driver requires no special hacking tools, only SYSTEM or SeLoadDriverPrivilege access.

This technique is documented in multiple intrusion cases and is operationally significant because it is simple, fast, and immediately effective. Once SysmonDrv is unloaded, Sysmon generates no further events until the service is restarted — giving an attacker a clean window for subsequent activity.

## What This Dataset Contains

The dataset spans 4 seconds (2026-03-17 17:34:08–17:34:12 UTC) and contains 96 PowerShell events, 4 Security events, 16 Sysmon events, and 1 System event.

The attack command is captured in full across Security EID 4688 and Sysmon EID 1:
```
"cmd.exe" /c fltmc.exe unload SysmonDrv
```
Decomposed into a second process creation event:
```
fltmc.exe  unload SysmonDrv
```

Security EID 4688 records 4 process creation events: `whoami.exe` (pre-check), `cmd.exe` (the fltmc wrapper), `fltmc.exe` (the actual unload command), and a second `whoami.exe` (post-check). All run as `NT AUTHORITY\SYSTEM`.

Sysmon EID 1 captures 3 process creation events with full hashes and parent-child relationships:
- `whoami.exe` (SHA256: `574BC2A2995FE2B1F732CCD39F2D99460ACE980AF29EFDF1EB0D3E888BE7D6F0`) with parent `powershell.exe`
- `cmd.exe` (SHA256: `A6E3B3B22B7FE8CE2C9245816126723EAA13F43B9F591883E59959A2D409426A`) tagged `RuleName: technique_id=T1059.003,technique_name=Windows Command Shell`, with parent `powershell.exe`
- `fltMC.exe` (SHA256: `729BEF61EC88D4C195D3BA78EB138B3D6E00DD43BA3E4B000FF655A95A4AD6D1`) with parent `cmd.exe` and the full `fltmc.exe  unload SysmonDrv` command line

Sysmon EID 7 records 9 ImageLoad events for the PowerShell DLL chain. Sysmon EID 10 records 2 ProcessAccess events. Sysmon EID 17 records 1 named pipe creation.

**Sysmon EID 255** is the terminal Sysmon event: `ID: DriverCommunication`, `Description: Failed to retrieve events - Last error: The I/O operation has been aborted because of either a thread exit or an application request.` This is Sysmon's self-reported error when its kernel driver communication channel is severed. This event is generated by the Sysmon service layer as it loses contact with SysmonDrv.

**System EID 1** records the Windows Filter Manager's own log of the operation: `File System Filter 'SysmonDrv' (Version 0.0, 2024-07-22T08:29:55.000000000Z) unloaded successfully.` This independent confirmation from the Windows kernel component is present regardless of whether Sysmon itself is running — it is written by the Filter Manager, not by Sysmon.

The PowerShell events are 95 EID 4104 script block logging and 1 EID 4103 module logging event. The 4104 events include the ART preamble (`Set-ExecutionPolicy Bypass`, `$ErrorActionPreference = 'Continue'`) and the cleanup invocation `Invoke-AtomicTest T1562.001 -TestNumbers 11 -Cleanup -Confirm:$false`.

## What This Dataset Does Not Contain

No Sysmon events generated after the driver unload. Once SysmonDrv is unloaded, the Sysmon service loses its kernel-level visibility. Any activity following the unload — including the cleanup operations and the second `whoami.exe` execution — would not be captured in Sysmon. Security EID 4688 continues to function independently and does capture the second `whoami.exe`, but the Sysmon record ends with EID 255.

No Sysmon EID 1 for fltmc.exe in the post-unload window. The Sysmon EID 1 for `fltMC.exe` itself is present because Sysmon captured the process creation before the unload completed. However, any processes spawned after SysmonDrv was fully unloaded are absent from the Sysmon channel.

Compared to the defended variant (27 Sysmon, 13 Security, 34 PowerShell, 1 System), this undefended run is structurally identical. The defended variant had slightly more Security events (13 vs 4 here) and more PowerShell events (34 vs 96 here). The higher PowerShell event count in the undefended run reflects more script block logging captured without Defender's presence. The Sysmon structure is nearly the same — both variants produce EID 255 and the System EID 1 filter unload event.

## Assessment

This dataset contains the most operationally significant event type in the entire collection: a successful kernel-level security tool disablement captured through its own dying signal. The combination of Sysmon EID 255 (self-reported driver communication failure) and System EID 1 (Windows Filter Manager confirmation) creates a correlated record of the exact moment Sysmon went blind.

The technique succeeds identically in both defended and undefended environments. Defender does not protect Sysmon's driver — `fltmc.exe unload SysmonDrv` completes in both cases. This means your detection strategy for this technique cannot rely on Defender intervention; it must focus on logging sources that survive the unload (Security event log, System event log) and on pre-unload Sysmon data.

This dataset demonstrates a fundamental principle of defense-in-depth: a single logging source can be disabled, but overlapping sources provide resilience. Security 4688 captures the `fltmc.exe` command line independently of Sysmon, and System EID 1 provides kernel-level confirmation independently of both.

## Detection Opportunities Present in This Data

**Security EID 4688 for `fltmc.exe`**: Command line `fltmc.exe  unload SysmonDrv` with parent `cmd.exe` running as SYSTEM. This is the single highest-confidence indicator: `fltmc.exe unload` targeting a security driver by name, invoked from a SYSTEM PowerShell-spawned cmd.exe. Few legitimate operations use `fltmc.exe unload` in production environments, and targeting `SysmonDrv` specifically has no legitimate use case.

**Sysmon EID 1 for `fltMC.exe`** tagged `RuleName: technique_id=T1518.001,technique_name=Security Software Discovery`: The sysmon-modular ruleset annotates `fltmc.exe` invocations as potential security software discovery. While this annotation is technically for discovery, the rule correctly flags `fltmc.exe` as worthy of investigation.

**Sysmon EID 255**: The self-reported driver communication error is an unambiguous indicator that SysmonDrv was forcibly unloaded. If you are ingesting Sysmon events in real time, EID 255 followed by a gap in Sysmon event generation from that host should trigger an immediate alert.

**System EID 1 for SysmonDrv**: The Windows Filter Manager event `File System Filter 'SysmonDrv' ... unloaded successfully` is written regardless of what security software is present and persists in the System event log. This event should be monitored on all endpoints where Sysmon is deployed.

**Temporal gap in Sysmon events**: After the driver unload, Sysmon events cease. If your SIEM expects regular Sysmon event flow from an endpoint and that flow stops abruptly — particularly following a `fltmc.exe` process creation — the absence of events is itself a detection signal.
