# T1574.001-1: DLL Search Order Hijacking — DLL Search Order Hijacking - amsi.dll

## Technique Context

T1574.001 (Hijack Execution Flow: DLL Search Order Hijacking) abuses the order in which Windows resolves DLL names when loading an executable. Windows searches several directories in sequence — starting with the application's own directory, then `System32`, then other system and PATH locations. An adversary who places a malicious DLL with the correct name in a directory that appears earlier in that search order will have their code loaded by the legitimate host process, executing in its security context.

This test targets `amsi.dll` — the Antimalware Scan Interface library loaded by PowerShell and other scripting hosts. The technique copies `powershell.exe` to `%APPDATA%\updater.exe` and copies `amsi.dll` to the same directory. When `updater.exe` runs, Windows should find `amsi.dll` in the application directory before reaching `System32`, loading the attacker-controlled copy into a process that appears to be a trusted binary.

## What This Dataset Contains

The dataset captures 126 events across two log sources — PowerShell (110 events: 107 EID 4104, 3 EID 4103) and Security (16 events: 10 EID 4689, 5 EID 4688, 1 EID 4703) — collected on ACME-WS06 (Windows 11 Enterprise, domain-joined, Defender disabled).

**The full attack execution chain is present in the data.** Security EID 4688 (process creation) records the complete command sequence issued from PowerShell:

```
"cmd.exe" /c copy %windir%\System32\windowspowershell\v1.0\powershell.exe %APPDATA%\updater.exe
          & copy %windir%\System32\amsi.dll %APPDATA%\amsi.dll
          & %APPDATA%\updater.exe -Command exit
```

A subsequent EID 4688 confirms that `updater.exe` actually launched from its AppData path:

```
New Process Name: C:\Windows\System32\config\systemprofile\AppData\Roaming\updater.exe
Process Command Line: C:\Windows\system32\config\systemprofile\AppData\Roaming\updater.exe -Command exit
Creator Process Name: C:\Windows\System32\cmd.exe
```

The cleanup step is also recorded — a `cmd.exe` process (EID 4688) was created with:

```
"cmd.exe" /c del %APPDATA%\updater.exe >nul 2>&1 & del %APPDATA%\amsi.dll >nul 2>&1
```

All five process creation events exit with status `0x0`, confirming the test framework ran to completion.

Security EID 4703 shows PowerShell (`powershell.exe`, PID 0x4364) being granted an extensive set of elevated privileges including `SeAssignPrimaryTokenPrivilege`, `SeLoadDriverPrivilege`, `SeRestorePrivilege`, and `SeDebugPrivilege`, consistent with the test framework running as SYSTEM.

PowerShell EID 4104 (script block logging) captures the ART test framework preamble: `Set-ExecutionPolicy Bypass -Scope Process -Force` and `$ErrorActionPreference = 'Continue'`. EID 4103 (module logging) records `Write-Host "DONE"` — confirming the test completed successfully.

## What This Dataset Does Not Contain

**No Sysmon events are present.** The dataset lacks Sysmon EID 1 (Process Create with full hashes and parent chain), EID 7 (Image Loaded — which would directly confirm whether `amsi.dll` was loaded from AppData or from System32), and EID 11 (File Created — which would show the DLL drop to disk). Without EID 7, you cannot determine from this dataset alone whether the DLL hijack succeeded and the attacker-controlled `amsi.dll` was loaded, or whether Windows loaded the legitimate copy from System32.

**The dataset does not include network events.** There are no Sysmon EID 3 connections or DNS queries (EID 22), which is expected — this technique does not involve network activity.

**No file write events are captured.** The `copy` commands that place `updater.exe` and `amsi.dll` into AppData are only visible through the process command line in EID 4688, not as dedicated file creation records.

## Assessment

Compared to the defended variant (which had 42 Sysmon, 12 Security, and 36 PowerShell events), this undefended dataset contains fewer total events (0 Sysmon, 16 Security, 110 PowerShell) but provides a cleaner view of the actual execution chain. The Security channel here shows `updater.exe` launching from AppData — an artifact that was present in the defended run but blocked from completing. The PowerShell log is dominated by boilerplate script block fragments generated by the ART framework itself.

The key addition in the undefended run is that `updater.exe` actually executes from `C:\Windows\System32\config\systemprofile\AppData\Roaming\updater.exe` — confirmed by EID 4688. In the defended variant, Defender's intervention prevented or cut short this execution. Here, the binary runs, exits cleanly (`0x0`), and cleanup proceeds. Whether the DLL hijack itself succeeded (i.e., whether the AppData copy of `amsi.dll` was loaded rather than the System32 copy) cannot be confirmed from this dataset without Sysmon EID 7.

## Detection Opportunities Present in This Data

**EID 4688 — PowerShell spawning cmd.exe to copy a system binary to AppData.** The command line `copy %windir%\System32\windowspowershell\v1.0\powershell.exe %APPDATA%\updater.exe` is a strong indicator. Legitimate software does not copy PowerShell into a user's AppData directory under a different name.

**EID 4688 — Execution of a binary from a SYSTEM profile's AppData path.** The process `C:\Windows\System32\config\systemprofile\AppData\Roaming\updater.exe` launching from cmd.exe is highly anomalous. Executables running from SYSTEM's roaming AppData path are uncommon in normal operation.

**EID 4688 — Cleanup via cmd.exe deleting files from AppData.** The command `del %APPDATA%\updater.exe >nul 2>&1 & del %APPDATA%\amsi.dll >nul 2>&1` following the suspicious execution is a recognizable post-execution cleanup pattern.

**EID 4688 — Process name / image path mismatch.** `updater.exe` running from a path that contains `windowspowershell` in its ancestry chain, combined with a name designed to appear benign, is detectable by correlating the process image hash against known-good hashes for `powershell.exe`.

**EID 4703 — PowerShell process token enrichment.** The addition of high-value privileges to a PowerShell process running as SYSTEM is contextually suspicious when paired with the other events in this window.
