# T1572-4: Protocol Tunneling — Run ngrok

## Technique Context

T1572 (Protocol Tunneling) covers encapsulation of C2 or exfiltration traffic within
other protocols. ngrok is a legitimate reverse proxy and tunneling service that creates
an encrypted tunnel from a local port to a public ngrok subdomain, commonly used by
developers for testing webhooks. Adversaries use ngrok to expose internal services
(RDP, C2 listeners, SMB) to the internet through HTTPS tunnels, bypassing firewall
egress controls. The ngrok agent authenticates to the ngrok cloud infrastructure and
creates a publicly addressable URL (`*.ngrok.io` or `*.ngrok-free.app`) mapped to the
specified local port. This test configures an authtoken (`N/A` as placeholder), starts
ngrok tunneling TCP port 3389 (RDP), waits 5 seconds, then stops it.

## What This Dataset Contains

**Sysmon EID 1** — two PowerShell process creates:

> First: `CommandLine: "powershell.exe" & {C:\Users\Public\ngrok\ngrok.exe config add-authtoken N/A | Out-Null`
> `Start-Job -ScriptBlock { C:\Users\Public\ngrok\ngrok.exe tcp 3389 } | Out-Null`
> `Start-Sleep -s 5`
> `Stop-Job -Name Job1 | Out-Null}`

> Second: a background `powershell.exe -Version 5.1 -s -NoLogo -NoProfile` child
> process used by `Start-Job` to run the ngrok tunnel job asynchronously.

The ngrok binary at `C:\Users\Public\ngrok\ngrok.exe` — a user-writable path — is
visible in the command line. The `tcp 3389` argument confirms RDP port forwarding.
Notably, no Sysmon EID 1 for `ngrok.exe` itself appears, suggesting Defender blocked
it before execution.

**Sysmon EID 7** — DLL image loads for multiple PowerShell processes, tagged with
`T1055` (Process Injection) and `T1059.001` (PowerShell) rule names. One DLL load
includes `MpOAV.dll` (Defender AMSI provider), confirming AMSI was engaged.

**Sysmon EID 10** — process access events (T1055.001) from one PowerShell process
accessing another — ART test framework instrumentation.

**Sysmon EID 11** — file creates at `SystemProfile\PowerShell\StartupProfileData-*`
for two PowerShell instances (interactive and non-interactive profiles for the job host).

**Sysmon EID 17** — named pipe creation for multiple PowerShell host processes,
including the background job worker.

**Security EID 4688/4689** — process lifecycle for `powershell.exe` and `whoami.exe`.
No 4688 for `ngrok.exe`.

**Security EID 4703** — token right adjustment.

**PowerShell EID 4104** — the test payload is captured clearly, including both the
outer ART wrapper and the inner script block with the ngrok command line.

**PowerShell EID 8193/8194/8195/8197/12039** — runspace lifecycle events:
> `Creating Runspace object | Creating RunspacePool object (MinRunspaces 1, MaxRunspaces 1)`
> `Opening RunspacePool | Runspace state changed to Opening/Opened/Closing/Closed`

These are generated by `Start-Job`, which creates a new PowerShell runspace for the
background job. Their presence is a secondary indicator of background job usage in the
attack chain.

## What This Dataset Does Not Contain (and Why)

**No ngrok.exe process create.** Windows Defender 4.18.26010.5 detected and blocked
the ngrok binary before it could execute. The `config add-authtoken` and `tcp 3389`
commands were not executed. No Sysmon EID 1, Security EID 4688, or network connection
events for ngrok appear.

**No outbound TCP connection to ngrok infrastructure.** With ngrok blocked, no
connection to ngrok's relay servers (`*.ngrok.io`, `*.ngrok-free.app`, or the ngrok
tunnel relay IPs) was established.

**No Sysmon EID 3.** The network connection from ngrok to its cloud relay is the
key forensic artifact in a successful ngrok deployment; its absence here is a direct
consequence of the Defender block.

**No DNS query for ngrok domains.** No EID 22 events for `ngrok.io` or similar
ngrok infrastructure domains are present.

## Assessment

This dataset represents a Defender-blocked execution pattern for ngrok RDP tunneling.
The most valuable telemetry is in the PowerShell EID 4104 script block, which clearly
shows the ngrok binary path, the `tcp 3389` tunnel type, and the use of `Start-Job`
for background execution. The absence of ngrok process creation and network activity
indicates Defender successfully blocked the technique at the point of execution.
The runspace lifecycle events (8193-8197) are an underutilized secondary signal for
detecting background job-based execution.

## Detection Opportunities Present in This Data

- **PowerShell EID 4104** — `ngrok.exe tcp 3389` as a `Start-Job` block is a direct
  indicator; `ngrok.exe tcp` with any port, or `ngrok.exe http`, from a script block
  under SYSTEM is highly anomalous.
- **Binary path** — `C:\Users\Public\ngrok\ngrok.exe` is a user-writable path; any
  invocation of `ngrok.exe` from non-standard locations (`AppData`, `Public`, `Temp`)
  should trigger an alert.
- **`Start-Job` usage** — `Start-Job` in PowerShell EID 4104 from SYSTEM context to
  run a third-party binary asynchronously, combined with runspace lifecycle events
  (8194/8195/8197), is an indicator of background execution evasion.
- **Security EID 4688** — in environments where Defender does not block ngrok, an ngrok
  process create with `tcp 3389` command arguments would appear here; detection rules
  should cover both the blocked (EID 4104 only) and unblocked (EID 4688 + network)
  scenarios.
- **Runspace lifecycle EID 8194/8197** — `RunspacePool` creation with `MinRunspaces 1`
  combined with concurrent `Start-Job` usage in EID 4104 is a pattern specific to
  background job execution, useful for detecting attempts to run offensive tools
  out-of-band from the main PowerShell session.
