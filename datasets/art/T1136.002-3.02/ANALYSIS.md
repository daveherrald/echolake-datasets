# T1136.002-3: Domain Account — Create a new Domain Account using PowerShell

## Technique Context

Domain account creation (T1136.002) via PowerShell's .NET `System.DirectoryServices.AccountManagement` namespace is a more operationally sophisticated approach than the traditional `net user /add /domain` command. By using the `UserPrincipal` and `PrincipalContext` classes directly, an attacker avoids spawning `net.exe` — a well-monitored binary — and instead creates the domain account entirely within the PowerShell process via LDAP operations to the domain controller. The technique requires the executing account to have domain write permissions. When Defender is enabled, it detects and blocks the script; without Defender, the .NET API calls proceed without endpoint interference.

## What This Dataset Contains

With Windows Defender disabled, this dataset captures PowerShell executing the full .NET domain account creation script on ACME-WS06.acme.local.

**Process execution chain in Security EID 4688:** Four process creation events are present:

1. PowerShell parent (PID 17512) spawns `whoami.exe` — pre-execution identity check
2. PowerShell parent spawns child PowerShell (PID 18356) with the full technique command: `"powershell.exe" & {$SamAccountName = 'T1136.002_Admin'; $AccountPassword = ConvertTo-SecureString 'T1136_pass123!' -AsPlainText -Force; Add-Type -AssemblyName System.DirectoryServices.AccountManagement; $Context = New-Object -TypeName System.DirectoryServices.AccountManagement.PrincipalContext -ArgumentList ([System.DirectoryServices.AccountManagement.ContextType]::Domain); $User = New-Object -TypeName System.DirectoryServices.AccountManagement.UserPrincipal -ArgumentList ($Context); $User.SamAccountName = $SamAccountName; ...}`
3. PowerShell parent spawns `whoami.exe` — post-execution identity check
4. PowerShell parent spawns child PowerShell for cleanup (`"powershell.exe" & {}`)

The account name `T1136.002_Admin` and plaintext password `T1136_pass123!` are fully visible in the EID 4688 command line, including `ConvertTo-SecureString 'T1136_pass123!' -AsPlainText -Force`.

**Sysmon EID 1:** Sysmon captures the same child PowerShell (PID 18356) process creation with all arguments visible, tagged with `RuleName: technique_id=T1083,technique_name=File and Directory Discovery` (a heuristic rule match on the PowerShell binary itself). The full command line is present including the account name and password.

**PowerShell test framework logging:** 112 events (106 EID 4104, 6 EID 4103) include the `Invoke-AtomicTest T1136.002 -TestNumbers 3 -Cleanup -Confirm:$false` cleanup block, confirming the test framework ran to completion.

**PowerShell EID 4103:** `New-Object -TypeName System.DirectoryServices.AccountManagement.UserPrincipal` — individual cmdlet invocations are logged, confirming the .NET API calls executed.

Compared to the defended dataset (37 Sysmon, 10 Security, 50 PowerShell), this undefended run has similar Security counts (4 vs. 10, the defended run had more 4688 events from Defender scanning) but confirming the technique command line is visible. The defended run described blocking at the PowerShell script level without the EID 4688 process chain being captured.

## What This Dataset Does Not Contain

**Domain account creation confirmation (EID 4720):** No Security EID 4720 event appears in the local workstation log. If the account `T1136.002_Admin` was created in Active Directory, the authoritative creation event is on ACME-DC01 (192.168.4.10), not on this workstation. The absence of local EID 4720 is expected — it is generated by the domain controller, not the initiating workstation.

**Network connections to domain controller:** The .NET `UserPrincipal.Save()` call would initiate an LDAP write operation to the domain controller (192.168.4.10:389). No Sysmon EID 3 (network connection) events appear in the samples; the Sysmon-modular configuration does not appear to capture LDAP connections from PowerShell.

**Active Directory object creation events:** Any EID 4742 (computer account changed), EID 4728 (member added to global group), or domain-side EID 4720 events are not present in this workstation-scoped dataset.

**Success or failure confirmation:** The dataset does not definitively confirm whether the LDAP write to the domain controller succeeded. The PowerShell process completed (test framework reached cleanup), but network-level or DC-side confirmation is absent.

## Assessment

This dataset captures one of the best-available detection signals for .NET-based domain account creation: the Security EID 4688 command line contains the account name, password, and the full `System.DirectoryServices.AccountManagement` API call chain in a single event. This is a high-fidelity indicator that requires no correlation — the technique's intent is fully exposed in the process creation log.

The PowerShell EID 4103 cmdlet-level logging for `New-Object -TypeName System.DirectoryServices.AccountManagement.UserPrincipal` provides a second independent detection opportunity based on the API being called rather than the command-line string.

The primary limitation of this dataset for operational detection work is that it only reflects the workstation-side view. Full coverage of T1136.002 requires domain controller telemetry correlated with workstation events. For defenders building cross-source detection logic, this dataset demonstrates what the "initiating side" looks like; the confirming side requires DC Security log collection.

## Detection Opportunities Present in This Data

- **Security EID 4688 / Sysmon EID 1:** Child PowerShell spawned with `$SamAccountName = 'T1136.002_Admin'`, `ConvertTo-SecureString 'T1136_pass123!' -AsPlainText -Force`, and `System.DirectoryServices.AccountManagement.PrincipalContext` all in a single command line — the account name and plaintext password are both exposed
- **PowerShell EID 4103:** `New-Object -TypeName System.DirectoryServices.AccountManagement.UserPrincipal` cmdlet invocation is unusual in workstation PowerShell and specifically tied to programmatic AD account manipulation
- **PowerShell EID 4104:** `Add-Type -AssemblyName System.DirectoryServices.AccountManagement` in script block logs — loading this assembly in a non-administrative script context is anomalous
- **Security EID 4688:** PowerShell spawning child PowerShell with a script block containing `ContextType::Domain` is a reliable indicator of programmatic domain operations
- **Pattern:** `-AsPlainText -Force` in a PowerShell command line alongside `ConvertTo-SecureString` is a consistent indicator of credential manipulation in automated scripts
