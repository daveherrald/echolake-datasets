# T1615-4: Group Policy Discovery — WinPwn GPORemoteAccessPolicy

## Technique Context

T1615 (Group Policy Discovery) includes targeted enumeration of specific GPO categories relevant to an adversary's objectives. WinPwn's `GPORemoteAccessPolicy` function focuses on GPOs that govern remote access configurations — Remote Desktop Protocol (RDP) settings, Windows Remote Management (WinRM) policies, and firewall rules distributed via GPO. This makes it particularly useful for adversaries planning lateral movement: by identifying which systems have RDP or WinRM enabled via policy, and which firewall rules are enforced domain-wide, an attacker can map viable pivot paths before attempting connections. The module loads via an IEX download cradle from raw.githubusercontent.com.

## What This Dataset Contains

This is the richest T1615 dataset in the series, spanning five distinct log channels: 125 PowerShell events, 23 Security events, 1 Sysmon event, 2 Application events, 1 System event, and 1 WMI event.

**Sysmon (1 event)**:
- **EID 22** (DNS query) — Resolution of `raw.githubusercontent.com`, confirming the download cradle reached the DNS layer.

**Security log (23 events)**:
- **EID 4688** (7 events) — Process creation events. Key events include: `whoami.exe` (identity checks), `powershell.exe` with the full WinPwn download cradle (`GPORemoteAccessPolicy -consoleoutput -noninteractive`), a second `powershell.exe` cleanup pass with empty body, `sppsvc.exe` (Windows Software Protection Platform service launched during the test window), `svchost.exe -k netsvcs -p -s BITS` (Background Intelligent Transfer Service), and `wbem\WmiApSrv.exe` (WMI service launched during the test window).
- **EID 4689** (7 events) — Process exit events for the above processes.
- **EID 4624** (2 events) — Service logon (Type 5) for NT AUTHORITY\SYSTEM, generated by the BITS and/or WMI service startups triggered during this test.
- **EID 4627** (2 events) — Group membership enumeration associated with the service logon sessions.
- **EID 4672** (2 events) — Special privileges assigned to the SYSTEM logon sessions, including `SeAssignPrimaryTokenPrivilege`, `SeTcbPrivilege`, `SeSecurityPrivilege`, `SeDebugPrivilege`, and others.
- **EID 4799** (2 events) — Local group membership enumeration, called by `svchost.exe`. These events indicate a system service queried local group membership (consistent with BITS or WMI initializing and checking service permissions).
- **EID 4703** (1 event) — Token right adjustment for `powershell.exe`.

**Application log (2 events)**:
- **EID 16394** — `Offline downlevel migration succeeded` — a Windows Update-related migration event unrelated to the technique.
- **EID 15** — `Updated Windows Defender status successfully to SECURITY_PRODUCT_STATE_ON` — Defender reporting its state, present in other T1615 tests as well.

**System log (1 event)**:
- **EID 7040** — The BITS service start type changed from `demand start` to `auto start`. This change was triggered by an OS component during the test window and is not directly related to the WinPwn execution.

**WMI log (1 event)**:
- **EID 5860** — A WMI temporary event subscription was registered: `SELECT * FROM Win32_ProcessStartTrace WHERE ProcessName = 'wsmprovhost.exe'` by NT AUTHORITY\SYSTEM from process 16716. This query monitors for `wsmprovhost.exe` (the WinRM provider host) process starts. This is an OS-level WMI subscription, likely from the WMI service startup (seen in `WmiApSrv.exe` 4688 event), but it is also the type of subscription an adversary would use to detect WinRM availability — an interesting coincidental alignment with the technique's focus on remote access policy discovery.

**PowerShell log (125 events)** — 120 EID 4104 script block events, 4 EID 4103 module pipeline events, and 1 EID 4100 error event. Samples include `Set-ExecutionPolicy Bypass` and PowerShell runtime stubs. As with the other T1615 WinPwn tests, no WinPwn module body script blocks appear in the sample set, consistent with download failure or blocked execution.

## What This Dataset Does Not Contain

The WinPwn `GPORemoteAccessPolicy` function output — remote access GPO configurations — is absent. No LDAP queries to the domain controller appear in any channel. The EID 4100 error event and the absence of WinPwn module script blocks both suggest the download failed or was blocked before the function executed.

No Sysmon EID 3 network connection event appears for the PowerShell outbound connection to GitHub. The DNS query (EID 22) confirms name resolution occurred but no TCP connection to raw.githubusercontent.com was logged.

Sysmon EID 1 is absent for the `powershell.exe` process with the WinPwn command. The defended variant captured 27 Sysmon events including EID 1 for this process; the reduced Sysmon coverage here is consistent with a timing gap in collection rather than a Sysmon configuration change.

## Assessment

This undefended dataset is notably more populated than the defended variant (27 Sysmon, 10 Security, 52 PowerShell). The additional Security events (23 vs. 10) reflect OS-level activity — BITS service startup, WMI service launch, service logon sessions — that occurred during the broader test window. These events are not attributable to the WinPwn execution itself but provide realistic context: in an actual incident investigation, you would need to distinguish technique-generated events from concurrent OS service activity.

The WMI EID 5860 subscription for `wsmprovhost.exe` process starts is a particularly interesting artifact. While it appears to be OS-generated in this case, the same event type would appear if an adversary registered a WMI event subscription to monitor WinRM activity — making this a useful template for detection logic targeting such subscriptions.

The full WinPwn command line in EID 4688, including the pinned commit hash and `GPORemoteAccessPolicy -consoleoutput -noninteractive`, is the primary actionable detection artifact.

## Detection Opportunities Present in This Data

- **EID 4688: Command line containing `WinPwn.ps1` and `GPORemoteAccessPolicy`** — The function name `GPORemoteAccessPolicy` in a `powershell.exe` command line is a high-fidelity indicator with no legitimate use outside WinPwn. Alert on this function name alone or in combination with the WinPwn repository URL.

- **EID 4688: `iex` + `downloadstring` + `raw.githubusercontent.com`** — The IEX download cradle pattern is consistent across T1615-3 and T1615-4. A single rule covering `iex(new-object net.webclient).downloadstring` in `powershell.exe` command lines catches both WinPwn tests and is broadly applicable to other download-cradle attacks.

- **WMI EID 5860: Subscription for `Win32_ProcessStartTrace` targeting `wsmprovhost.exe`** — A WMI event subscription monitoring for `wsmprovhost.exe` process starts from an unexpected process or user context is a detection opportunity. Alert on EID 5860 events where the `NotificationQuery` contains `wsmprovhost.exe` and the registering process is not a known-good WMI service PID. This catches both coincidental OS activity (as seen here) and adversarial WMI-based WinRM monitoring.

- **System EID 7040: BITS service start type change during suspicious activity** — Changes to the BITS service start type (from `demand start` to `auto start`) during a window of other suspicious activity (IEX download cradles, WinPwn execution) can serve as a correlation signal. BITS is a common LOLBin for file downloads; its auto-start configuration change warrants investigation when co-occurring with offensive tooling.

- **Security EID 4799: Local group membership query by `svchost.exe`** — Unexpected local group membership queries (EID 4799) correlated with a window of IEX-based download activity may indicate that an attacker has pivoted to using service-layer mechanisms. While the specific events here are OS-benign, this correlation pattern is worth tracking.

- **Sysmon EID 22: DNS query for `raw.githubusercontent.com`** — As with T1615-3, DNS queries for GitHub's raw content CDN from `powershell.exe` are anomalous in most enterprise environments and should be alerted on.
