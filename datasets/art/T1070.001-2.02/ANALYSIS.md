# T1070.001-2: Clear Windows Event Logs — Delete System Logs Using Clear-EventLog

## Technique Context

T1070.001 (Clear Windows Event Logs) represents one of the most direct forms of anti-forensic activity available to an attacker with local administrator access. By erasing event log contents, an adversary removes the historical record of process executions, logon events, network connections, and other security-relevant activity captured since the last log rotation or backup.

The PowerShell `Clear-EventLog` cmdlet used in this test can clear any registered Windows event log channel by name. The ART test takes a comprehensive approach: it first enumerates all registered logs with `Get-EventLog -List`, then iterates through every log and clears each one with `Clear-EventLog -LogName $_`. This is not a targeted clearing of just the Security or System log — it wipes every channel that `Get-EventLog` can enumerate.

The attack is notable because it is self-documenting up to a point: the clearing actions generate EID 1102 (Security log cleared) and EID 104 (System/other log cleared) events, but only for the logs cleared before those channels are themselves cleared. The Security log records its own erasure because that event is generated during the clear operation; logs cleared afterward leave no such trace in the Security channel.

In the defended variant, Windows Defender was active. The technique completed without being blocked — `Clear-EventLog` is a legitimate PowerShell cmdlet and is not treated as malicious by Defender. The undefended dataset captures the same execution in an environment where no endpoint controls were active.

## What This Dataset Contains

The clearest technique evidence is Security EID 1102, which records the Security audit log being cleared. The event attributes show `SubjectUserName: SYSTEM`, `SubjectDomainName: NT AUTHORITY`, and `ClientProcessId: 6672` — the PID of the PowerShell process that executed the clearing operation. This event is generated by the Windows Security subsystem itself and cannot be suppressed during the clear operation.

The System channel contributes two EID 104 events recording that the System log and the "Windows PowerShell" log were cleared. Each event attributes the action to the same `SYSTEM` account with `ClientProcessId: 6672`. EID 104 is the System-channel equivalent of Security EID 1102 — it fires when any non-Security log is cleared via the log service API.

Sysmon EID 1 captures the PowerShell process launch with the full payload command line: `"powershell.exe" & {$logs = Get-EventLog -List | ForEach-Object {$_.Log} $logs | ForEach-Object {Clear-EventLog -LogName $_ } Get-EventLog -list}`. The parent is another `powershell.exe` process (`powershell` — the ART orchestration shell), giving a PowerShell-spawns-PowerShell process lineage. Sysmon tags this with `technique_id=T1059.001,technique_name=PowerShell`.

Security EID 4688 records both the `whoami.exe` pre-check and the payload `powershell.exe` launch, the latter with the same command line visible in the Sysmon EID 1 entry.

PowerShell script block logging (EID 4104) captures 120 events. The meaningful content is in the Sysmon EID 1 parent command line rather than as a standalone script block, because the payload was passed inline as a `& {...}` block. The script block log does record the ART module import (`Import-Module 'C:\AtomicRedTeam\invoke-atomicredteam\Invoke-AtomicRedTeam.psd1' -Force`).

Sysmon EID 11 records two file creation events for PowerShell startup profile cache files: `C:\Windows\System32\config\systemprofile\AppData\Local\Microsoft\Windows\PowerShell\StartupProfileData-NonInteractive` and `StartupProfileData-Interactive`. These are written by PowerShell when it initializes and are normal background activity, not technique artifacts.

The dataset contains 161 total events across four channels: 120 PowerShell, 3 Security, 36 Sysmon, and 2 System.

## What This Dataset Does Not Contain

The dataset does not contain EID 1102 or EID 104 events for every log that was cleared. The Security log's own clear generates a self-referential EID 1102, and a few System/PowerShell log clears generate EID 104s — but logs cleared after those initial channels produce no corresponding evidence in these channels. The absence of evidence for subsequent clears is itself the technique's intended effect.

There are no file system audit events for the log files themselves (typically under `C:\Windows\System32\winevt\Logs\`). Object access auditing was not enabled, so there are no Security EID 4663 events showing file-level writes to the `.evtx` files.

The dataset does not include events from log channels that were cleared before event collection could capture them. If you correlate this dataset against other log sources (e.g., Sysmon EID 1 timestamps versus the gap in Security log entries), you can infer what was erased, but the content of those erased entries is not present here.

No Defender behavioral telemetry, WMI subscription events, or network artifacts are present.

## Assessment

This is a high-fidelity capture of a destructive log clearing technique. The execution is unambiguous: EID 1102 and EID 104 confirm log channels were cleared, Sysmon EID 1 captures the exact PowerShell command that did it, and the process attribution to `NT AUTHORITY\SYSTEM` via `ClientProcessId` is precise.

Compared to the defended variant (36 Sysmon, 127 Security, 49 PowerShell, 2 System), the undefended dataset has significantly fewer Security events (3 vs. 127). This is consistent with the technique actually working: in the defended run, 127 Security events accumulated before and after the clear, while in the undefended run the log was nearly empty when cleared, leaving only 3 Security events total (the EID 1102 itself plus two EID 4688s for process creation). The technique's destructive effect is visible in the data.

## Detection Opportunities Present in This Data

**Security EID 1102 (Security log cleared):** This single event directly records the Security audit log being erased. It includes the responsible process ID (`ClientProcessId: 6672`), enabling attribution to the PowerShell process visible in Sysmon EID 1. This event fires even when the caller attempts to clear the Security log — Windows generates it as the clear completes.

**System EID 104 (event log cleared):** Fires for each non-Security log cleared. The two EID 104 events here show the System log and Windows PowerShell log were cleared by the same process, at the same time. A burst of EID 104 events across multiple channels within a short window is a strong indicator of programmatic bulk clearing rather than routine maintenance.

**Sysmon EID 1 with `Clear-EventLog` in command line:** The parent PowerShell command line contains `Clear-EventLog -LogName` as a literal string, providing a reliable detection anchor. The combination of `Get-EventLog -List` (enumeration) followed by iterative `Clear-EventLog` calls in a single command block characterizes this as bulk clearing rather than a targeted administrative operation.

**PowerShell spawning PowerShell with clearing payload:** The Sysmon EID 1 parent-child chain (`powershell.exe` → `powershell.exe`) with the clearing payload in the child's command line is a behavioral pattern worth tracking. Legitimate administrative log clearing typically uses purpose-built tools or `wevtutil.exe cl`, not inline PowerShell one-liners.
