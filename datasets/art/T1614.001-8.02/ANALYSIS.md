# T1614.001-8: System Language Discovery — Discover System Language by Windows API Query

## Technique Context

T1614.001 (System Language Discovery) covers adversary enumeration of system language and locale settings. This test variant uses Windows API calls directly — functions such as `GetSystemDefaultLCID()`, `GetUserDefaultLCID()`, `GetSystemDefaultLocaleName()`, or `GetUserDefaultUILanguage()` — rather than command-line tools like `reg.exe`, `chcp.com`, or `dism.exe`. The API approach is favored by malware that wants to minimize process-creation artifacts: the locale check can be done entirely within the calling process's memory, with no child process spawned and no shell command logged. The test invokes a compiled or script-based binary (`LanguageKeyboardLayout.exe`) from the ART ExternalPayloads directory via `cmd.exe`.

## What This Dataset Contains

The dataset spans approximately 5 seconds across two log sources: 107 PowerShell events and 13 Security events. Sysmon events are not present in this dataset.

**Security log (13 events)** provides the primary execution record:

- **EID 4688** (4 events) — Process creation with full command lines. The key event shows `powershell.exe` spawning `cmd.exe` with the command: `"cmd.exe" /c C:\AtomicRedTeam\atomics\..\ExternalPayloads\LanguageKeyboardLayout.exe`. A second `cmd.exe` spawned later with an empty body (`"cmd.exe" /c`) represents the cleanup pass. `whoami.exe` appears in two 4688 events (identity checks at start and end of the test).
- **EID 4689** (8 events) — Process exit events for each process in the chain, including `LanguageKeyboardLayout.exe`. Exit status is not visible in the samples, but exit events confirm execution proceeded.
- **EID 4703** (1 event) — Token right adjustment (`SeSecurityPrivilege` disabled) for the `powershell.exe` process.

The presence of `C:\AtomicRedTeam\atomics\..\ExternalPayloads\LanguageKeyboardLayout.exe` in the 4688 command line is the most operationally significant event in this dataset. The path traversal (`atomics\..`) is characteristic of how ART resolves ExternalPayloads but would equally appear if an adversary placed a similarly-named tool in that directory.

**PowerShell log (107 events)** — 104 EID 4104 script block events and 3 EID 4103 module pipeline events. The script blocks are PowerShell runtime infrastructure fragments and `Set-ExecutionPolicy Bypass -Scope Process -Force`. The `LanguageKeyboardLayout.exe` binary executes through `cmd.exe` outside the PowerShell process and generates no script block entries.

## What This Dataset Does Not Contain

The Windows API calls themselves (`GetSystemDefaultLCID`, `GetUserDefaultUILanguage`, etc.) are invisible in these logs. Capturing individual Win32 API invocations requires an EDR with userspace API hooking or a kernel ETW provider — neither is present in this collection setup. The return value of the locale query (the discovered language) is not recorded anywhere.

Sysmon events are absent from this dataset. Unlike `dism.exe`, `LanguageKeyboardLayout.exe` would not necessarily match sysmon-modular's ProcessCreate include rules (which target known LOLBins and suspicious process names). The Security log's EID 4688 with command-line auditing is the primary and more complete coverage source for this technique variant.

No registry reads, file writes, or network connections are generated by this technique. The API calls operate entirely in memory.

## Assessment

This is the most evasion-resistant variant of T1614.001 from a process-name perspective: there is no `reg.exe`, `dism.exe`, or `chcp.com` to alert on. The only reliable indicator is the process creation showing `cmd.exe` spawning a binary whose name and path may change across implementations.

Compared to the defended variant (26 Sysmon, 10 Security, 34 PowerShell), the undefended run produces comparable Security log coverage — 4 vs. 4 EID 4688 events — but no Sysmon events in either case, reflecting the consistent absence of a Sysmon rule matching `LanguageKeyboardLayout.exe`. The main difference between defended and undefended execution here is that Defender's real-time protection was active in the defended run; with defenses disabled, the binary executes without interference, which is reflected in process exit events confirming successful completion.

You will find the `LanguageKeyboardLayout.exe` path in the EID 4688 command-line field, which is the single most actionable event in this dataset.

## Detection Opportunities Present in This Data

- **EID 4688: `LanguageKeyboardLayout.exe` in the command line** — The specific binary name `LanguageKeyboardLayout.exe` appearing as a child of `cmd.exe` or `powershell.exe` is detectable. While attackers would use a differently-named binary, this specific ART payload name is a useful rule for detecting ART-based testing and tuning.

- **EID 4688: Unknown executable spawned by `cmd.exe` from `C:\AtomicRedTeam\` or `ExternalPayloads` paths** — The path `C:\AtomicRedTeam\atomics\..\ExternalPayloads\` in a 4688 `CommandLine` field is a reliable indicator of ART execution in a test environment. More generally, alerting on executables launched from unusual staging directories (non-standard paths, path traversal sequences) is a useful behavioral heuristic.

- **EID 4688: Short-lived process chain `cmd.exe` → unknown binary** — A `cmd.exe` spawned by `powershell.exe` that immediately exits after launching a single executable, with no additional child processes, is consistent with a wrapper invocation pattern used in language-discovery malware. Correlating 4688/4689 pairs by process ID to identify single-execution `cmd.exe` launchers narrows the search space.

- **EID 4703: `SeSecurityPrivilege` disabled** — Token adjustment events showing `SeSecurityPrivilege` being disabled on a `powershell.exe` process running as SYSTEM are a secondary indicator of the SYSTEM-level execution context used in this test. In a real environment, this privilege being disabled on a long-running PowerShell session warrants investigation.
