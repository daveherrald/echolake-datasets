# T1123-1: Audio Capture — Using Device Audio Capture Commandlet

## Technique Context

T1123 Audio Capture describes adversaries recording audio from a victim system's microphone to collect sensitive conversations, meeting audio, or ambient sound. This is a collection technique used in targeted espionage operations and some financially motivated campaigns where voice-based credentials or sensitive business discussions are the target.

This test uses the PowerShell `AudioDeviceCmdlets` module, which provides `Get-AudioDevice` and `Set-AudioDevice` cmdlets for managing Windows audio devices. The specific execution pattern is:

```powershell
$mic = Get-AudioDevice -Recording
Set-AudioDevice -ID $mic.ID
Start-Sleep -Seconds 5
```

This retrieves the default recording device (typically the microphone), sets it as the active audio device, and sleeps for 5 seconds — simulating a brief recording window. The actual audio capture in a real attack would be performed by subsequent code; this test exercises the device enumeration and selection step of the audio capture workflow.

The test does not use any built-in Windows utilities; it relies on a PowerShell module that must be available on the host. The presence of `AudioDeviceCmdlets` is itself an indicator of interest in audio device manipulation.

## What This Dataset Contains

The dataset captures 45 Sysmon events, 9 Security events, 125 PowerShell events, 1 Application event, 1 System event, and 1 WMI event recorded on ACME-WS06 with Windows Defender fully disabled.

The technique execution is recorded in Security EID 4688 with the full command line of the spawned PowerShell:

```
"powershell.exe" & {$mic = Get-AudioDevice -Recording
Set-AudioDevice -ID $mic.ID
Start-Sleep -Seconds 5}
```

The parent is `powershell.exe` (the outer ART test framework), making this a PowerShell-to-PowerShell chain. Sysmon EID 1 records the same process creation with the spawned `powershell.exe` SHA256 hash `3247BCFD...` and IMPHASH `AFACF6DC...`.

The Security channel (9 events) also includes two EID 4799 (Security-Enabled Local Group Membership Enumeration) events with subject `ACME-WS06$`. These are generated by `AudioDeviceCmdlets` or its dependencies when checking group membership, and are not test framework artifacts. The Security channel also contains EID 4624 (Logon) and EID 4672 (Special Logon), which recorded a system context logon event during the 5-second sleep window. The EID 4672 event records `SYSTEM` with an extensive privilege list including `SeDebugPrivilege`, `SeImpersonatePrivilege`, and `SeTcbPrivilege` — these reflect the privileges of the existing SYSTEM session, not a new privilege assignment by the technique.

Sysmon EID 11 records two file creations:
- `C:\Windows\Temp\01dcb62c36affe98` — created by `C:\ProgramData\Microsoft\Windows Defender\Platform\4.18.26010.5-0\MsMpEng.exe` (Defender engine creating a temp file, present even with real-time protection disabled as Defender's processes still run)
- `C:\Windows\System32\config\systemprofile\AppData\Local\Microsoft\Windows\PowerShell\StartupProfileData-NonInteractive` and `StartupProfileData-Interactive` — standard PowerShell initialization artifacts

The WMI channel contains a single EID 5860 event attributed to `NT AUTHORITY\SYSTEM`. This corresponds to a WMI subscription or temporary provider activation triggered during the test window, most likely by BITS or another background service activity.

The System channel EID 7040 likely records a service state change for a background Windows service that was incidentally active during the test.

The PowerShell channel (125 events) is dominated by EID 4104 test framework boilerplate. Notably, no script block captures the `Get-AudioDevice` or `Set-AudioDevice` calls specifically — these cmdlet invocations are executed as command-line arguments in the spawned PowerShell process rather than as a loaded script file, so they appear in the process creation log rather than as distinct script block events in this sample.

## What This Dataset Does Not Contain

This dataset does not contain any audio output data, audio API call events, or Media Foundation telemetry. The `Set-AudioDevice` cmdlet configures the device but this test does not invoke any recording APIs (e.g., `AudioGraph`, `waveInOpen`, `IAudioCaptureClient`). There are no Windows Audio service events in the captured channels.

No Security registry audit events are present (no EID 4657 or 4663), consistent with the host's audit policy.

There are no Sysmon EID 22 or EID 3 network events, confirming the technique was entirely local — `AudioDeviceCmdlets` is already installed on the host and no download was required.

Compared to the defended variant (46 Sysmon / 12 Security / 50 PowerShell), this dataset is larger in PowerShell (125 vs. 50) and similar in Sysmon (45 vs. 46). The Security channel here (9 events) is smaller than the defended version (12) because the defended execution triggered Defender-related process creation events during scanning. The additional EID 4799 group membership query events here are interesting — they were present with Defender disabled but less visible in the defended sample because Defender activity dominated that dataset's Security events.

## Assessment

This is a complete dataset for the PowerShell-based audio device enumeration pattern. The key indicators are all present: the `Get-AudioDevice -Recording` and `Set-AudioDevice -ID` command strings are recorded verbatim in the spawned process command line (EID 4688 / Sysmon EID 1). The presence of the `AudioDeviceCmdlets` module (implied by the cmdlet names) is itself a finding — this module is not installed by default on Windows 11 Enterprise.

The auxiliary events (EID 4799 group membership queries, EID 4624 logon, EID 4672 special logon) are authentic background activity that occurred during the test window. These are not technique artifacts but are representative of the environmental noise present in a real deployment. Detection analytics built against this dataset need to account for this context.

The dataset accurately reflects what undefended audio device discovery looks like at the event level — there is no audio content, only the device manipulation sequence.

## Detection Opportunities Present in This Data

**`Get-AudioDevice` and `Set-AudioDevice` in a PowerShell command line.** Security EID 4688 and Sysmon EID 1 record the verbatim command block. These cmdlet names from the `AudioDeviceCmdlets` module are unusual in legitimate administrative contexts. Their presence in a process creation event — particularly from a SYSTEM-context PowerShell spawning another PowerShell — warrants investigation.

**PowerShell spawning PowerShell with audio device cmdlets.** The process chain (outer test framework PowerShell → child PowerShell with `Get-AudioDevice`) is characteristic of scripted attack tool execution. Legitimate audio device management tools do not typically launch from a PowerShell test framework.

**Security EID 4799 (Security-Enabled Local Group Membership Enumeration) correlated with audio device activity.** The two EID 4799 events are generated by the audio cmdlet module performing background group checks. While EID 4799 alone is not a high-fidelity indicator, its correlation with `Get-AudioDevice` command-line activity strengthens the overall detection case.

**MsMpEng.exe writing temp files during the test window.** The Sysmon EID 11 recording `C:\Windows\Temp\01dcb62c36affe98` from `MsMpEng.exe` reflects Defender's passive monitoring processes — present even with real-time protection disabled. This artifact is not directly useful for detecting the technique but is useful context for understanding why temp files with hash-like names appear in Sysmon EID 11 data from this host.
