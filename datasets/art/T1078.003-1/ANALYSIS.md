# T1078.003-1: Local Accounts — Create local account with admin privileges

## Technique Context

T1078.003 focuses on adversary abuse of local accounts for persistence, privilege escalation, defense evasion, and initial access. Attackers commonly create new local accounts with administrative privileges to maintain access and blend in with legitimate user activity. This technique is particularly attractive because local accounts bypass many cloud-based authentication controls and monitoring systems that focus on domain authentication.

The detection community typically focuses on monitoring for suspicious account creation activities, especially when accounts are immediately added to privileged groups like Administrators. Key indicators include `net user` command usage with `/add` parameters, accounts created outside normal provisioning processes, and rapid elevation to administrative groups. Account creation events in Windows are primarily captured through Security event logs (4720 for user creation, 4732 for group membership changes) and process creation telemetry showing the commands used.

## What This Dataset Contains

This dataset captures a complete local account creation and privilege escalation sequence executed via PowerShell and cmd.exe. The core technique implementation is visible in Security event 4688 with the command line: `"cmd.exe" /c net user art-test /add & net user art-test -4RTisCool!-321 & net localgroup administrators art-test /add`.

The process execution chain shows:
1. PowerShell (PID 21564) spawning cmd.exe (PID 33940) 
2. cmd.exe executing three sequential `net.exe` processes for account creation, password setting, and group addition
3. Each `net.exe` process spawning corresponding `net1.exe` child processes (the actual implementation)

Sysmon captures detailed process creation events for the entire chain, including:
- EID 1 for cmd.exe with the full command line showing all three operations
- EID 1 for each net.exe process with individual command arguments (`net user art-test /add`, `net user art-test -4RTisCool!-321`, `net localgroup administrators art-test /add`)
- EID 1 for each corresponding net1.exe child process

Security event logs provide process creation (4688) and termination (4689) events for all processes, with detailed command-line logging showing the complete attack sequence. Notably, the net1.exe processes exit with status codes of 0x2 and 0x1, indicating the operations encountered errors—likely because the account already existed or permission issues occurred.

## What This Dataset Does Not Contain

This dataset lacks the most critical Windows events for detecting account management activities. There are no Security events 4720 (user account created), 4722 (user account enabled), or 4732 (member added to security-enabled local group), which would normally be generated by successful account creation and group membership changes.

The absence of these events, combined with the non-zero exit codes from net1.exe processes, suggests the account creation operations failed. This could be due to the account already existing, insufficient permissions despite running as SYSTEM, or other system-level restrictions.

The dataset also doesn't contain any SAM database access events or registry modifications that would accompany successful local account creation. No authentication events (4624/4625) are present to show successful logon with the newly created account.

## Assessment

This dataset provides excellent telemetry for detecting *attempted* local account creation and privilege escalation, but limited value for understanding successful account compromise. The process creation telemetry from both Sysmon EID 1 and Security EID 4688 events offers comprehensive command-line visibility that would enable detection of the technique attempt.

The failure of the actual account operations (evidenced by error exit codes) makes this dataset most valuable for testing detection logic around suspicious process execution patterns rather than successful account manipulation. The complete command-line capture showing the chained operations (`net user /add & net user password & net localgroup administrators /add`) represents a classic attack pattern that detection engineers should monitor for.

For building robust detections of T1078.003, security teams would ideally want datasets that include successful account creation events alongside the process telemetry shown here.

## Detection Opportunities Present in This Data

1. **Chained net.exe Command Detection** - Monitor for cmd.exe processes executing multiple net user and net localgroup commands in sequence, particularly with `&` command chaining operators visible in Security EID 4688 command lines.

2. **Account Creation with Immediate Privilege Escalation** - Detect net user commands followed immediately by net localgroup administrators commands targeting the same username, captured in Sysmon EID 1 process creation events.

3. **Suspicious PowerShell-to-CMD Process Chain** - Alert on PowerShell processes spawning cmd.exe with net user command arguments, visible in the parent-child process relationships in Sysmon ProcessCreate events.

4. **Net.exe Process Anomalies** - Monitor for unusual net.exe execution patterns, including rapid sequential execution and specific command-line arguments like `/add` combined with administrator group modifications.

5. **System Context Account Management** - Flag account creation activities performed by SYSTEM context processes outside of normal system provisioning, identifiable through the User field in both Sysmon and Security events.

6. **Failed Account Operations** - Track net1.exe processes exiting with error codes (0x1, 0x2) as indicators of attempted but unsuccessful account manipulation, visible in Security EID 4689 exit status fields.
