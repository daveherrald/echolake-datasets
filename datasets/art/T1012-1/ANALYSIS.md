# T1012-1: Query Registry — Query Registry

## Technique Context

T1012 Query Registry is a Discovery technique where adversaries enumerate registry keys and values to gather information about system configuration, installed software, persistence mechanisms, and potential security tools. This technique is fundamental to Windows post-exploitation as the registry contains extensive system and application configuration data. Attackers commonly query persistence locations (Run keys, Winlogon, services), security software configurations, system information, and network settings. The detection community focuses on monitoring registry queries targeting sensitive keys, particularly when performed by suspicious processes or in bulk enumeration patterns. While registry queries are extremely common in legitimate operations, the specific keys queried and the context (process, user, frequency) provide detection opportunities.

## What This Dataset Contains

This dataset captures a comprehensive registry enumeration sequence executed through PowerShell and cmd.exe. The Security channel shows the complete process execution chain: PowerShell (PID 6812) spawning cmd.exe (PID 6404) with a massive command line containing 23 chained `reg query` commands targeting persistence locations: `"cmd.exe" /c reg query "HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Windows" & reg query HKLM\Software\Microsoft\Windows\CurrentVersion\RunServicesOnce & reg query HKCU\Software\Microsoft\Windows\CurrentVersion\RunServicesOnce & reg query HKLM\Software\Microsoft\Windows\CurrentVersion\RunServices & reg query HKCU\Software\Microsoft\Windows\CurrentVersion\RunServices & reg query "HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon\Notify" & reg query "HKLM\Software\Microsoft\Windows NT\CurrentVersion\Winlogon\Userinit" & reg query "HKCU\Software\Microsoft\Windows NT\CurrentVersion\Winlogon\\Shell" & reg query "HKLM\Software\Microsoft\Windows NT\CurrentVersion\Winlogon\\Shell" & reg query HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\ShellServiceObjectDelayLoad & reg query HKLM\Software\Microsoft\Windows\CurrentVersion\RunOnce & reg query HKLM\Software\Microsoft\Windows\CurrentVersion\RunOnceEx & reg query HKLM\Software\Microsoft\Windows\CurrentVersion\Run & reg query HKCU\Software\Microsoft\Windows\CurrentVersion\Run & reg query HKCU\Software\Microsoft\Windows\CurrentVersion\RunOnce & reg query HKLM\Software\Microsoft\Windows\CurrentVersion\Policies\Explorer\Run & reg query HKCU\Software\Microsoft\Windows\CurrentVersion\Policies\Explorer\Run & reg query HKLM\system\currentcontrolset\services /s | findstr ImagePath 2>nul | findstr /Ri ".*\.sys$" & reg query HKLM\Software\Microsoft\Windows\CurrentVersion\Run & reg query HKLM\SYSTEM\CurrentControlSet\Control\SafeBoot & reg query "HKLM\SOFTWARE\Microsoft\Active Setup\Installed Components" & reg query "HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Group Policy\Scripts\Startup"`.

Sysmon captures 22 individual reg.exe process creations (EID 1), each with specific registry paths: Winlogon locations, Run keys (HKLM/HKCU), RunServices*, policies, services enumeration with filtering for .sys files, SafeBoot configuration, Active Setup components, and Group Policy scripts. The exit status codes in Security events show mixed results - many reg.exe processes exit with code 0x1 (indicating the queried key doesn't exist), while others exit with 0x0 (successful query).

PowerShell telemetry contains only test framework boilerplate - `Set-ExecutionPolicy Bypass` commands and various `Set-StrictMode` script blocks without actual technique content.

## What This Dataset Does Not Contain

This dataset lacks the actual registry query results - while we see all the process creations and command lines, there's no capture of what data was returned from successful queries. The dataset doesn't include any Registry ETW events that would show the actual registry keys being read. Some expected Sysmon ProcessCreate events may be missing due to the include-mode filtering configuration, though reg.exe appears to be captured consistently. The PowerShell channel doesn't contain the actual registry enumeration commands, only execution policy changes, suggesting the core technique was implemented via cmd.exe rather than PowerShell cmdlets. File system artifacts from the enumeration (if any temporary files were created) are not captured.

## Assessment

This dataset provides excellent telemetry for detecting T1012 Query Registry through process-based monitoring. The Security channel's 4688 events with full command-line logging capture the complete technique execution with perfect fidelity - the massive chained command line is a high-confidence indicator of systematic registry enumeration. Sysmon's process creation events complement this with additional metadata (hashes, parent relationships, integrity levels). The combination of 22+ registry queries targeting known persistence locations in rapid succession creates a strong behavioral signature. However, the dataset's value is somewhat limited by the lack of actual registry access telemetry and query results. The predominance of exit code 0x1 suggests many queried keys don't exist on this system, which is normal but reduces the dataset's utility for testing detections of successful enumeration.

## Detection Opportunities Present in This Data

1. **Bulk Registry Enumeration Pattern**: Multiple reg.exe processes spawned in rapid succession (22 processes within ~45 seconds) querying persistence-related registry paths
2. **Persistence Location Targeting**: Registry queries specifically targeting Run keys, Winlogon entries, services, and other common persistence mechanisms in both HKLM and HKCU hives
3. **Chained Command Execution**: Single cmd.exe process with extremely long command line containing multiple ampersand-separated reg query commands
4. **Services Enumeration with Filtering**: Specific pattern of `reg query HKLM\system\currentcontrolset\services /s | findstr ImagePath 2>nul | findstr /Ri ".*\.sys$"` indicating driver/service discovery
5. **PowerShell to CMD Registry Pattern**: PowerShell process spawning cmd.exe specifically for registry enumeration rather than using native PowerShell registry cmdlets
6. **High-Value Registry Key Aggregation**: Simultaneous querying of Winlogon\Userinit, Winlogon\Shell, ShellServiceObjectDelayLoad, Active Setup, and Group Policy Scripts locations
7. **Cross-Hive Enumeration**: Systematic enumeration of both HKLM and HKCU variants of the same registry locations (Run, RunOnce, Policies\Explorer\Run)
